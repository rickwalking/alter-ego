## Proposals (for approval — NOT yet created)

### P1 — Frontend unused-export / dead-code gate (knip), baseline-ratchet  [ratchet: UP] — T2

- **Failure class:** #1. Root cause: frontend has no dead-export detector; the
  backend/frontend tooling is asymmetric (`vulture` exists only on the backend).
  ESLint structurally cannot see unused exports.
- **External evidence:** `knip` is the 2025/2026 de-facto standard (ts-prune is
  archived and points to knip; `eslint import/no-unused-modules` false-positives
  on our `@/*` absolute imports + 29 barrel files and bloats lint time). knip's
  Next.js plugin natively understands app-router entrypoints
  (`page/layout/route.tsx`), so it won't flag framework files.
- **Enforcement (exact files):**
  - `frontend/package.json` — add `knip` devDep + `lint:dead-code` /
    `dead-code:baseline` scripts.
  - `frontend/knip.json` — Next.js + Storybook + Vitest entrypoints; `exports`/`files` as `warn`.
  - `frontend/scripts/check-dead-code.mjs` + `generate-dead-code-baseline.mjs` +
    `dead-code-baseline.json` — reuse the proven `check-component-type-location.mjs`
    baseline-ratchet pattern (grandfather current count + allowlist; block only
    NET-NEW; DOWN-only).
  - `scripts/ci/gates.sh` — `gate_frontend_dead_code` + register `dead-code` in `FRONTEND_GATES`.
  - `.github/workflows/frontend-quality-gates.yml` — `dead-code` job, **advisory
    (`continue-on-error: true`) for the introductory wave** (mirror jscpd
    test-advisory/mutation), flip to blocking in a follow-up.
  - `scripts/ci/check-integrity.sh` — register the baseline as a ratchet artifact
    (raised count = loosening → flagged), like `BASELINE_*` / jscpd `"threshold"`.
- **Why baseline (not zero-tolerance):** a first run would surface dozens–100+
  findings (barrel re-exports, public-API constants, shared types). Genuine dead
  code is a small fraction → must grandfather + ratchet down, exactly like the
  jscpd threshold and component-type baseline.
- **AC:** gate fails on a seeded net-new unused export; passes at the grandfathered
  baseline; baseline count may only decrease (raising it flagged by check-integrity).
- **Eliminates:** failure class #1 going forward; closes the backend/frontend
  dead-code asymmetry.

### P2 — Document when `.feature` files are required vs unit-tests-suffice  [ratchet: HOLD/UP] — T1

- **Failure class:** #2. Root cause: "Gherkin-first" is stated without scope, so
  refactor/CI-config tickets are flagged for "missing .feature" even when unit
  tests fully cover them.
- **Enforcement (exact files):** clarify in `CLAUDE.md` (Testing section) +
  `frontend/AGENTS.md` / `backend/AGENTS.md`: behavior-changing feature tickets
  require `.feature` scenarios (happy + edge + failure); pure refactors and
  CI/config tickets may substitute focused unit tests + the gate's own
  seeded-violation test, citing them in the ticket. No gate loosened — this only
  removes ambiguity so QA stops raising a non-actionable warning.
- **AC:** the standard names the two cases explicitly; QA checkpoints reference it.
