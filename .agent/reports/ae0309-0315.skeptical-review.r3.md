# Cold Critic Review

## Verdict
BLOCK

Two tickets (AE-0314, AE-0315) rest on critical unverified assumptions that could invalidate their core design. One (AE-0313) has a latent correctness gap in the idempotent path it explicitly relies on. Multiple cross-ticket dependencies are undeclared, risking half-applied shared surfaces. The set is sound in intent but not yet in execution foundation.

---

## Findings

### [BLOCKER] AE-0314: `aupdate_state` on an END-state thread is unverified — the entire completed-project edit flow depends on it

- **Assumption:** Writing to a LangGraph checkpoint via `aupdate_state` on a thread that has reached END "simply appends a new checkpoint record" and is valid.
- **Risk:** The codebase **deliberately avoids END** to sidestep this exact class of question. Domain constants (`carousel_workflow.py:11-15`) say "after final-review approval the graph holds here at an interrupt **instead of reaching END**, so the approved carousel stays resumable." The `approved_hold_phase` node (`carousel_workflow_nodes.py:386-405`) parks held carousels explicitly for this reason. END path is "only the explicit finalize." The only existing recovery from an END-stuck thread uses `ainvoke(Command(goto=...))` — **not `aupdate_state`** (`carousel_workflow_engine.py:134-140`). There is **zero** existing test or doc proving `aupdate_state` on an END-state thread writes successfully. Two prior skeptical reviews flagged this as unverified.
- **Impact:** If `aupdate_state` silently writes to a terminal checkpoint or produces undefined state, the completed-project edit — the entire publish-page editor feature — is non-functional. The ticket itself says "if it fails, escalate before proceeding," which means the feature has a single-point-of-failure assumption with no fallback.
- **Suggested mitigation:** Gate the entire ticket on a **pre-implementation proof**: write the integration test against a real END-state thread **before** authoring any frontend or backend code. If it fails, AE-0314 must be redesigned (option b — status-aware read derivation — has wider blast radius but is the only realistic alternative). Do not ship the ticket as Ready until this test exists and passes.
- **Open question for author:** Has anyone confirmed what LangGraph's `AsyncPostgresSaver.aupdate_state` does when the thread's `next` tuple is empty? Does it succeed silently and produce a checkpoint with no resumable state? Does it also create a spurious `as_node=None` path that clears pending interrupts (the documented footgun in `carousel_workflow_engine.py:110-114`)?

### [BLOCKER] AE-0313: `_activate_existing` skips `write_current_index` — the idempotent republish path leaves `current.json` stale

- **Assumption:** Idempotent republish on unchanged content "re-activates" the existing version via `_activate_existing`, which AE-0313 relies on for "unchanged content reproduces the same digest and re-activates it."
- **Risk:** Codebase audit confirms `_activate_existing` (`artifact_build_service.py:172-209`) **does not call `write_current_index`**. It only runs the DB CAS (`activate_build`) and stamps `project.artifact_version`. The on-disk `current.json` is never rewritten. A lazy reconciler fixes it on the read path (`artifact_path_resolver.py:90`), but between activation and the next read, `current.json` can lag the DB indefinitely.
- **Impact:** AE-0313's acceptance criterion — "republish with unchanged content is a safe no-op" with "current.json reflects exactly one coherent activation" — is not met by the existing code. A republish that flips the DB then leaves `current.json` pointing at the prior version. The reconciler masks this for normal reads, but the acceptance criterion says "current.json reflects exactly one coherent activation" — under `_activate_existing`, it reflects the prior one until the next read.
- **Suggested mitigation:** Either (a) call `write_current_index` inside `_activate_existing` (the call is cheap), or (b) explicitly relax the acceptance criterion to "current.json is reconciled lazily on the next read" and add a test proving the reconciler covers the republish + no-subsequent-read-interaction path. Option (a) is cheaper and removes the risk surface entirely.
- **Open question for author:** Is the reconciler guaranteed to run before any downstream consumer (PDF serve, slide indexing) after a republish? If a user clicks "Rebuild PDF" and nothing reads the project before they download, do they get the old PDF?

### [BLOCKER] AE-0315: "every mutation site checks the cancellation marker" is a ~28-site refactor, not a hardening

- **Assumption:** A `run_cancelled` flag the reaper sets before flipping a dead row, "every mutation site checks," is a small enablement — the ticket budgets it within one T2 ticket.
- **Risk:** Codebase audit counts **~28 distinct hard DB/checkpoint mutation sites** across 6 layers (12 node-return-checkpoint sites, 3 direct `aupdate_state` patches, 13 `carousel_projects` row writes through `CarouselProjectWriteOwner`, 11 per-slide asset writes, 5 finalize writes, plus event/SSE side-effects). Node-return dicts are **implicit writes** — LangGraph applies them after the node returns, so a check inside the node body does not gate the commit. One site (`editorial_workflow_service.py:327-344` `_append_image_visual_feedback`) **bypasses the write owner entirely** and commits directly. Per-slide image writes fan out in `asyncio.gather`-style concurrency. The checkpoint and the DB row are **two sources of truth** that diverge — a cancel flag on the DB row is invisible to an in-flight node until the next checkpoint read.
- **Impact:** A single missed site = a reaped zombie task that writes after the reap, corrupting state — the exact failure class the cancellation marker is designed to prevent. The ticket claims "cancellation safety" as an acceptance criterion; with ~28 sites and no enumeration in the ticket, the test cannot prove completeness (you cannot test what you haven't listed).
- **Suggested mitigation:** Do not attempt a per-site marker check. Survey says the writes funnel through ~3 chokepoints: (a) the node-return boundary via a harness pre-commit hook, (b) `CarouselProjectEngine.update_state`, and (c) `CarouselProjectWriteOwner.commit`. Gate at those chokepoints instead and document that lateral bypass sites (`_append_image_visual_feedback`, direct `db.commit()` at `editorial_workflow_service.py:344`) are either routed through the owner or covered by an additional guard. Add the enumeration to the ticket as a non-goal ballot (this list IS the ticket's completion criterion) and budget a T3, not T2.
- **Open question for author:** How does the reaper's cancellation flag propagate to the in-flight `asyncio.gather` per-slide image generation? If a slide is mid-generation and the reaper sets the flag, does the gather observe it without an explicit cancellable event?

### [WARN] AE-0315: `phase_status` atomic-write invariant spans 7 write sites (3 bypass the owner) — application-level enforcement is fragile

- **Assumption:** "`run_started_at` must be cleared in the same UPDATE statement that writes any non-in_progress `phase_status`" — the ticket says "no separate-statement path exists."
- **Risk:** Codebase audit confirms 7 write sites total (mostly centralized through `CarouselProjectWriteOwner.sync_phase` and `set_phase_status_and_commit`). Three production sites bypass the owner entirely: (1) `WorkflowTimeoutRepository._reject` (`workflow_timeout_repository.py:102`), (2) `Phase5MigrationService._backfill_workflow_state` (`phase5_migration_service.py:301`), and (3) `CarouselProjectModel.update_from_entity` — the ORM hydrator that runs on essentially every `update_project` call including terminal finalize. A pristine "same-UPDATE" invariant requires all three to also clear `run_started_at` simultaneously.
- **Impact:** If any bypass site transitions `phase_status` to a non-in_progress value without clearing `run_started_at` (the timeout reaper is the most likely), a dead run's `run_started_at` survives, the row can be re-flagged as needing reap, and the watchdog could flip a healthy post-rejection row back to `awaiting_human`.
- **Suggested mitigation:** Enforce the invariant at the DB layer with a SQLAlchemy `before_update` event listener on `CarouselProjectModel` keyed on `phase_status` transitions — only this catches all 7 sites uniformly. Add an assertion test that every non-in_progress transition in the suite clears `run_started_at`. Application-level enforcement at the owner alone leaves the 3 bypass sites unguarded.
- **Open question for author:** Does ` CarouselProjectModel.update_from_entity` (the hydrator) trigger a `before_update` event when the entity's `phase_status` equals the row's existing carry-over (a no-op hydrate)? If so, the listener would spuriously clear `run_started_at` on unrelated `update_project` calls.

### [WARN] Cross-ticket: AE-0311 depends on AE-0313's advisory lock and AE-0315's typed-409 framework, but neither is declared as a blocked-by

- **Assumption:** AE-0311 claims it "acquires the same per-project advisory lock AE-0313's republish uses" and returns "typed 409" details. The asserted "shared serialization domain" couples two tickets.
- **Risk:** Codebase confirms **no advisory lock and no republish endpoint exist today** — both are net-new in AE-0313. If AE-0311 ships before AE-0313, the lock construct doesn't exist yet; either AE-0311 synthesizes it independently (causing a future merge conflict on the lock domain) or it ships without the lock (allowing repair-vs-republish races on completed carousels, the exact class of bug AE-0311 is supposed to prevent). The same applies to AE-0315's typed-409 detail framework — AE-0311 returns "409 with a typed detail" but the framework/classification is owned by AE-0315, which AE-0311 doesn't list as blocked-by.
- **Impact:** Two scenarios: (a) AE-0311 first → duplicate lock implementation dance, or (b) AE-0311 ships without the lock → documented race window between repair and republish on completed projects that AE-0311 explicitly promises to serialize.
- **Suggested mitigation:** Add explicit dependencies: AE-0311 `blocked_by: AE-0313` (for the shared lock) and `blocked_by: AE-0315` (for the typed-409 framework). Alternatively, extract the lock definition and the typed-409 schema into a tiny ticket (T1) that both AE-0311 and AE-0313/0315 depend on — this is the standard move when two features share a first-classified surface.
- **Open question for author:** What is the intended ordering: AE-0313 before AE-0311 (lock first, repair consumer second) or AE-0311 before AE-0313 (repair defines lock, republish adopts)? The dependencies block doesn't enforce either order.

### [WARN] AE-0312: v1 carousel in-flight stays unprotected from casing — existing incomplete carousels receive no casing check without manual opt-in

- **Assumption:** "New carousels route to v2; v1 untouched and still supported for in-flight projects." A v1 project wanting casing enforcement opts in by "upgrading the project's `presentation_policy_version` to v2 (an informational note in the repair response when lowercase-looking copy is detected on v1)."
- **Risk:** A project created **just before** the v2 deploy is on v1, approaches content review **after** the deploy, and the user correctly spots lowercase copy. The AE-0311 repair returns a clean report (no casing violations on v1) with a passive "you can upgrade your policy version" note. The user must know what a `presentation_policy_version` is, find a way to bump it (no documented endpoint in scope), and re-run the repair. The whole point of AE-0312 is to prevent prod lowercase shipping — this v1 gap lets it recur for any pre-deploy-created carousel still in flight.
- **Impact:** The incident class AE-0312 directly addresses (66014ba3, lowercase PT headings shipped to prod) can still ship for any carousel in-flight at deploy time — exactly the projects most likely to be review-active immediately after deploy.
- **Suggested mitigation:** Either (a) auto-upgrade the policy version for any unfinished (non-completed) project on first content/design/final_review touch after the v2 deploy (a one-shot run-once migration), or (b) explicitly document the v1→v2 upgrade mechanism in scope — at minimum an admin/ops script to bump in-flight projects — and let AE-0311's informational note link to it. "Informational only, no auto-upgrade" alone does not satisfy the stated goal of "Portuguese carousel copy is held to the same casing bar as English."
- **Open question for author:** How long will the fleet contain v1 carousels after deploy — are they expected to drain naturally (completed, archived), or are there long-lived draft carousels where this gap is permanent?

### [WARN] AE-0311: dual-write partial failure has no active reconciliation — only "retry converges" — but no one is guaranteed to retry

- **Assumption:** "On partial failure the endpoint reports which store was updated and the retry (idempotent) converges them." The reconciliation check "reports drift until it does."
- **Risk:** The endpoint writes projection first, checkpoint second. If the process dies between commits, the split state (repaired projection, unrepaired checkpoint, stale blocking report) persists **indefinitely** until someone invokes the repair again. The reconciliation step is built into **the repair endpoint itself** — there is no scheduled reconciler, alert, or watchdog that detects drift autonomously. The user sees a 500 / generic error and may not click "Fix issues" again.
- **Impact:** For an in-flight carousel, the split state means checkpoint state still carries a blocking validation report (the repair endpoint wrote the projection; the checkpoint's `presentation_validation` is stale-broken). The next workflow step reads from checkpoint (for in-flight projects), so it will re-interrupt on a violation that the projection already fixed — the session is stuck unless someone retakes the repair.
- **Suggested mitigation:** Add a scheduled reconciliation job (or fold into AE-0315's watchdog tick) that detects drift between projection and checkpoint `presentation_validation` for in-flight projects and either (a) auto-converges (idempotent by design) or (b) emits an `carrousel_repair_drift_detected` alert — both bootstrap the retry the acceptance criterion assumes will happen.
- **Open question for author:** For completed projects, is the projection authoritative (so a projection-only success is enough to unblock service even if the checkpoint lag stays stale)? For in-flight projects, can the workflow progress past a stale-blocking checkpoint report?

### [WARN] AE-0310: adding validation to design ensure creates a "louder but still-looping" UX when the reviewer does not pick an explicit recovery action

- **Assumption:** A plain design revise with blocking violations re-validates and "if the fresh report still blocks, the node re-interrupts with it (the reviewer then picks an explicit recovery action — edit or send-back)."
- **Risk:** The ticket removes the "silent no-op" (good) but replaces it with "re-interrupt with the same violations and wait for the reviewer to explicitly choose." If the reviewer's mental model is "revise means fix it," they will keep clicking revise (now they get a visible but identical violation each time) until the design revision cap (from AE-0310's own change) trips a 409 — which under AE-0315 will surface as "revision cap exceeded." The UX effectively shifts the user from "silently stuck" to "noisily stuck until capped" without an in-product hint that the only escape paths are Edit or Send back.
- **Impact:** Reviewer confusion, premature cap exhaustion, support tickets. The revision cap fix is correct mechanically but operationally a UX regression for reviewers who haven't learned the new explicit-action vocabulary.
- **Suggested mitigation:** In the re-interrupt payload, include a client-displayable hint: "Direct text edits or 'Send back to content' can resolve these violations. 'Revise' alone does not modify content." Surface this in the design-step UI as a guided CTA, not just a free-text violation list. The explicit recovery actions should be visually dominant when a blocking report is present, not equal-weight with "revise."
- **Open question for author:** After the cap 409, does the reviewer have any escape? Or is the carousel dead-locked (no more design revisions, can't send back because content cap is shared? The ticket says design→content send-back consumes content's cap, not design's) — what is the user's recovery path when both the design cap and content cap are exhausted?

### [INFO] AE-0313: `current.json` flip and DB CAS are not co-transactional even within `build_and_activate` — write order leaves on-disk index ahead of DB on CAS loss

- **Assumption:** The republish endpoint is serialized via a new advisory lock so concurrent republishes don't interleave.
- **Risk:** Within `build_and_activate` itself, `write_current_index` (line 128) **precedes** `activate_build` CAS (line 129) by design. The CAS protects against concurrent republish-vs-republish (acceptance criterion). However, if a concurrent republish loses the DB CAS, today's `_activate_existing` was already interrupted by the same advisory lock — but the losing republish has already written `current.json` to disk. Wait — with the advisory lock, the loser doesn't run concurrently, so this ordering is fine *as long as the lock is held from `write_current_index` through `activate_build`*. The ticket doesn't pin the lock's start and release boundaries explicitly.
- **Suggested mitigation:** Pin in the implementation plan: acquire the advisory lock at the start of `build_and_activate` (before the digest computation), release after the DB commit. Today the lock could be acquired only at the file-step boundary, leaving the digest/manifest computation unlocked (a minor but real correctness sliver).
- **Open question for author:** At what code point does the per-project advisory lock scope begin — at the endpoint entry, at `build_and_activate` entry, or at the moment of the `write_current_index` call?

### [INFO] AE-0312: proper-noun list maintenance burden not addressed — adding new proper nouns requires code+deploy

- **Assumption:** "Words from a configurable proper-noun list (constants file; seeded with `Claude`, `Anthropic`)."
- **Risk:** A proper-noun list living in source code constants requires a code PR + deploy for every new product/company the copy might reference. Carousels about a rotating cast of AI products/companies will hit lowercase `gpt`, `openai`, `gemini` etc. faster than a code change ships. The list will either under-fit (real violation not flagged) or over-fit (every new product gets a hard-coded entry), and there's no per-project override path.
- **Mitigation:** At minimum document who owns the maintenance cadence and when to extend (e.g. quarterly review). Ideally move it into the policy YAML (`hero_lower_third_v2.yaml`) — same file as the casing rules — so it ships with a policy update no code PR required. The ticket's own design already uses "policy-version-aware" repair, which suggests the policy file is the natural home for the proper-noun constants.
- **Open question for author:** Is there a per-project escape hatch (e.g. a project-specific proper-noun allowlist) when the fleet-wide list isn't relevant to a particular carousel's subject?

## Residual risks if plan proceeds unchanged

- If AE-0314 ships as Ready, the implementer spends the full frontend/backend investment, hits the `aupdate_state`-on-END test, possibly fails, and the work blocks escalations — potentially weeks of wasted implementation with an unclear fallback (option b is rejected for blast radius, and no third fallback is documented).
- If AE-0315 ships with an incomplete mutation-site sweep, a single uncovered site lets a reaped zombie task write after reap (the silent corruption the reaper was specifically designed to prevent) — turning the fix into a new class of bug.
- For completed carousels repaired via AE-0311, the split-state window between projection commit and checkpoint commit is only closed by client retry — if the client doesn't retry (browser closed, network drop), in-flight projects can be left stuck on a stale-blocking checkpoint report with no one to converge them.
- Existing v1 carousels in flight at the v2 deploy will continue to ship lowercase PT headings until each individual project's policy version is manually bumped — the exact prod incident class AE-0312 was sold to preventpersists for the immediate post-deploy window, the window of highest review traffic.
