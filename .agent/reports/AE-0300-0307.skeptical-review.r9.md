# Cold Critic Review

## Verdict
WARN

## Findings

### [WARN] AE-0302's consumer enumeration is bounded to *backend* code — a shared Redis likely also serves Langfuse (a third-party container), and the ticket neither enumerates nor wires it through the authed factory
- Assumption: The only Redis consumers are backend services routable through the new single factory; enabling `requirepass` + `rename-command` only affects backend code paths.
- Risk: The prod stack inventory (`langfuse`, `langfuse-worker`, `clickhouse`, `minio`, `postgres`, `redis`) reads like one shared `alter-ego-redis-1`. Langfuse self-hosted v3 commonly uses Redis for caching/queue. If Langfuse's container connects to this Redis, then enabling `requirepass` NOAUTHs Langfuse (and `rename-command` could break ops verbs it uses) — and Langfuse cannot be routed through the *backend* factory. The factory enforcement lint (`from redis import` outside the factory module) does not and cannot cover a third-party container.
- Impact: Post-deploy Langfuse outage (traces/observability down) discovered as a runtime regression, not a build failure; or, worse, a partial "smoke passed (backend OK)" sign-off while Langfuse silently retries.
- Suggested mitigation: Add an AC to *enumerate every container/process* connecting to `alter-ego-redis-1` (not just backend Python consumers) and to prove each is authed *or* explicitly excluded by splitting Redis into backend-only + Langfuse-only instances. The factory lint scope must be restated as "no backend code bypasses auth," not "all consumers are authed."
- Open question for author: Is `alter-ego-redis-1` shared with Langfuse? If yes, where is Langfuse's Redis password configured and is that wiring in scope of this ticket?

### [WARN] AE-0300's primary control (L4 CF-IP allow-list) is silently bypassable via Cloudflare Workers egress, yet the AC asserts the bypass is "refused"
- Assumption: Allowing only Cloudflare's published egress IPs on 80/443 makes direct-to-origin requests non-Cloudflare and thus refused.
- Risk: Cloudflare Workers (and any other CF tenant) egress from IPs within the published Cloudflare ranges. An attacker can `fetch(<origin-ip>)` from a Worker — the L4 firewall sees a CF IP and admits it; nginx sees a request with no CF zone headers (or forged ones) and serves the app. The WAF/rate-limit the ticket claims to enforce remains bypassable via this path. The ticket explicitly defers true zone-binding (`cloudflared` / per-zone origin cert) to a follow-up, which is honest — but its AC #1 ("direct-to-origin request that does not originate from Cloudflare is refused") will literally pass while the threat (WAF bypass against origin) remains.
- Impact: Net isolation improvement is real, but the headline guarantee ("origin only serves Cloudflare traffic") is overstated; a future reviewer may close the gap as fixed when only the trivial `curl --resolve` vector is closed. mTLS (Auth Origin Pulls) does not help either, since the same shared public CF origin-pull CA would be used by a Worker? — actually Workers do not present the origin-pull mTLS cert, so `ssl_verify_client on` *would* reject a Worker fetch. That is the one thing mTLS buys — contradicting the ticket's "near-cosmetic" framing.
- Suggested mitigation: Reconcile the mTLS claim with the Workers-bypass: IF `ssl_verify_client on` actually rejects Worker fetches (no client cert), then mTLS is NOT near-cosmetic — it is the only thing closing the L4 bypass. If Workers *can* present the origin-pull cert, state it. Either way, the AC and the "L4 = THE control" wording need correction, not just a footnote.
- Open question for author: Does a Cloudflare Worker fetch to the origin complete a TLS client-cert handshake against `ssl_verify_client on`? This single fact changes whether the residual is "Workers bypass closed" or "Workers bypass open."

### [WARN] The shared rotation/reboot lock has an unowned dependency direction: AE-0306 (rotation, 14-day deadline) consumes the lock that AE-0303 (reboot, Tier T1, gated only on AE-0304) is supposed to define — AE-0306 is NOT blocked by AE-0303
- Assumption: AE-0303 lands before AE-0306 and therefore the lock file path/holder/TTL/force-clear semantics exist when AE-0306 needs them.
- Risk: Nothing enforces that order. AE-0306 (T2, High) is blocked by AE-0301 only; AE-0303 (T1) is blocked by AE-0304. AE-0306 can land first — especially given the 14-day AE-0306 deadline versus AE-0303's dependence on AE-0304's ≥24h soak. If AE-0306 lands first, it "references a lock" that has no defined location, holder, or recovery semantics; the implementer will either invent one (duplicating AE-0303's work and risking divergence) or skip the lock and run the rotation unguarded (the exact mid-rotation reboot failure the lock exists to prevent).
- Impact: Either two divergent lock implementations, or a rotation run against a non-existent lock — the cross-ticket invariant the whole hardening set relies on is a coincidence, not a gate.
- Suggested mitigation: Make AE-0306 explicitly `Blocked by: AE-0303` for the lock definition (or extract the lock definition into its own T1 precursor ticket that both AE-0303 and AE-0306 consume). The "AE-0303 owns the lock" statement must be backed by a real dependency edge, not prose.
- Open question for author: Which ticket is the canonical lock *landlord* if AE-0306 slips ahead of AE-0303 in the queue?

### [INFO] AE-0305's CSP `img-src https:` is permissive enough that dropping `http://localhost:8000` is cosmetic
- Assumption: Removing the localhost dev artifact materially tightens the prod CSP.
- Risk: `img-src 'self' data: blob: https:` already allows any HTTPS image source; the localhost entry adds no real exfil surface. This is fine as cleanup, but the QA "automated CSP regression guard" will be built to protect a no-op change, and may obscure the fact that the *real* CSP weakness (the bare `https:`) is untouched and out of scope.
- Impact: Low; just transparency. A reviewer reading "CSP hardening" may over-credit the ticket.
- Suggested mitigation: State in the ticket body that this is dev-artifact hygiene, not a CSP tightening, so the residual (`https:` wildcard in `img-src`) is visible and can be evaluated on its own in a later ticket if desired.
- Open question for author: N/A.

### [INFO] AE-0301's "name its exerciser" placeholders for GLM and Pinecone are unbacked — they may not exist, and "add a minimal exerciser" silently expands a T1 clean-up ticket
- Assumption: A named per-provider exerciser can be produced before execution at acceptable cost.
- Risk: If no GLM/Pinecone smoke path exists, the gate forces either building one (scope creep on a T1 file-perms ticket), accepting revoke-on-faith (exactly what the gate was added to forbid), or excluding (defers the rotation, leaving those keys live after the 644 window). All three are pre-listed — good — but none is pre-chosen, so the decision is deferred to execution time under deadline pressure.
- Impact: Two of three branches either balloon the ticket or leave keys exposed. The gate is honest but the cost branch is unestimated.
- Suggested mitigation: Pre-decide the GLM and Pinecone resolution (add minimal exerciser vs. accept-on-faith vs. exclude) *in the ticket* with the rationale already recorded — execution should resolve a stated choice, not make it.
- Open question for author: Does a GLM system-model exerciser even exist that can run cheaply in the deploy-verify path?

## Missing evidence
- Whether `alter-ego-redis-1` is shared with the Langfuse/Langfuse-worker containers and, if so, how Langfuse is configured for Redis auth (decisive for the AE-0302 finding).
- Whether `ssl_verify_client on` with the CF origin-pull CA rejects a Cloudflare Worker `fetch` to the origin (decisive for the AE-0300 mTLS-vs-L4 finding).
- The deploy ordering graph for AE-0303 vs AE-0306 (no Blocked-by edge exists despite the lock ownership claim).
- Whether a named ops alert sink actually exists today (AE-0300 staleness, AE-0303 post-reboot smoke both assume one; AE-0303 only says "if none exists, email the admin").
- The actual source range of the GitHub deploy runner post-AE-0307 and whether an on-box self-hosted runner is being considered despite the discouragement (cost pressure typically forces it).

## Residual risks if plan proceeds unchanged
- A Cloudflare-Worker origin bypass may remain live post-AE-0300 while the ticket is closed as "direct-to-origin refused," because its AC measures the trivial vector, not the threat model.
- AE-0302 may silently strand Langfuse if Redis is shared — the backend factory + backend lint give no visibility into the third-party container's auth state.
- AE-0306 may run a `POSTGRES_PASSWORD` rotation unguarded (or with an ad-hoc local lock) if it lands before AE-0303, the ticket it trusts to provide the lock.
- The "automated CSP regression guard" in AE-0305 protects a low-value cleanup while the broad `img-src https:` remains unexamined, giving a false sense of CSP coverage.
- AE-0301's GLM/Pinecone rotation may slip out of the 644 remediation window because the exerciser gap is recognized only at execution time.
