# Cold Critic Review

## Verdict
BLOCK

Three findings reach blocker severity. The set is well-rehearsed (r2–r4 inline) and most per-ticket logic is sound, but the cross-ticket surfaces — the watchdog tick, the advisory-lock domain, the severity model's ship order, the rollback story for a run-once migration, and the cancellation-marker lifecycle — have unresolved gaps that can reintroduce the exact incident classes the tickets exist to prevent.

## Findings

### [BLOCKER] AE-0315 cancellation marker is a boolean, not a fencing token — its lifecycle is unsolvable

- Assumption: A boolean `run_cancelled` column on `carousel_projects` lets the reaper mark a zombie so its writes are rejected, while a replacement run can proceed.
- Risk: The marker is per-project, not per-run. To reject the zombie it must persist until the zombie is provably dead. To let the replacement run write, it must be cleared. These constraints conflict: clear it on the replacement run's `phase_status=in_progress` → a late-reviving zombie writes after the clear; keep it → the replacement run's writes are also rejected (deadlock). There is no third state in a boolean. The standard solution is a monotonic **epoch/fencing token**: each run captures the epoch at start and stamps every write (CAS against current epoch); the reaper bumps the epoch; the zombie's stale-epoch writes fail the CAS, the replacement's current-epoch writes pass. The ticket specifies neither.
- Impact: Either replacement runs deadlock (workflow unrecoverable after a reap) or zombies write post-reap (the split-brain the prod incident risked). The AC "a reaped task that revives cannot land writes" and "the user's next resume attempt is accepted" cannot both hold under a boolean marker.
- Suggested mitigation: Replace `run_cancelled` with a `run_epoch` (monotonic int). The reaper increments it in the flip UPDATE. Every write path (ORM flush guard, checkpoint commit, raw UPDATEs) compares the writer's captured epoch against the row's current epoch and rejects on mismatch. This makes zombie-rejection and replacement-acceptance simultaneous and correct.
- Open question for author: Why is a boolean marker sufficient when the zombie and the replacement run share the same project row? What clears the marker, and how does that not unblock a zombie?

### [BLOCKER] AE-0312 auto-upgrade leaves stale stored reports — reviewers approve past casing violations they never see

- Assumption: Auto-upgrading in-flight projects to v2 policy means "worst case is new visible warnings with a working repair button."
- Risk (code-verified): `validate_localized_slides` is the only thing that builds a v2 report (`presentation_review.py:91-102`), and AE-0314 itself admits the read fast-path serves the **stored** `presentation_validation` verbatim (re-derives only when none stored). The run-once migration bumps `presentation_policy_version` to v2 but does **not** re-run validation or refresh the stored report. A project parked at `final_review` (the last gate before `approved_hold`) is auto-upgraded, but final_review approve does not re-validate (validation is content-approve-only historically; AE-0310 adds it to *design*, not final_review). So the reviewer hits approve on a stale v1 report (`blocking=False`, no casing check), the carousel parks at `approved_hold` and ships lowercase headings — the precise 66014ba3 incident.
- Impact: The auto-upgrade creates a false sense of coverage for the highest-risk cohort (in-flight at final_review) while not actually surfacing violations. The ticket's own safety premise is false for that cohort.
- Suggested mitigation: The migration must re-validate every upgraded project's `localized_slides` under v2 and store the fresh report (or invalidate the stored report so the fast path re-derives). OR pin that auto-upgrade only applies to phases whose approve path re-validates (content, design post-AE-0310) and explicitly exclude final_review until a final_review re-validate path exists.
- Open question for author: For a project parked at final_review when the migration runs, what code path re-validates under v2 before the reviewer can approve? Does the publish-page health card re-validate or serve the stored report?

### [BLOCKER] AE-0312 rollback story is unspecified — a code rollback after the run-once migration can deadlock every v2 project

- Assumption: The rules/severity/policy can deploy any time and the auto-upgrade is a separate deliverable; rollback is straightforward.
- Risk: The auto-upgrade is a run-once, irreversible-ish column bump on all non-completed projects. If AE-0312's code is rolled back (main auto-deploys on every merge), those rows still say v2. The previous code's `SUPPORTED_PRESENTATION_POLICY_VERSIONS` does not include v2. The policy loader's behavior on an **unsupported** version is unspecified: if it raises, every v2 project (all in-flight + all created post-AE-0312) is unreadable → prod freeze; if it falls back to v1, the v2 casing rules silently disappear (safe but the migration's value is lost and re-deploy may re-trigger it).
- Impact: Given "pushing to main auto-deploys production," an unspecified unsupported-version fallback is a deploy-safety hole. The consequence of the wrong behavior is a broad production freeze affecting every carousel in flight.
- Suggested mitigation: Specify and test the loader's unsupported-version behavior *before* shipping the migration (fallback-to-v1 is safe; raise is not). Make the migration idempotent and reversible (a v2→v1 down-grade path), or gate the migration behind a feature flag so a rollback doesn't strand v2 rows.
- Open question for author: What does `load_presentation_policy(version)` do when `version` is not in `SUPPORTED_PRESENTATION_POLICY_VERSIONS`? Is there a down migration?

### [WARN] AE-0316 transaction-scoped advisory lock does not cover AE-0311's two-commit boundary

- Assumption: `pg_advisory_xact_lock` held during repair serializes repair-vs-republish for the full dual-write.
- Risk: A transaction-scoped lock is released when the `carousel_slides` projection transaction commits — **before** the checkpoint `aupdate_state` write (separate connection pool, no shared transaction; stated system fact). Between commit 1 and the checkpoint write, a republish can acquire the lock and render. AE-0311's AC ("repair and republish can never interleave") is not met across the two-commit seam. The practical PDF impact is bounded (republish reads committed slides), but the stated guarantee is overstated and the checkpoint/projection divergence window is unguarded by the lock.
- Suggested mitigation: Use a session-scoped lock (`pg_advisory_lock`/`unlock`) held across both commits, or document that the lock covers only the projection write and the checkpoint convergence is best-effort (then drop the "never interleave" AC).
- Open question for author: Which lock scope — transaction or session — and does it span the checkpoint write?

### [WARN] AE-0314's completed-project slide-update endpoint is outside the AE-0316 serialization domain

- Assumption: The advisory lock serializes "the two artifact-affecting mutators" (AE-0316 contract: repair + republish only).
- Risk: AE-0314 adds a THIRD artifact-affecting mutator — a completed-project slide-update endpoint that writes `carousel_slides` then chains republish. It specifies "write-owner guarded" but **not** the AE-0316 advisory lock nor a `lock_version` CAS on the slide write. A user editing text (AE-0314) and a repair (AE-0311) on the same completed project can both write `carousel_slides` with no shared guard → lost update. AE-0311's CAS guards the `carousel_projects` row, but AE-0314's endpoint is not specified to participate.
- Suggested mitigation: Add AE-0314's slide-update endpoint to the AE-0316 serialization-domain contract and have it acquire the advisory lock (or take the lock_version CAS) before writing slides.
- Open question for author: Does the completed-project edit endpoint acquire the advisory lock or a CAS before mutating `carousel_slides`?

### [WARN] AE-0311's drift reconciler and AE-0315's reaper co-own the watchdog tick with undefined interaction

- Assumption: The workflow-workers tick can host both the drift reconciler (AE-0311) and the stale-run reaper (AE-0315) independently.
- Risk: For a project with both a stale heartbeat AND projection↔checkpoint drift, the reaper sets the cancellation marker (or epoch) and flips `phase_status`+`lock_version`. AE-0311's drift reconciler then calls `aupdate_state` — which AE-0315's enforcement layer (b) gates on the cancellation marker. If the marker is set, the reconciler's `aupdate_state` is **rejected**, so the drift is never converged — the opposite of AE-0311's goal. The ordering between the two responsibilities in the same tick is unspecified.
- Suggested mitigation: Define tick ordering (reaper first, then reconciler only for non-reaped rows) and exempt the reconciler's convergence writes from the cancellation guard (or have the reaper's reconciliation subsume the drift convergence in one step).
- Open question for author: When both conditions hold for one project, which runs first, and does the cancellation guard block the drift reconciler's `aupdate_state`?

### [WARN] AE-0315 heartbeat write failure reaps a live task and spawns a duplicate run

- Assumption: Heartbeat freshness reliably distinguishes dead from alive.
- Risk: A single transient DB error on the heartbeat UPDATE (the task is alive but `run_heartbeat_at` goes stale) causes the reaper to reap a **live** task. The cancellation guard rejects the zombie's writes (good), but the task itself is not cancelled (only its writes are rejected) — it keeps running its ~15-min LLM call. The row is flipped to `awaiting_human`, a replacement resume starts, and now two LLM regenerations run concurrently: wasted spend + the live task's work is silently discarded.
- Impact: The reaper's own design goal ("never reap a slow-but-alive run") is violated by a heartbeat-write failure, which is operationally common (DB blips). The `run_overdue` alert catches slow-but-heartbeating runs, but a heartbeat-write *failure* looks identical to a dead task.
- Suggested mitigation: Require N consecutive stale heartbeat observations (not one) before reaping, or make the heartbeat write retry-on-failure within the task, or cancel the zombie's asyncio task by reference on reap (not just gate its writes).
- Open question for author: What makes a single missed heartbeat safe to reap on, given transient DB write failures?

### [WARN] AE-0310 cap-accounting fix changes existing final_review→content behavior for in-flight projects

- Assumption (code-verified): Today the cap CHECK reads `revision_count[current_phase]` (`editorial_workflow_service_helpers.py:121`) while the send-back INCREMENT bumps `revision_count[target_phase]` (`editorial_workflow_feedback.py:48-57`), so final_review→content send-backs are effectively uncapped against content today.
- Risk: AE-0310's fix makes the check target-aware for ALL send-back sources. An in-flight project at final_review that already accumulated ≥5 content send-backs under the old (uncapped-against-content) accounting is newly rejected on its next send-back. No migration or compat story for counts accumulated under the divergent semantics.
- Impact: Low likelihood (5+ content send-backs is rare) and recoverable via uncapped direct edits, but it is an unannounced behavior change on a live path with no in-flight consideration.
- Suggested mitigation: Document the in-flight transition; cap the new check at `max(existing_target_count, ...)` or grant a one-time amnesty, or accept it and rely on the edit escape hatch (then state it explicitly).
- Open question for author: Is there any in-flight project whose accumulated send-back count would newly exceed the cap under the corrected semantics?

### [WARN] AE-0310 plain-revise-while-blocking silently burns the design revision cap

- Assumption: De-emphasizing "revise" while a blocking report is present is sufficient to prevent cap exhaustion.
- Risk: A plain design revise now re-validates and re-interrupts with the same blocking report (by design — revise can't fix content). Each such revise increments `revision_count[design]`. A reviewer who clicks "revise" 5 times exhausts the design cap before being forced to edit/send-back. The UI de-emphasizes but does not disable revise while blocking.
- Suggested mitigation: Disable the plain "revise" action (or make it non-cap-consuming) while a blocking report is present, since it is provably a no-op for content violations.
- Open question for author: Should a revise that cannot change the blocking outcome consume a revision at all?

### [WARN] AE-0313 idempotency rests on digest determinism that is unspecified

- Assumption: Unchanged slide data reproduces the same content digest, so republish is a no-op.
- Risk: If the digest covers rendered artifact **bytes** (PDFs commonly embed creation timestamps/producer metadata), every republish produces a new digest → never idempotent → every click creates a new version dir (unbounded version growth; the "safe to click twice" AC is false). If the digest covers only slide **data** (pre-render), it's stable. The ticket says "content addressing dedupes outputs" without stating what is hashed.
- Suggested mitigation: Specify and test that `compute_artifact_version`/`compute_slides_fingerprint` (already present at `artifact_build_service.py:246-248`) hash slide data, not rendered PDF bytes; add a regression test that two renders of identical data yield identical digests.
- Open question for author: Does the fingerprint include rendered PDF bytes or only slide/manifest data?

### [INFO] AE-0311 writes the checkpoint on approved-hold threads without the as_node proof AE-0314 requires

- Assumption: AE-0311's `aupdate_state` on completed (approved-hold) projects is safe.
- Risk: AE-0314 correctly requires a pre-implementation proof that `aupdate_state` on an approved-hold interrupted thread preserves the pending interrupt (the `as_node` footgun at `carousel_workflow_engine.py:110-114` is real). AE-0311 performs the same kind of write on the same kind of thread but imposes no such proof. The existing `CarouselWorkflowEngine.update_state` wrapper *does* infer `as_node` from `pending_next[0]` (`carousel_workflow_engine.py:160-165`), which reduces the risk — but only if AE-0311 routes through that wrapper rather than calling `_app.aupdate_state` directly.
- Suggested mitigation: Require AE-0311 to use the `update_state` wrapper (which infers `as_node`) for approved-hold writes, or share AE-0314's proof obligation.
- Open question for author: Does AE-0311's repair go through `CarouselWorkflowEngine.update_state` (as_node-inferred) or directly through `_app.aupdate_state`?

### [INFO] AE-0312 severity model is a shared-surface refactor with no pinned ship order or integration gate vs AE-0309/0310/0311

- Assumption: AE-0309 keys on the report's `blocking` flag, so the severity refactor satisfies the cross-ticket invariant "automatically."
- Risk: AE-0312 modifies `_blocking_from_*`, `build_validation_report`, and `validate_localized_slides` — the same decision functions 0309/0310/0311 depend on. AE-0312 has "Blocked by: none" for these deliverables, so it can ship in any order. AE-0312's tests cover the casing-only case and a per-rule blocker sweep, but there is no integration test spanning AE-0312's refactor + AE-0309's fail-closed chain + AE-0310's design recovery together.
- Suggested mitigation: Add a cross-ticket integration scenario (casing warning + a real blocker in the same report) asserting the fail-closed chain fires only on the blocker, or pin AE-0312 to ship before/after the set with a joint gate.

## Missing evidence

- **Policy loader unsupported-version behavior** — needed to judge AE-0312 rollback safety (BLOCKER #3).
- **Which phases re-validate on approve** — needed to judge whether AE-0312's auto-upgrade surfaces casing warnings to in-flight final_review projects (BLOCKER #2). The packet states validation is content-approve-only historically + design-ensure after AE-0310; final_review is unconfirmed.
- **What `compute_artifact_version`/`compute_slides_fingerprint` hash** — needed to judge AE-0313 idempotency (WARN #9).
- **The write-site survey completeness** — AE-0315's cancellation safety is only as good as the manual enumeration of ~30-34 mutation sites; `activate_build` is confirmed to be a raw Core UPDATE bypassing ORM events (`carousel_artifact_build_repository.py:127-138`), proving the ORM-only guard is insufficient, but no mechanical completeness proof is specified.
- **AE-0314's `aupdate_state`-on-approved-hold feasibility** — acknowledged as unverified (task #1); the fallback is specified only for legacy END threads, not for "proof fails on approved-hold."
- **The advisory lock scope (xact vs session)** — needed to judge whether AE-0311's two-commit contract is actually serialized (WARN #4).

## Residual risks if plan proceeds unchanged

- A reap under AE-0315 either deadlocks replacement runs or unblocks zombies (BLOCKER #1) — the prod split-brain class is not actually closed.
- In-flight final_review carousels auto-upgraded by AE-0312 can ship lowercase headings the reviewer never saw (BLOCKER #2) — the 66014ba3 incident recurs for the highest-traffic cohort.
- A rollback after AE-0312's migration can freeze every v2 project if the loader raises on unsupported versions (BLOCKER #3).
- AE-0311 and AE-0313 can interleave at the two-commit seam despite the stated non-interleaving AC (WARN #4); AE-0314's edits can lose to a concurrent repair (WARN #5).
- AE-0311's drift reconciler is silently defeated by AE-0315's cancellation guard when both conditions hold (WARN #6) — partial-failure split state can persist indefinitely.

**Recommendation:** Resolve the three blockers (fencing token, auto-upgrade re-validation, rollback/unsupported-version fallback) and pin the watchdog-tick ordering before any of these merge. The shared-surface tickets (AE-0312 severity model, AE-0316 lock domain, AE-0315 watchdog) need a joint integration gate, not just per-ticket unit tests.
