# AE-0028 Architecture Plan: Carousel Presentation Contract

**Status:** Validated
**Tier:** T3
**Area:** Carousel editorial workflow
**Created:** 2026-06-08
**ADR impact:** Amend ADR-007 implementation notes in
`docs/decisions/0007-consolidate-carousel-pipelines-under-deepagents.md`.
The amendment keeps the three-layer decision but records the canonical
machine-readable contract, the `skills/runtime` path, and the CI drift gate. A
new ADR is not required because JSONB remains the compatibility store for slide
extras and the orchestration decision is unchanged.

## Goal

Turn the one-off carousel corrections into a versioned, enforceable production
contract that applies to every generation, translation, refinement, render, and
publish path.

The target output is a seven-slide bilingual carousel where slides 1 through 6
use full-bleed artwork with concise lower-third copy and slide 7 uses a validated
creator card. Visible slide text must be readable, correctly capitalized, free
of decorative emoji and dash punctuation, and safe from overlap at both standard
and HD export sizes.

## Executive Findings

### P0: The active editorial agents do not receive the documented carousel rules

- `skills/carousel-pipeline/phases/content/SKILL.md` and `_shared/*.md` describe
  carousel standards, but `CarouselEditorialOrchestrator` directly constructs
  `OutlineAgent` and `ContentDraftAgent`.
- `ContentDraftAgent` uses `PROMPT_CONTENT_DRAFT` from
  `domain/constants/ai_agents.py`. That prompt only requests a free-form
  `draft_text`; it does not include the seven-slide presentation contract,
  copy budgets, English capitalization, emoji policy, lower-third layout, or
  structured extras.
- `phase_subagents.py` creates metadata listing phase skills, but that metadata
  is not injected into `OutlineAgent` or `ContentDraftAgent`.
- `build_editorial_carousel_subagent()` attaches only the first registry
  entry's skills, which are research skills. It does not create executable
  phase subagents or load content standards into content generation.
- The richer `agents/prompts/carousel/v2/content_prompt.yaml` is used by the
  legacy/template content path, not by the editorial workflow that generated
  the affected carousel.

This is the primary reason that editing skill markdown alone will not fix future
carousels.

### P0: The persisted slide model is too weak for the required contract

- Outline titles become persisted slide headings.
- `draft_text` becomes the complete slide body.
- English translation accepts arbitrary heading/body strings and silently
  returns an empty translation set on parse failure.
- There is no typed distinction between concise presentation copy and long-form
  article material.
- Blog markdown is assembled by concatenating slide bodies. Tightening slide
  copy without separating long-form content would make the blog shallow.
- `CarouselSlide.heading`, `body`, and `extras.translation_en` contain no policy
  version or validation result.

### P0: Approval gates do not enforce presentation quality

- Content approval only blocks on persona score.
- Review state exposes generic slide drafts, not the exact bilingual copy that
  will be persisted and rendered.
- Refinement writes arbitrary LLM output directly to slide fields and rerenders
  it without applying generation-time rules.
- Artifact health checks file existence, JPEG validity, dimensions, raw image
  presence, and PDF page count. It does not check copy rules, text overflow,
  lower-third placement, broken avatar assets, or image decode failures.

### P1: Renderer and exporter can create overlap even with acceptable copy

- Hero strategies render unbounded heading/body content into the same lower
  section.
- `.slide-hero-main` permits a large portion of the slide and structured grids
  can use scrollable overflow, which is unsuitable for static screenshots.
- Playwright injects separate sizing and height rules with `!important`, so
  export can override the template composition.
- Export waits a fixed four seconds instead of waiting for fonts and images to
  decode.
- `_crop_border_artifact()` does not reliably remove odd one-pixel excess
  dimensions.
- No browser geometry is measured before screenshots are accepted.

### P1: Creator branding is not an asset pipeline

- The CTA strategy hardcodes the website and a relative fallback avatar path.
- The renderer assumes the avatar already exists inside each project output
  tree.
- A valid frontend asset can therefore become a broken image in the backend
  container.
- Final health does not validate the avatar referenced by the closing HTML.

### P1: Documentation and runtime behavior conflict

- Shared skills and phase skills still contain six-slide assumptions.
- Runtime editorial outline generation requires seven slides.
- Image documentation says slides 1 through 4 receive images, while current
  Python behavior generates images for intro, summary, content, and closing,
  which means slides 1 through 6 in the canonical seven-slide structure.
- Shared contracts request emoji icons while the desired visible-slide policy
  disallows emoji and requires Lucide semantic icons.
- Prompt text describes dense paragraphs, while the hero design requires
  concise lower-third copy.
- `_shared/README.md` points to prompt `v1`, while template prompt builders
  explicitly load `v2`.

## Product Contract

Create one named and versioned contract: `hero_lower_third_v1`.

### Canonical slide structure

| Slide | Type | Image | Presentation rule |
|------:|------|-------|-------------------|
| 1 | `intro` | Required | Hook title, short subtitle, optional TLDR |
| 2 | `summary` | Required | One heading plus three compact narrative points |
| 3 | `content` | Required | One idea with one structured extra |
| 4 | `content` | Required | One idea with one structured extra |
| 5 | `content` | Required | One idea with one structured extra |
| 6 | `closing` | Required | Three or four concise actions |
| 7 | `cta` | Creator avatar | Creator identity, website, single CTA |

### Visible slide text rules

These rules apply to headings, bodies, TLDR text, feature titles and bodies,
summary points, insight text, CTA text, and attribution rendered on a slide.
They do not automatically apply to captions or blog prose.

1. No decorative emoji.
2. No em dash or en dash.
3. No ASCII hyphen used as prose punctuation. Validation parses visible
   Markdown text nodes and skips fenced code, inline code, URLs, CLI flags,
   signed numbers, package names, and identifiers. Renderer replacement is a
   final defense and may not rewrite those protected tokens.
4. English headings find the first Unicode cased alphabetic character after
   stripping Markdown emphasis, quotes, brackets, numbers, and inline-code
   delimiters. That character must be uppercase. Acronyms remain unchanged.
   Intentional lowercase brands must appear in a versioned policy allowlist;
   title-casing the complete heading is forbidden.
5. Portuguese and English slides have the same slide type, information intent,
   and structured-extra shape.
6. No drafting labels such as `SLIDE 3`, `TITLE`, `TITULO`, `BODY`, source URLs,
   or instructions to the next slide.
7. Body copy does not repeat the heading.
8. Visible text uses at most one accent-highlighted phrase. Highlighting is
   optional, not mandatory.
9. Structured icons are semantic Lucide icon names from an allowlist, not emoji
   characters. The renderer maps names to controlled Lucide SVG output.

### Initial copy budgets

Character counts are a fast preflight, not the final layout authority.

| Field | Initial limit |
|-------|---------------|
| Intro heading | 72 characters, maximum 3 rendered lines |
| Intro subtitle | 180 characters, maximum 4 rendered lines |
| Summary heading | 64 characters, maximum 3 rendered lines |
| Summary point title | 36 characters |
| Summary point body | 90 characters |
| Hero content heading | 64 characters, maximum 3 rendered lines |
| Hero content body | 220 characters, maximum 6 rendered lines |
| Feature title | 32 characters |
| Feature body | 100 characters |
| Closing action title | 30 characters |
| Closing action body | 90 characters |
| CTA creator name | 60 characters |
| CTA handle and website | 80 characters each |

The final implementation must tune these values against PT and EN fixture
renders. It must not reduce font sizes below design tokens to make invalid copy
fit.

### Geometry contract

Geometry is measured in Playwright after `document.fonts.ready`, required font
checks, and image decoding. The standard viewport is exactly 1080 by 1350; HD
is exactly 2160 by 2700. Measurements use these selectors:

- slide root: `[data-slide-number]`,
- artwork: `.slide-artwork`,
- presentation region: `.slide-presentation`,
- copy region: `.slide-presentation-copy`,
- footer: `.slide-footer`,
- CTA card: `.creator-card`,
- CTA avatar: `.closing-avatar img`.

For slides 1 through 6, `copy_start_ratio` is `0.55` for intro, `0.54` for
summary and content, and `0.52` for closing. The blocking predicates are:

```text
presentation.top >= slide.top + slide.height * copy_start_ratio - tolerance
presentation.bottom <= footer.top - footer_gap + tolerance
copy.scrollHeight <= copy.clientHeight + tolerance
copy.scrollWidth <= copy.clientWidth + tolerance
every visible copy descendant rect is contained by presentation rect + tolerance
footer rect is contained by slide rect + tolerance
```

`footer_gap` is 24 CSS pixels at standard scale and 48 at HD. `tolerance` is
1 CSS pixel at standard scale and 2 at HD. A result within 8 standard pixels or
16 HD pixels of a boundary emits `layout_near_limit` but passes; crossing the
predicate emits a blocking violation. Static slides must have computed
`overflow-x` and `overflow-y` other than `auto` or `scroll`.

Rendered line counts are authoritative. Count lines by grouping the top
coordinates returned by `Range.getClientRects()` for visible text nodes with a
1-pixel standard or 2-pixel HD grouping tolerance. Character budgets remain a
fast preflight. The exporter must require the configured heading and body font
families with `document.fonts.check`; silently falling back to another family
is a blocking `font_unavailable` error.

For all required images, `HTMLImageElement.decode()` must resolve,
`complete=true`, and `naturalWidth` and `naturalHeight` must be positive. For
slide 7, `.creator-card` must be contained inside the slide safe rectangle,
the avatar must decode, and the configured website must appear exactly once in
visible text. The CTA does not use the lower-third predicates.

## Target Architecture

### 1. Canonical contract and typed policy loader

Preserve ADR-007's three-layer model with one explicit ownership rule:

1. `skills/runtime/carousel-pipeline/contracts/hero_lower_third_v1.yaml` is the
   canonical source for rule identifiers and machine-readable values, including
   slide count, type sequence, image scope, visible-text rules, copy budgets,
   geometry ratios, font requirements, and intentional-lowercase allowlist.
2. Human-readable files under `skills/runtime/carousel-pipeline/_shared/`
   explain those rule identifiers and may not redefine numeric values.
3. `presentation_policy.py` parses the packaged YAML into immutable typed policy
   objects. Python validators and renderers are the executable enforcement
   layer, not a second value source.
4. `agents/prompts/carousel/v3/` receives a generated policy fragment from the
   typed object. Prompt templates may add instructions but may not duplicate
   policy constants.

Add `application/services/carousel/presentation_policy.py` containing:

- `CarouselPresentationPolicy`
- `SlideTypePolicy`
- `TextBudget`
- `GeometryBudget`
- `VisibleTextPolicy`
- `load_presentation_policy(version)`

The backend image packages the YAML and shared runtime instructions. Startup
fails if the configured default policy cannot be loaded or its schema is
invalid. Persist the SHA-256 of canonicalized policy YAML as
`presentation_policy_checksum`.

Add a CI contract-alignment test that loads the packaged YAML, typed policy,
generated prompt context, and shared documentation references. It fails when:

- a required rule ID is missing from shared documentation,
- prompt context omits or changes the policy version, slide count, image scope,
  visible-text rule IDs, Lucide icon allowlist, or geometry ratios,
- Python policy values differ from canonical YAML,
- packaged runtime files differ from repository runtime files,
- ADR-007 still points to the pre-separation skill path.

Add `presentation_policy_version` to `CarouselProject` and its database model.
Default new projects to `hero_lower_third_v1`. Existing projects retain a
legacy value until explicitly regenerated.

Use existing `CarouselSlide.extras` JSONB to persist:

```json
{
  "presentation_contract": {
    "version": "hero_lower_third_v1",
    "validation_status": "valid",
    "validated_at": "ISO-8601 timestamp",
    "violations": []
  }
}
```

This avoids a second slide table. Add dedicated columns only if reporting or
query performance later requires them.

### 2. Structured generation schemas

Replace free-form `draft_text` as the editorial generation contract with
Pydantic models configured with `extra="forbid"`:

- `CarouselDraftPackage`
- `SlideDraft`
- `IntroSlideCopy`
- `SummarySlideCopy`
- `ContentSlideCopy`
- `ClosingSlideCopy`
- `CtaSlideCopy`
- `FeatureItem`
- `StatItem`
- `InsightItem`
- `SlideValidationViolation`
- `SlideValidationReport`

Each `SlideDraft` includes:

- `slide_index`
- `slide_type`
- `presentation_pt`
- `presentation_en`
- `long_form_notes`
- `source_ids`
- `confidence_score`

`presentation_pt` and `presentation_en` are a discriminated union on
`slide_type`. The exact compatibility schema is:

| Type | Required | Optional | Forbidden |
|------|----------|----------|-----------|
| `intro` | `heading`, `body` | `tldr_strip` | `summary_points`, `features`, `stats`, `insight`, `actions` |
| `summary` | `heading`, `body`, exactly 3 `summary_points` | none | `tldr_strip`, `features`, `stats`, `insight`, `actions` |
| `content` | `heading`, `body`, `content_kind`, the field selected by `content_kind` | none | all unselected structured fields, `tldr_strip`, `summary_points`, `actions` |
| `closing` | `heading`, `body`, 3 or 4 `actions` | none | `tldr_strip`, `summary_points`, `features`, `stats`, `insight` |
| `cta` | `heading`, `body`, `creator_name`, `creator_handle`, `creator_website` | none | every structured-extra field |

`content_kind` is `features`, `stats`, or `insight`. `features` contains 2 to 4
`FeatureItem` values, `stats` contains exactly 3 `StatItem` values, and
`insight` contains one `InsightItem`. `actions` is a distinct field in the new
contract and is not overloaded onto `features`.

Structured items that need a visual marker use `icon_name`, not raw `icon`, with
values from the policy's Lucide allowlist, for example `chart-column`,
`book-open`, `newspaper`, `brain`, `target`, `eye`, `message-circle`,
`shield-check`, `wrench`, and `flask-conical`. Validation rejects emoji
characters, arbitrary icon strings, and unsupported Lucide names before
persistence.

PT and EN must have the same `slide_type`, `content_kind`, selected structured
field, and item count. Text values are localized independently. Structural
parity failure blocks persistence.

The canonical API returns this union. During one compatibility release,
repository adapters also serialize existing top-level keys
`features`, `stats`, `insight`, `summary_points`, and `tldr_strip`; closing
`actions` has no legacy alias. Existing rows are read through a legacy adapter
but are not rewritten until explicit regeneration. New writes use the union
shape in `extras.presentation_pt` and `extras.translation_en`, while the
existing heading/body columns remain denormalized PT compatibility fields.

`long_form_notes` preserves material for blog generation and is never rendered
directly on a slide.

Refactor blog generation to use research, outline, and `long_form_notes`.
Do not build the blog by concatenating the constrained slide body.

Move active editorial prompts out of `domain/constants/ai_agents.py` into the

backend/src/rag_backend/agents/prompts/carousel/v3/

#### Skill directory boundary

Separate production behavior skills from repository delivery skills:

```text
skills/
  runtime/
    carousel-pipeline/
      SKILL.md
      bmad-skill-manifest.yaml
      _shared/
      phases/
    carousel-refinement/
      SKILL.md
    knowledge-base/
      SKILL.md
  delivery/
    architect-skill/
    developer-skill/
    orchestrator-skill/
    planner-skill/
    qa-agent/
    release-manager-skill/
    ticket-writer-skill/
```

Classification rule:

- `runtime/` contains behavior instructions that may be loaded by the deployed
  application while serving end-user workflows.
- `delivery/` contains planning, implementation, QA, ticket, release, and
  repository orchestration instructions used only by developers and coding
  agents.
- A skill may not be present in both canonical trees.
- Every delivery skill is explicitly invoked by a slash command and declares
  `disable-model-invocation: true`. Delivery skills must not activate
  implicitly from ordinary application or coding conversation.

The backend image copies only:

```dockerfile
COPY skills/runtime/ /app/skills/runtime/
```

It must not copy `skills/delivery/`, architecture reviewer prompts, ticket
templates, QA playbooks, or release instructions.

Configure one explicit runtime root:

```text
ALTER_EGO_RUNTIME_SKILLS_ROOT=/app/skills/runtime
```

Local development resolves the same logical paths from
`<repository>/skills/runtime`; production resolves them from the configured
absolute root. Application code stores logical identifiers such as
`carousel-pipeline/phases/content`, not repository-relative paths.

Some agent clients discover only `skills/*/SKILL.md` and may not recursively
scan category directories. Preserve the current development workflow during a
staged migration:

1. Move canonical delivery skills to `skills/delivery/`.
2. Keep temporary top-level compatibility links such as
   `skills/architect-skill -> delivery/architect-skill` and
   `skills/developer-skill -> delivery/developer-skill`.
3. Keep temporary runtime links such as
   `skills/carousel-pipeline -> runtime/carousel-pipeline` only for local agent
   clients that still require the old path.
4. Update project documentation, scripts, manifests, tests, and agent
   configuration to canonical category paths.
5. Verify each supported agent client discovers nested skills or follows the
   compatibility links.
6. Remove compatibility links only in a later cleanup after all supported
   clients use canonical paths.

Compatibility links are repository conveniences only. Docker uses the exact
`skills/runtime/` copy source, so links outside that subtree cannot pull
delivery skills into the image.

#### Delivery slash-command contract

The folder migration is not complete until every delivery slash command works.
Validate this matrix:

| Command | Required result |
|---------|-----------------|
| `/planner-skill` | Loads the canonical planner skill and produces an epic/ticket breakdown without editing production code |
| `/architect-skill` | Loads plan mode and writes an architecture plan |
| `/architect-skill validate` | Loads validate mode and writes a plan-validation report |
| `/architect-skill research "<question>"` | Loads research mode and writes the documented research artifacts |
| `/architect-skill skeptical` | Loads skeptical mode or gives the documented external-review handoff |
| `/architect-skill bugfix <paths>` | Loads bugfix mode and writes a read-only bugfix design |
| `/ticket-writer-skill` | Creates or updates canonical `.agent/tasks/` files and runs ticket validation |
| `/orchestrator-skill` | Reads board state and performs only permitted delivery-state operations |
| `/developer-skill` | Loads implementation workflow, reads the active ticket, and produces the expected development handoff |
| `/qa-agent` | Loads QA workflow and produces the expected consolidated QA artifact |
| `/release-manager-skill` | Loads release workflow and produces PR/release preparation without merging |

All canonical delivery `SKILL.md` files must have:

```yaml
name: <folder-name>
disable-model-invocation: true
```

The `name` must exactly match the slash-command segment. Folder aliases and
compatibility links may not change command names.

Validation has two layers:

1. **Deterministic repository validation**
   - recursively discover canonical `skills/delivery/*/SKILL.md`,
   - parse YAML frontmatter,
   - verify unique names and folder-name equality,
   - require `disable-model-invocation: true`,
   - verify every required command is documented,
   - verify compatibility links resolve to the canonical delivery folder,
   - reject broken or circular links.
2. **Slash-command smoke validation**
   - run each command in the supported coding-agent client against a disposable
     test ticket/worktree,
   - assert the canonical skill path reported or loaded by the client,
   - assert its expected artifact or mode-specific response,
   - assert slash-only skills do not activate from a plain-language control
     prompt,
   - record client name/version and evidence in the task QA report.

Developer, QA, ticket-writer, orchestrator, and release-manager smoke tests must
use disposable fixtures and dry-run or no-op states where possible. They must
not modify a real active ticket, publish code, push branches, merge, or release.

Because slash-command discovery is client behavior, deterministic file tests
alone are insufficient. The foundational skill-migration ticket cannot pass QA
without the command matrix evidence for every delivery skill and every
architect subcommand.

The generated tasklist must make this a blocking checklist, not a narrative
note. Its foundational delivery-skill migration task must include separate,
individually checkable items for:

- frontmatter and canonical-name validation for every delivery skill,
- invocation of every slash command and architect subcommand in the matrix,
- verification that each command resolves to the canonical skill and expected
  mode,
- verification that plain-language control prompts do not activate delivery
  skills,
- compatibility-link validation during the staged migration,
- captured client name, client version, command, exit status, and evidence
  artifact for each smoke test.

The planner and QA agents must reject a tasklist that collapses these checks
into a generic "validate skills" item or omits any command from the matrix.

Update at least:

- `CLAUDE.md` delivery-skill locations,
- `scripts/architect/run_cold_critic.sh`,
- `.agent` workflow documentation,
- `phase_subagents.py`,
- carousel consolidation feature tests,
- ADR-007 path examples,
- BMAD manifest paths,
- all current `skills/carousel-*` documentation references.

Add a skill-boundary validator that fails CI when:

- a runtime manifest references `skills/delivery`,
- backend code references a delivery skill,
- required runtime files are absent,
- duplicate canonical skill names exist,
- a delivery skill name differs from its folder or slash command,
- a delivery skill omits `disable-model-invocation: true`,
- a required delivery slash command is missing from the compatibility matrix,
- a Docker image contains `/app/skills/delivery`,
- a runtime instruction file references local developer-only paths.

Container verification must build the backend image and assert:

```text
/app/skills/runtime/carousel-pipeline/... exists
/app/skills/runtime/carousel-refinement/SKILL.md exists
/app/skills/delivery does not exist
```

Add `CarouselInstructionContextLoader` that reads:

- the current phase skill,
- only the shared files named by the phase registry,
- the selected `CarouselPresentationPolicy`,
- the slide type being generated.

The loader returns a bounded system-instruction string plus a checksum. Pass
that context directly into `OutlineAgent` and `ContentDraftAgent`; do not depend
on registry metadata being interpreted elsewhere.

Required instruction hierarchy:

1. Safety and factual-source constraints.
2. Presentation policy and structured output schema.
3. Locale rules.
4. Slide-type-specific constraints.
5. Persona voice.
6. Reviewer revision notes.

Log and persist:

- prompt version,
- presentation policy version,
- instruction checksum,
- model ID,
- phase and slide number.

Remove or correct misleading phase-subagent registration. Either build actual

### 4. Deterministic validation and bounded repair

Add `carousel/presentation_validation.py`.

Validation runs:

1. after outline generation,
2. after PT/EN content generation,
3. after persona enforcement,
4. after reviewer edits and refinement,
5. immediately before render,
6. as part of final artifact health.

The validator returns stable codes such as:

- `slide_count_invalid`
- `slide_type_invalid`
- `heading_empty`
- `heading_not_sentence_case_en`
- `visible_emoji_forbidden`
- `dash_punctuation_forbidden`
- `body_too_long`
- `heading_too_long`
- `structured_extra_missing`
- `structured_extra_shape_mismatch`
- `heading_repeated_in_body`
- `drafting_scaffold_present`
- `translation_missing`
- `translation_shape_mismatch`

When LLM output fails:

1. Send the original structured object and exact violations to a repair prompt.
2. Permit one repair attempt per locale.
3. Revalidate deterministically.
4. If still invalid, set content phase to `failed` with the violation list.
5. Never persist or approve invalid presentation copy.

Translation parse failures must no longer silently return `{}`.

Normalization may remove surrounding whitespace and replace prohibited prose
punctuation. It must not guess English capitalization by title-casing every
word. English casing errors should be repaired from context so brands, acronyms,
and technical identifiers remain correct.

All waits are bounded and configurable with these defaults:

| Operation | Timeout | Retry |
|-----------|---------|-------|
| Initial outline/content/translation LLM call | 120 seconds per call | Existing provider transient retry policy only |
| Policy repair LLM call | 60 seconds per locale | No second repair attempt |
| `document.fonts.ready` plus required font checks | 15 seconds per page | None |
| Required image decode | 20 seconds per image, 30 seconds total per page | None |
| Standard or HD export | 120 seconds per locale and scale | One worker-level retry after staging cleanup |
| Creator upload decode and normalization | 10 seconds | None |

Timeout errors use stable codes and include project, slide when applicable,
locale, operation, elapsed milliseconds, and configured limit. A timeout may
never become an empty translation, partial export, or successful approval.

Persist the full structured extras for both locales. Do not reconstruct image

Update `refine_copy.py`:

- load the same policy and current structured slide,
- ask for structured replacement text,
- validate and repair,

- return validation failures to the caller instead of reporting a generic

All render-triggering changes must update a dedicated `artifact_version` after

#### Mutation, idempotency, and concurrency protocol

Reuse `carousel_projects.lock_version` and the existing `expected_version`
request contract for every content, design, creator-branding, refinement,
regeneration, and approval mutation.

1. The request reads and submits `expected_version`.
2. Candidate content is generated and validated without mutating active
   artifacts.
3. Persistence performs a compare-and-swap update where
   `lock_version = expected_version`; success increments it once.
4. Export captures that resulting source lock version and computes the
   deterministic artifact version.
5. Immediately before activation, export performs a second compare-and-swap
   requiring the same source lock version and prior `artifact_version`.
6. The winning activation increments `lock_version` once more. A loser returns
   HTTP 409 or a workflow conflict, leaves the active artifact unchanged, and
   marks its candidate build superseded.

The export idempotency key is:

```text
sha256(project_id | source_lock_version | artifact_version)
```

Add `carousel_artifact_builds` with a unique constraint on
`(project_id, artifact_version)` and fields `operation_id`,
`source_lock_version`, `status` (`staging`, `ready`, `active`, `failed`,
`superseded`), `staging_path`, `error_json`, timestamps, and attempt count.
Repeated work with an `active` build returns the existing manifest. A `ready`
build may resume activation after validation. A live `staging` build returns
accepted/in-progress instead of starting a duplicate. A failed build may retry
only after staging cleanup and an incremented attempt count.

Existing AE-0020 `generation_key` remains the only image-provider idempotency
key. Artifact builds reference image content hashes and do not introduce a
second image-generation key.

On cancellation or worker restart:

- close Playwright resources,
- mark the build failed unless a complete ready manifest exists,
- retain complete `ready` candidates for deterministic recovery,
- remove incomplete staging directories older than 24 hours,

### 8. Export preflight and manifest

Before screenshot:

1. wait for `document.fonts.ready`,
2. wait for every required image to decode,
3. evaluate geometry for each slide,
4. fail export on blocking errors,
5. capture standard and HD slides only after preflight passes.

- per-slide geometry reports,
- decoded image reports,
- policy version,
- content hash.

Use this exact project layout:

```text
<project.output_dir>/
  assets/
    images/<content_sha256>.jpg
    creators/<content_sha256>.webp
  .staging/<operation_id>/
  versions/<artifact_version>/
    pt/
      slide_1.jpg ... slide_7.jpg
      hd/slide_1.jpg ... slide_7.jpg
      carousel.pdf
    en/
      slide_1.jpg ... slide_7.jpg
      hd/slide_1.jpg ... slide_7.jpg
      carousel.pdf
    artifact-manifest.json
  current.json
```

`project.output_dir` remains the absolute project root established by AE-0023.
It never points at a staging or version directory. `artifact_version` is
`sha256-<64 lowercase hex>` over canonical JSON containing the policy checksum,
prompt version, template version, renderer contract version, exporter contract
version, normalized PT and EN union payloads, raw image hashes, creator asset
hash, theme and design tokens, and export dimensions. Timestamps and filesystem
paths are excluded.

Write `artifact-manifest.json` inside the version directory with:

- project ID,
- artifact version,
- policy, prompt, template, renderer, and exporter versions,
- PT and EN source hashes,
- expected slide numbers,
- raw image hashes,
- avatar hash,
- standard and HD slide hashes and dimensions,
- PDF paths and page counts,
- layout validation reports.

Promotion ordering is fixed:

1. Create `.staging/<operation_id>` on the same filesystem as `versions/`.
2. Render every PT/EN standard and HD slide, both PDFs, and the manifest.
3. Validate hashes, counts, dimensions, image decode reports, and geometry.
4. `fsync` written files and the staging directory where supported.
5. Atomically rename staging to `versions/<artifact_version>`. If an identical
   valid version exists, discard staging and reuse it.
6. Activate with the `lock_version` compare-and-swap described above and commit
   `artifact_version`.
7. Atomically replace `current.json` using write-to-temp plus `os.replace`.

Media, preview, publish, download, and PDF routes resolve the active directory
from the database `artifact_version`; `current.json` is an operational index,
not the transactional authority. Therefore:

- rename success plus DB failure leaves an unreferenced immutable version that
  is never served and is eligible for cleanup,
- DB success cannot precede rename, so the database cannot point to an
  unpromoted directory,
- `current.json` failure after DB commit emits a repair event but does not break
  serving; a reconciler rewrites it from the database.

For projects with null `artifact_version`, routes use the current legacy
`<output_dir>/pt`, `<output_dir>/en`, and `<output_dir>/images` layout. No
automatic move occurs. AE-0023 remains responsible for normalizing the project
root; AE-0028 creates version directories below that root.

Retention keeps the active version, the immediately previous version, and any
version younger than 30 days. A daily cleanup removes unreferenced older
versions and incomplete staging directories older than 24 hours. Rollback is a
compare-and-swap update to a retained manifest-valid version followed by
`current.json` reconciliation; it never renames mutable directories over the
active version.

Fix border cropping by calculating an exact target box:

```text
left = max(0, (width - target_width) // 2)
top = max(0, (height - target_height) // 2)
right = left + target_width
bottom = top + target_height
```

Any undersized image or mismatch beyond the tolerated border artifact fails
instead of being silently accepted.

The UI must never resolve files from `.staging` or an unreferenced version.

Add `carousel_creator_assets` with `id`, `owner_id`, `content_sha256`,

`carousel_projects.creator_asset_id` as a foreign key. Only the project owner
or an admin may upload, select, replace, or remove a creator asset.

The authenticated upload endpoint accepts one multipart file and enforces:

- maximum request and decoded file size of 5 MiB,
- MIME allowlist `image/jpeg`, `image/png`, and `image/webp`,

- maximum decoded dimensions 4096 by 4096 and maximum 16 million pixels,
- Pillow decompression-bomb warnings are errors,
- animation is rejected,

metadata endpoint, redirect, DNS-rebinding, and path-traversal SSRF classes are
eliminated at the boundary. A future remote importer requires a separate
security review and is outside this plan.

but the asset cannot be staged or decoded, final export fails with a precise
error. It must never emit a broken `<img>` and continue.

known application-owned files under configured roots into managed assets.

writes reject `creator_avatar_url`.

  with the mismatched rule identifiers.
- [ ] WHEN the backend image is built THE IMAGE SHALL contain
  `/app/skills/runtime` and SHALL NOT contain `skills/delivery`.
- [ ] WHEN delivery skills are validated THE tasklist SHALL prove every
  documented slash command and architect mode loads canonically, every
  frontmatter name matches, implicit activation is disabled, and the
  plain-language controls remain inactive.
- [ ] WHEN a seven-slide draft is generated THE SYSTEM SHALL return matching PT
  and EN discriminated unions plus separate long-form notes.
- [ ] WHEN visible copy or structural parity is invalid THE SYSTEM SHALL perform
  at most one bounded repair per locale and SHALL block persistence if the
  result remains invalid or times out.
- [ ] WHEN structured slide markers are generated THE SYSTEM SHALL persist only
  allowlisted Lucide `icon_name` values, SHALL reject emoji/raw icon strings,
  and SHALL render icons through controlled SVG markup.
- [ ] WHEN content is approved or refined THE UI SHALL submit the displayed
  union payload with `expected_version` and SHALL reject stale or invalid edits.
- [ ] WHEN slides 1 through 6 render THE Playwright preflight SHALL satisfy the
  exact standard and HD geometry, line, font, overflow, and image predicates.
- [ ] WHEN a creator asset is uploaded or selected THE API SHALL require owner
  or admin access, accept only the managed upload contract, and reject URLs,
  paths, invalid formats, oversized images, and decompression bombs.
- [ ] WHEN artifact input is unchanged THE SYSTEM SHALL reuse the same valid
  artifact build without duplicate provider or rendering work.
- [ ] WHEN concurrent mutation or activation loses a compare-and-swap THE
  SYSTEM SHALL return a conflict and SHALL leave the active artifact unchanged.
- [ ] WHEN export, decode, font loading, PDF generation, or activation fails THE
  prior active artifact SHALL remain readable and no staging file SHALL be
  served.
- [ ] WHEN export succeeds THE SYSTEM SHALL activate one immutable
  `sha256-<64 hex>` version whose manifest hashes, counts, dimensions, policy,
  images, avatar, geometry, and PDFs all validate.
- [ ] WHEN `current.json` is missing or stale THE media routes SHALL resolve the
  database artifact version and the reconciler SHALL repair the index.
- [ ] WHEN a legacy project has null `artifact_version` THE dual-read routes
  SHALL continue serving its existing root layout without forced regeneration.
- [ ] WHEN release QA runs THE recorded commands, mutation targets, machine
  gates, slash-command evidence, and PT/EN contact sheets SHALL all pass.

## Non-Goals

- Replacing the current image provider registry.
- Regenerating all historical carousels automatically.
- Shipping development workflow skills inside the backend container.
- Applying the no-emoji rule to Instagram captions or blog prose.
- Using OCR as the primary correctness check. Source validation and browser
  geometry are deterministic and should remain authoritative.
- Shrinking typography dynamically until arbitrary copy fits.

## Decision Log

1. The packaged runtime YAML owns rule identifiers and values; shared skill
   markdown explains them, generated prompt context reflects them, and typed
   Python code enforces them.
2. ADR-007 is amended, not superseded, because its three-layer alignment remains
   intact.
3. Slide presentation copy and blog source material are separate outputs.
4. Locale correctness is validated independently for PT and EN.
5. Export must fail on layout or asset breakage before producing publishable
   artifacts.
6. Existing JSONB slide extras remain the storage location for rich localized
   structures and validation metadata.
7. Runtime and delivery skills have separate canonical folders; production
   images contain runtime skills only.
8. Top-level skill links are temporary discovery compatibility, not canonical
   storage.
9. Delivery workflows are slash-only and their command matrix is a blocking QA
   gate for the folder migration.
10. `project.output_dir` remains the stable root; immutable artifact versions
    live below it and the database artifact version is serving authority.
11. Creator avatars use authenticated managed uploads only; API URL and local
    path ingestion are rejected.
12. Existing `lock_version`, `expected_version`, and AE-0020 generation keys are
    reused rather than replaced by parallel concurrency mechanisms.
