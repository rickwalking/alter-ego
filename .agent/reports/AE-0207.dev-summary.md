### Summary

Stopped the prod schema-drift failure class. Prod was bootstrapped via SQLAlchemy
`create_all` (which creates missing tables but NEVER `ALTER`s existing ones) and
never ran Alembic, so every model column added after its table existed silently
500'd prod on first reference (hit live twice: `caption_en`, then `origin` +
`distribution`). This ticket: (1) makes `deploy.yml` run `alembic upgrade head`
before the app serves traffic, (2) adds a reusable schema-vs-models drift detector
that compares `Base.metadata` columns against the live `information_schema.columns`
and fails when a mapped column is absent — wired as the `schema-drift` backend gate
and as a deploy step, (3) records the policy in ADR-012 (accepted), and (4)
documents the one-time operator `alembic stamp`/reconcile for the existing drifted
prod (an audited manual step — NOT executed by CI or this ticket).

### Drift-gate design
Pure comparison (`find_missing_columns`: `Base.metadata` columns vs live columns)
is separated from I/O (`reflect_live_columns` reads `information_schema.columns`)
so the algorithm is unit-tested without a DB. A table the ORM maps but the DB
lacks entirely reports every column as missing (same 500 class). Extra DB columns
the ORM does not map are NOT drift (additive-only concern). The `check_drift_cli`
entrypoint imports the models package (via `importlib`, no `# noqa`) to register
tables, builds a sync engine from `DATABASE_URL`, and returns exit 0 (no drift) /
1 (drift) / 2 (no URL). Gate is Postgres-dependent → SKIP locally without
`DATABASE_URL`, runs in CI (added to `CHANGED_ONLY_SKIP`).

### Seeded test
`test_mapped_column_missing_from_live_db_fails` (live Postgres): creates a probe
table, drops a column, then maps an extra `origin` column the live schema lacks
(mirrors the real prod failures caption_en/origin/distribution); asserts
`report.has_drift is True` and the missing columns are reported. Paired with
`test_matching_live_schema_passes` (no drift). Plus 8 unit tests on the pure core.

### Operator runbook (one-time prod stamp/reconcile — DO NOT run from CI)
Full copy-pasteable version: `docs/deployment/prod-migration-reconcile.md`. Run
ONCE on the prod droplet before the first migrate-on-deploy ships. Summary:
1. **Backup** prod (`pg_dump` to a timestamped file).
2. Confirm no `alembic_version` yet:
   `... exec -T postgres psql -U "$POSTGRES_USER" -d "$POSTGRES_DB" -c "SELECT to_regclass('public.alembic_version');"`
   (`null` → continue; a table name → already reconciled, stop).
3. Run the drift CLI to decide the stamp target:
   `... run --rm --no-deps backend python -m rag_backend.infrastructure.database.check_drift_cli`
   - **OK (matches head)** → `... run --rm --no-deps backend alembic stamp head`.
   - **Drift reported** → `... run --rm --no-deps backend alembic stamp 63eaefa67b8c`
     then `... run --rm --no-deps backend alembic upgrade head`.
4. If `upgrade head` fails on a migration whose object prod ALREADY has
   (hand-patched origin/caption_en/distribution): `... alembic stamp <rev>` for
   exactly that revision, then resume `... alembic upgrade head`. Revision order:
   `63eaefa67b8c → a1b2c3d4e5f6 → b2c3d4e5f6a7 → c3d4e5f6a7b8 → d4e5f6a7b8c9 → e5f6a7b8c9d0`.
5. Verify: `... alembic current` shows head AND the drift CLI prints
   `OK: live schema matches all mapped ORM columns.`
After this, every deploy migrates + drift-checks automatically (ADR-012).
