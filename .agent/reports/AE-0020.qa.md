## Edge Cases
- **Duplicate generation key**: Upsert prevents duplicates
- **Matching metadata with missing image file**: Record not reused; triggers regeneration
- **Matching metadata with corrupted image file**: Record not reused; triggers regeneration
- **Legacy file with no DB record**: Backfilled with computed metadata
- **Failed generation**: Recorded with generation key for retry tracking
- **Partial metadata match**: Treated as new generation; old record preserved for audit
- **Concurrent writes**: Repository handles upsert atomically

## Known Findings
- `file_sha256()` loads entire image into memory via `Path.read_bytes()` — potential OOM for very large images, consider streaming hash
- Standalone recovery/backfill CLI tooling not yet implemented (remaining scope)
- Richer provider response metadata (e.g., provider image IDs) not yet captured
