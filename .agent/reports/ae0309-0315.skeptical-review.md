# Cold Critic Review

## Verdict
WARN

The set is well-motivated and the prod-incident root causes are real (all three bug claims I verified against the code). But the packet carries **one correctness defect that would re-create the class of incident it is fixing** (AE-0311's non-atomic "dual-write" + a TOCTOU in-progress guard), plus **one materially under-scoped ticket** (AE-0312 introduces a severity tier it claims already exists), and **one operational dead-end** (AE-0315 surfaces `in_progress` with no staleness reaper, against a watchdog that explicitly excludes `in_progress`). Address these before implementation.

## Findings

### [BLOCKER] AE-0311: the "dual-write" to checkpoint + projection is NOT atomic, and the in-progress guard is a TOCTOU read, not a CAS
- **Assumption (in packet):** the repair endpoint writes repaired values "to the workflow checkpoint … **and** to the `carousel_slides` projection **in the same operation**" and returns "409 when the workflow is `in_progress` (never race a live run)."
- **Risk (verified):** the LangGraph `AsyncPostgresSaver` owns its own connection pool (`checkpointer.py:36-49`); every `carousel_slides` write commits on a **separate** SQLAlchemy session (`carousel_repository.py:192`). There is **no** shared transaction anywhere — zero `aput_writes` usage; the only external checkpoint mutation path is `aupdate_state` (`editorial_workflow_feedback.py:40-64`), which appends a new checkpoint record on the saver's connection. So a repair is **two commits**, not one. A crash, OOM, or deploy between the two commits leaves checkpoint state repaired but projection stale (or vice-versa) — i.e. a *new* source of the exact checkpoint↔projection drift that caused incidents `38affb3e` and `66014ba3` and that this ticket exists to eliminate. Worse, `resolve_presentation_review_from_state` re-derives validation from checkpoint state on every read (`presentation_review.py:317-332`), so a half-repaired project can display contradictory validation in the client vs. the served PDF.
- **Compounding risk (verified):** "never race a live run" has no enforcement. The resume path closes its check→start race with the `lock_version` CAS (`bump_resume_lock_version` → `optimistic_lock_service.py:75-103`, atomic `WHERE lock_version=?` UPDATE). AE-0311 reuses `ensure_resume_not_in_progress` — which is only a **READ** of `phase_status` (`editorial_workflow_routes_validate.py:105-122`) with the same TOCTOU window the resume path had to add a CAS to close. A workflow can start **between** the repair's in-progress read and its dual-write, and the running workflow's next super-step supersedes the external `aupdate_state` repair (no saver-level guard; documented hazard at `carousel_workflow_engine.py:110-114`). The packet provides no CAS/lock for the repair path.
- **Impact:** a slow healthy run + an impatient "Fix issues automatically" click (the exact user behavior AE-0315 documents) corrupts state deterministically. This is the same incident pattern, reintroduced as a feature.
- **Suggested mitigation:** (a) make the repair write **projection-first, checkpoint-derived**: write `carousel_slides` in a transaction and write checkpoint via `aupdate_state` only after commit, with a reconciliation check; OR accept non-atomicity explicitly and add a drift detector (see cross-ticket finding). (b) The repair endpoint must `bump_resume_lock_version` (take the same CAS) before mutating, so a concurrent resume fails the CAS instead of racing. (c) Write a `.feature` with "repair concurrently with a started resume" — the current Gherkin has none.
- **Open question for author:** what is the post-repair consistency guarantee between checkpoint and projection, and what reconciles them on partial failure? Is "single operation" a contract or aspirational wording?

### [BLOCKER] AE-0312: "warning-level (non-blocking)" is net-new model behavior, not an additive rule — and it silently couples to AE-0309's fail-closed chain
- **Assumption (in packet):** casing rules are "Severity: warning-level (non-blocking) by default … casing must not dead-end a workflow the way blocking rules can (AE-0310)." Delta labels this "MODIFIED: presentation policy version … otherwise additive rules" and "Frontend: no new components."
- **Risk (verified):** the validation model has **no severity tier**. `SlideValidationViolation` has no `severity` field; `build_validation_report` (`presentation_validation.py:291-309`) produces exactly two outcomes — empty→`blocking=False`, any violations→`blocking=True` — and `validate_localized_slides` always passes `blocking=True` (`presentation_review.py:102`). So "warning-level" requires defining a severity enum, threading it through `build_validation_report`, *and* changing the blocking decision (`_blocking_from_repaired_localized`/`_blocking_from_drafts`, `presentation_review.py:335-362`) to ignore warnings. That is a model change, not "additive rules."
- **Cross-ticket coupling (verified, undeclared):** AE-0309 makes the content phase **fail closed** on *any* blocking violation. Once casing is "warning," AE-0309's validate→repair→retry→blocking-report chain must **distinguish** scaffold/body-blockers from casing warnings to avoid (i) re-running the LLM retry for a casing warning, or (ii) letting a casing warning block at content review. AE-0309's Gherkin and AC never mention casing warnings; AE-0312 lists AE-0309 only as "Related." If AE-0309 ships first (it blocks AE-0311), every PT carousel will fail-closed on lowercase "claude" until AE-0312 lands the warning tier — i.e. AE-0309 as written *introduces* the casing dead-end AE-0312 is meant to prevent.
- **Impact:** AE-0312 is under-scoped by a full validationtier + a coupling the set doesn't declare; "Frontend: no new components" is false (warnings need distinct visual treatment vs. blockers in the panel that today renders all violations as blocking — confirmed there's **no** shared violation panel; the inline list is in `content-phase-review.tsx:124-146` only).
- **Suggested mitigation:** split AE-0312 into (i) the severity model + blocking-decision refactor (must land before AE-0309's fail-closed ships, or AE-0309 must gate casing warnings out by rule code), and (ii) the PT casing rules + `repair_casing` (additive once the tier exists). Add a cross-ticket invariant: "AE-0309's fail-closed chain ignores warning-severity violations."
- **Open question for author:** which ticket guarantees a PT carousel with lowercase "claude" does **not** interrupt at content review? Today no ticket does.

### [BLOCKER] AE-0315: `run_overdue` + reload-reconstruction create a permanent "stuck" banner with no reaper
- **Assumption (in packet):** "Page reload during a run reconstructs the banner from workflow state (no dependency on having witnessed the SSE start event)" and "A run exceeding the watchdog threshold emits `run_overdue`."
- **Risk (verified):** the existing watchdog `auto_reject_stuck` **explicitly excludes `in_progress`** from auto-rejection (`workflow_timeout_repository.py:46-52`), and the only in-flight stuck detector (`_detect_resume_stuck`) runs **inside the background resume task** — if that task dies (container restart, OOM-kill, deploy during a ~15-min run), nothing reaps it. There is **no `run_started_at`** column today — only `updated_at`, overwritten on any row touch. If `run_started_at` is intended to live in checkpoint state (AE-0315 says "Database: none"), the **watchdog tick reads DB rows** (`workflow_timeout_repository.py:88`) and would have to load workflow state per in_progress row per tick to compute elapsed minutes — a new per-tick state-read load the plan doesn't cost.
- **Worst case:** backend restart mid-run leaves `phase_status=in_progress` with no live run and no `run.finished` ever firing. AE-0315's reload-reconstruction AC says the banner is rebuilt from state — so **every** page load across all steps shows an in-progress banner with actions disabled, indefinitely, and the watchdog's `run_overdue` only tells *operators* (logs), never the *user* — the UI has no "this looks stuck, clear it" affordance. This is strictly worse than today's bare 409, because today the buttons at least fail-fast; under AE-0315 they're permanently disabled with no recovery path in the packet.
- **Impact:** AE-0313 deploys flip the server `.env` and restart the container (per CLAUDE.md deploy note) — a deploy landing during a ~15-min run produces exactly this stuck state. The packet surface this edge ("backend restart mid-run leaving stale in_progress") under QA edge cases but assigns it no fix.
- **Suggested mitigation:** (a) AE-0315 must extend the watchdog to *reap* `in_progress` rows whose `run_started_at` (DB column, needs migration — contradicts "Database: none") exceeds the threshold by resetting to `awaiting_human` and emitting `run.finished(stale)`; (b) frontend banner needs a "stale > N min → offer re-check / re-resume" escape; (c) add a `run_started_at` DB column (a migration, not "none") so the watchdog doesn't load state per tick.
- **Open question for author:** what clears an `in_progress` banner after a backend restart killed the run? No ticket in the set does.

### [WARN] Cross-ticket: three (soon four) writers to slide copy with no reconciliation — completed-project edits leave checkpoint validation stale
- **Assumption:** AE-0310 (design-phase edits via resume), AE-0311 (repair dual-write), AE-0314 (completed-project direct `carousel_slides` edit + republish) all mutate slide copy and trust `resolve_presentation_review_from_state` for validation.
- **Risk (verified):** AE-0314's completed-project edit endpoint writes **projection only** ("update `carousel_slides` … no workflow-node changes") with no checkpoint update. But `resolve_presentation_review_from_state` re-derives the validation report from **checkpoint state** on every read (`presentation_review.py:317-332`). After a completed edit, the checkpoint still holds the **old** copy and **old** validation, so any state read returns stale (potentially still-blocking) violations over fixed content — the client could show a blocking violation panel for copy the PDF already reflects as correct. The same class of silent divergence the incidents were about.
- **Suggested mitigation:** define "for `completed` projects, projection is the source of truth and state-derivation returns no blocking report" — or have AE-0314's endpoint also `aupdate_state` the checkpoint copy + validation. State which is canonical per status today is **not pinned** anywhere in the set.
- **Open question for author:** after AE-0314 edits a completed carousel, what returns `presentation_validation` to the client, and does it reflect the edit?

### [WARN] AE-0313: "existing CAS/lock semantics" for `another build in flight` is asserted but not shown; the completed-project `phase_status` is unguarded
- **Assumption (in packet):** republish returns "409 … when a workflow run or another build is in flight (existing CAS/lock semantics)" and "only for `completed` projects."
- **Risk (verified):** I found **no** advisory lock or CAS around `build_and_activate`/`carousel_artifact_builds`; content-addressing dedupes **outputs** but two concurrent `build_and_activate` could both flip `current.json`/`artifact_version`. Also, a `completed` carousel can still have `phase_status` values the in-progress guard keys on — the set never pins what `phase_status` a completed carousel holds. If AE-0314 edits chain republish while a (rare) workflow run is mid-flight on the *same* completed project, the in_progress-409 for republish must come from somewhere, and the dual-idempotency ("unchanged content re-activates") must not race a build activation.
- **Suggested mitigation:** add a `republish` advisory lock keyed by `project_id` (or a `build_in_progress` CAS on `carousel_artifact_builds`), and pin the `phase_status`↔`completed` invariant in a Gherkin scenario. Don't inherit the resume guard's semantics without re-deriving them for a completed state.
- **Open question:** is `phase_status` ever `in_progress` on a `completed` carousel? If yes, republish + resume can both be valid concurrently and need a lock; if no, the "409 on in_progress" AC is vacuous.

### [WARN] Ordering: AE-0311's design-step "Fix issues" button has no host to attach to without AE-0310
- **Assumption:** AE-0311 lists AE-0310 as "Related," not blocked-by, and ships its button at "content step, design step, and the publish page."
- **Risk (verified):** the design step **renders no violations today** (`design-phase-review.tsx` is read-only, show `design_applied`/outline/drafts); the inline violation list exists **only** in `content-phase-review.tsx:124-146`. AE-0311's AC "the button appears with any violation panel" at the design step therefore **depends** on AE-0310's design-step violation panel landing first or co-shipping. Marking it "Related" understates the ordering constraint; shipping AE-0311 first leaves a dead UI affordance.
- **Suggested mitigation:** make AE-0311 `blocked_by: AE-0310` for the design-step surface, or split AE-0311 into the endpoint (unblocked) and the design-step button (post AE-0310).

### [WARN] AE-0310: send-back attribution to target phase is asserted but never defined — it crosses AE-0315's typed 409
- **Assumption (in packet):** "Revision cap accounting attributes the revise to the phase that actually re-runs."
- **Risk:** the cap is enforced per-phase (`DEFAULT_REVISION_CAP_PER_PHASE=5`). A design→content send-back consumes **which** phase's budget — content (target, surprising) or design (current, where the user clicked)? AE-0315 then must tell the user which phase's cap was hit in its typed revision-cap 409. Neither ticket defines the rule; AE-0310 asserts an outcome ("attributed") without the accounting logic, and AE-0315 surfaces it.
- **Suggested mitigation:** add an AC to AE-0310 stating the exact counter incremented, and an AE-0315 scenario "send-back to content exhausts content cap; 409 names content phase."

### [INFO] AE-0312: policy versioning rule is ambiguous against the codebase's flat (not folder-based) policy layout
- **Note (verified):** AE-0312's Delta says "new version folder per prompt/policy versioning rules … otherwise additive rules." But the policy is **flat** file `hero_lower_third_v1.yaml` in `contracts/` with a singleton supported set (`presentation_policy.py:10-12`); **not** folder-versioned like prompts. Modifying `v1.yaml` to add PT casing violates the "never modify existing" rule; shipping `v2` means adding `hero_lower_third_v2` to `SUPPORTED_PRESENTATION_POLICY_VERSIONS` and routing new carousels to v2. The packet defers ("if the policy is versioned") and could ship an un-reproducible policy edit. Decide explicitly.

## Missing evidence
- **AE-0311:** proof that checkpoint + projection can be written atomically (they cannot, per code) — or an explicit reconciliation/drift contract on partial failure.
- **AE-0311:** a concurrency `.feature` (repair vs. concurrent resume / concurrent repair).
- **AE-0315:** where `run_started_at` is persisted (DB column ⇒ migration ⇒ "Database: none" is wrong; checkpoint state ⇒ watchdog can't read it per-tick without new load).
- **AE-0315:** the reaper that clears an `in_progress` banner after the owning task dies.
- **AE-0313:** the advisory-lock/CAS around `build_and_activate`; the invariant tying "completed `phase_status`" to "no run in flight."
- **AE-0314:** the canonical-source rule for `completed` projects (projection vs. checkpoint) and what `resolve_presentation_review_from_state` returns post-edit.
- **AE-0310:** the exact revision-counter attribution rule and a Gherkin asserting it.
- **AE-0312:** the severity model is not additive — needs a design note that `SlideValidationViolation.severity` and the blocking decision refactor are in-scope.
- **Per-ticket rule-fires tests** exist as ACs (good), but none covers the *concurrency* or *partial-write* paths above.

## Residual risks if plan proceeds unchanged
- AE-0311 as written **re-introduces** checkpoint↔projection drift under crash or concurrent-run — the failure class the whole packet is meant to eliminate. Highest-severity residual.
- AE-0312's "non-blocking" assumption **can't be satisfied** without a severity-tier refactor; shipping AE-0309 first fail-closes on casing warnings, the exact UX AE-0312 exists to prevent.
- AE-0315 leaves users with a **permanently disabled** create flow after a backend restart mid-run, no UI recovery, no auto-reaper — strictly worse than today's failing-but-fast bare 409.
- AE-0314 completed edits + AE-0310 design edits + AE-0311 repair + resume edits = **four slide-copy writers** with no declared source-of-truth per project status; silent divergence is now reachable through normal UI paths, not just manual surgery.
- AE-0313 republish under concurrent invocation has **no shown lock**; `current.json`/`artifact_version` flips are not provably serialized.
