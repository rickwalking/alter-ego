# Cold Critic Review

## Verdict
PROCEED_WITH_CAUTION

## Findings

### [WARN] Uniform-404 property is only as strong as the id-only resolution it rests on — but slug is "deferred" with an unenforceable guard
- Assumption: Deferring slug resolution and gating the follow-up on "no slug may take a UUID shape" preserves the uniform-404 property.
- Risk: The guard is a *future* ticket with a *future* test. There is no artifact in this packet proving the guard is implementable — i.e., no check that the existing `slug` values (if any rows already exist) are UUID-free. If even one legacy row has a UUID-shaped slug, the deferred slug resolution reintroduces the exact existence oracle v1 was built to avoid, and the "deferred" framing makes it feel low-risk when it is the core security property.
- Impact: Re-opens the existence/lifecycle disclosure the whole public design rejects; and it ships silently later because the risk was declared "resolved" now.
- Suggested mitigation: In *this* contract, add a data assertion (or at least a verified finding) on the current `slug` population: count UUID-shaped slugs among existing rows. If zero, document it as the precondition; if nonzero, the deferral is unsound and slug resolution must either ship now with an explicit id-then-slug precedence and a collision test, or be dropped from scope.
- Open question for author: Has anyone queried prod for UUID-shaped slugs, or is "defer with a guard" aspirational on the data state?

### [WARN] Carousel-fallback "no new exposure" rests on an unverified assumption about the carousel-feed schema staying lean forever
- Assumption: The existing public carousel feed (`GET /api/carousels?public=true`) never contains any of the `blog_posts` excluded keys, so falling back to it on public-endpoint error "surfaces nothing not already public today."
- Risk: The recursive-leak test asserted for the *blog* lean schema is explicitly extended to the carousel-fallback payload — good. But that test is a *snapshot of today's* carousel schema. The carousel feed is a *separate* evolving endpoint (AE-0299, AE-0296 adjacent tickets, future carousel work). If a future carousel change adds an editorial/AI-internal field to `GET /api/carousels?public=true`, the AE-0297 fallback test would catch it *only if AE-0297's test still runs* — and AE-0297's test asserts a *blog-specific* excluded-key set, not the carousel endpoint's own contract. There is no ticketed owner asserting "the public carousel endpoint stays lean" *as a property of the carousel endpoint*; the invariant is borrowed, cross-cutting, and tested from the wrong module.
- Impact: A carousel PR widens the fallback leak surface without tripping any carousel-side gate; AE-0297's test catches it incidentally or not at all (if the excluded-key list diverges from what the carousel endpoint can carry).
- Suggested mitigation: Either (a) add an equivalent recursive-leak test owned by the carousel public endpoint itself (not AE-0297), so the invariant is defended where it lives; or (b) explicitly scope the AE-0297 fallback test to assert the *intersection* of the blog excluded-keys and the carousel payload, and document that the carousel feed's own safety is an external precondition, not a result of AE-0297. The packet currently blurs which is true.
- Open question for author: Who owns the "public carousel endpoint stays lean" invariant after AE-0297 merges — AE-0297's test or the carousel endpoint's own suite?

### [WARN] "DELETE 409 for carousel-origin" + "carousel deletion is the canonical erasure path" creates an untested cross-ticket ordering hazard at deploy time
- Assumption: Deleting a carousel project flips its orphaned `origin==carousel` blog row to `draft` (per ADR-0013) and/or sets `project_id` NULL via `ON DELETE SET NULL`, and either makes the row hard-deletable — so the 409 is never a dead end.
- Risk: The 409 guard (AE-0296) and the "parent-carousel deletion flips orphan to draft" flow (ADR-0013) are described as coordinated but live in *different tickets with different rollback units and no stated deploy order*. ADR-0013 says the flip happens "as part of that flow"; AE-0296 ships the guard. If AE-0296 lands and the carousel-deletion flip is not yet implemented, then the 409 blocks casual delete while the carousel-deletion path *leaves a phantom published row* — exactly the degraded default-theme content ADR-0013 warns against, now *guaranteed* by the guard that was meant to prevent orphans. The two changes are mutually load-bearing.
- Impact: A mid-rollout window where carousel-origin posts are un-deletable AND deleting their parent leaves them degraded-public. Worse than either change alone.
- Suggested mitigation: State an explicit deploy sequence in the packet: the carousel-deletion→flip-to-draft change must go out **before or atomically with** the AE-0296 409 guard, with an integration test that deletes a carousel project whose blog row is currently `published` and asserts the row is `draft` (not public, not orphaned) afterward. If the flip is already implemented, cite its location; if not, treat the 409 as blocked-on, not coordinated-with.
- Open question for author: Is the carousel-deletion→orphan-flip already landed, or is it part of AE-0296/AE-0297? The packet implies it exists ("deliberate, not silent") but no file:line is cited for the flip itself (only `models/blog_post.py:28-32` for the `ON DELETE SET NULL`, which is the *opposite* of a flip to draft).

### [INFO] `unpublish` If-Match "ideally" vs "committed" hedging
- Assumption: AE-0296's "unpublish concurrency (committed)" says If-Match on unpublish is in scope, but the API contracts section earlier only lists DELETE If-Match as "added."
- Risk: Minor — the body is internally inconsistent on whether unpublish If-Match is committed ("in scope for this ticket") or aspirational. Not a material leak (unpublish is a status flip, worst case is a lost update on visibility not content), but the hedged wording invites an implementer to drop it.
- Impact: Low; unpublish clobbering a concurrent edit is a visibility race, not a content-corruption race.
- Suggested mitigation: Strike "ideally" from the plan; the body already commits it. Make all three mutations (PUT/DELETE/unpublish) carry If-Match in the .feature.
- Open question for author: None — the intended state is clear; just remove the hedge.

### [INFO] Rate-limit bound ≤120/min is asserted at "fires at/under the bound" — enumeration cost is bounded per-IP, not per-attacker
- Assumption: A per-IP rate limit (≤120 req/min/IP) bounds "full-corpus list scraping and id enumeration."
- Risk: The bound is per-IP, so a rotating-IP / distributed scraper is not bounded by this control. The plan asserts the *enforcement* is testable (good) but the *enumeration-cost claim* in the rationale ("this bounds full-corpus list scraping") overclaims what a per-IP limit bounds.
- Impact: Low for v1 (no auth = identity-bound limits would require the avoided `get_optional_user`); the claim's wording, not the control, is the issue.
- Suggested mitigation: Reword the rationale to "bounds per-IP enumeration/scraping; distributed scraping remains an out-of-band concern (WAF/bot-defense follow-up, not auth-bearing)." Keeps the control honest without weakening it.
- Open question for author: None.

## Missing evidence
- Current `slug` population: count of UUID-shaped slugs among existing `blog_posts` rows (precondition for the deferred-slug guard being implementable).
- File:line for the carousel-deletion→orphan-flip-to-draft implementation (ADR-0013 frames it as existing-and-deliberate; AE-0296/AE-0297 don't cite where it lives; if unimplemented, the 409-deploy-order hazard is real).
- Ownership of the "public carousel endpoint stays lean" invariant (to resolve Finding-2B's cross-module test gap).
- Whether `unpublish` endpoint currently honors `If-Match` (the plan says "if it does not yet honor it, adding it is in scope" — but "verified" is asserted for the status-flip effect, not for If-Match honoring; the implementer needs to know if it's a new check or a no-op).

## Residual risks if plan proceeds unchanged
- A legacy UUID-shaped slug silently breaks the uniform-404 property when slug resolution ships in the deferred follow-up.
- A future carousel-PR widens the public carousel feed into a leak via the AE-0297 fallback, with no carousel-side gate catching it.
- A deploy-order gap between the 409 guard and the carousel-deletion→flip-to-draft flow produces a window where published carousel-origin posts are both un-deletable and orphanable-while-public.

No blockers remain. The three WARNs are addressable with evidence citations or a stated deploy sequence; none requires redesigning the two-endpoint split or the lean-schema approach, both of which are sound.
