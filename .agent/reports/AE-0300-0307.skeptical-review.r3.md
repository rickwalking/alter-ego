# Cold Critic Review

## Verdict
WARN

## Findings

### [WARN] Cloudflare-range allow-list refresh can silently strand the origin
- Assumption: The scheduled CF-range fetch script updates the DO firewall and a staleness alert will catch a failed fetch.
- Risk: Cloudflare rotates/grows its egress ranges periodically. If the fetch fails *partially* (returns a subset, or a stale cached response) and the script applies it idempotently, the live Cloudflare egress nodes fall outside the new allow-list before any staleness alert fires — the origin refuses 443 for real users. "Schedule + staleness alert" is reactive, not preventive; the failure mode is "site goes dark for customers," and the alert notifies *after*.
- Impact: Production outage with no app-level cause; the admin path (22) survives but the site is down until manual intervention.
- Suggested mitigation: The apply step should be *additive*: only add new CF ranges, never remove ranges not present in the fetch, OR require a diff-approval gate before shrinking. At minimum the script must refuse to apply a fetch that is smaller than the last-known-good set (a monotonic-growth guard) and alert on the refusal.
- Open question for author: What happens to live traffic if the scheduled fetch returns a *subset* of the real Cloudflare range — does the script shrink the allow-list in place, or refuse?

### [WARN] AE-0300/AE-0305 SSH admin-IP allow-list assumes a stable admin IP
- Assumption: The admin IP is stable enough to allow-list on the SSH firewall object (AE-0300) and put in fail2ban `ignoreip` (AE-0305).
- Risk: The packet itself notes the deploy reaches the droplet over "SSH 22" and Cloudflare fronts HTTP — the admin/operator IP may be dynamic (residential ISP, travel, CI runner egress). If the allow-list is wrong, AE-0300 locks out SSH *and* (per its own break-glass note) recovery is via DO console only. AE-0305 even hedges ("if the admin IP is too dynamic, fail2ban may be dropped") but AE-0300 has no equivalent escape valve for its own SSH allow-list.
- Impact: Operator locked out of the production box; only out-of-band DO console remains.
- Suggested mitigation: Verify the actual admin IP source (ISP? static? a jump host?) and the GitHub Actions runner egress range *before* pinning the SSH allow-list. If dynamic, the SSH object must allow a broader-but-trusted source (e.g., a VPN/jump host, GitHub Actions metadata ranges) or this becomes a prerequisite blocker, not a silent assumption.
- Open question for author: Is the admin IP static, and has it been confirmed against where deploys *and* operator logins actually originate?

### [WARN] AE-0306 Langfuse encryption-key rotation may be unrecoverable, not just "needs a procedure"
- Assumption: `LANGFUSE_ENCRYPTION_KEY`/`SALT` rotation "needs a documented procedure (rotate with data re-encryption, or an accept-loss decision)."
- Risk: The packet does not establish that Langfuse *supports* key rotation with re-encryption at all. If Langfuse encrypts at write time with the live key and has no re-encrypt migration tool, "rotate with data re-encryption" is not an available procedure — the only real option is "accept loss of encrypted historical data," which is a data-integrity decision dressed as a security ticket. The AC "Langfuse still starts and reads its data" would pass on *new* data while *old* data is silently unreadable.
- Impact: Loss of historical trace/observation data with no rollback; the AC as written does not verify old-data readability.
- Suggested mitigation: Before the window, verify the actual Langfuse key-rotation capability (does a re-encrypt tool exist for the installed version?). Split the AC: "new traces write+read" vs "pre-rotation traces remain readable" — and require explicit acceptance if old data is lost. Do not let "documented procedure" hand-wave a data-loss event.
- Open question for author: Does the installed Langfuse version ship a re-encryption tool, and has old-data readability been tested, not just cold-start?

### [WARN] AE-0303 auto-reboot + `unattended-upgrades` can reboot mid-deploy or mid-carousel
- Assumption: A scheduled `Automatic-Reboot-Time` chosen from traffic data avoids user impact.
- Risk: `unattended-upgrades` auto-reboot is *ungated by app state* — it can fire during a running carousel generation (AE-0300 notes WebSocket/SSE streaming), mid-deploy (the deploy.yml SSH write of `.env`), or while AE-0306 rotation is in progress. The ticket checks restart policies and traffic timing but not *app/deploy invariants*. A reboot mid-deploy could leave `.env` partially written; a reboot mid-rotation could strand half-rotated secrets.
- Impact: Container restart is the intended blip, but an *uncoordinated* reboot during another ticket's maintenance window or a long-running carousel is a corrupting event.
- Suggested mitigation: Auto-reboot must coordinate with deploy and rotation windows — either a lock file the deploy/rotation holds (auto-reboot skips if locked), or auto-reboot is deferred until *after* AE-0306 completes. At minimum the AC should assert "auto-reboot does not fire while a deploy or rotation is in flight."
- Open question for author: What prevents `unattended-upgrades` from rebooting the box while AE-0301's deploy write or AE-0306's rotation is actively running?

### [WARN] AE-0301 rotation of cheap keys has a live-key exposure window during the redeploy itself
- Assumption: Rotating cheap keys = regenerate, update GitHub Secrets, redeploy; old revoked.
- Risk: Between "new key in GitHub Secrets" and "containers restarted with new env," the running containers still hold the *old* key — and the old key is revoked at the provider only *after* redeploy succeeds. But the redeploy writes `.env` (now 600, per this ticket), restarts services, and if *any* service fails to restart cleanly, you have revoked-yet-still-running references or a half-deployed stack holding stale keys. The packet specifies no ordering invariant between "revoke old" and "confirm new is live."
- Impact: A failed deploy mid-rotation leaves the app using a key that is already revoked (provider kills it) → second outage layered on the first.
- Suggested mitigation: Revoke old keys only *after* a successful post-deploy smoke confirms the new key is in use, not as an atomic step of the deploy. Make "old keys revoked" a post-verification step with its own AC, not bundled with the redeploy.
- Open question for author: At what point are the old provider keys revoked — before or after the post-deploy smoke confirms the new values are live?

### [INFO] AE-0304 "24h soak" before AE-0303 reboot is asserted but unenforceable
- Assumption: A 24h soak between AE-0304 cleanup and AE-0303 reboot validates that nothing depended on the removed containers.
- Risk: The soak is a `Blocked by` gate but there is no automated signal that "24h passed *and* nothing broke"; it relies on operator observation over a day. A hidden dependency with a lazy/cron-triggered reference (e.g., a daily job that calls the `minio-test` endpoint) would surface only at the 24h mark, exactly when AE-0303 reboots — compounding two changes.
- Impact: Low-to-medium; the inspect/snapshot artifact mitigates reversibility, but the "soak detected no dependency" claim is softer than the gate implies.
- Suggested mitigation: During the soak, explicitly check *cron and systemd timers that fire on a daily cycle* ran at least once against the post-removal state before the reboot. Narrow "soak" to "≥24h *and* all daily-scheduled jobs have executed once post-removal."
- Open question for author: Has a daily-cycle cron/systemd job actually run against the post-removal state during the soak, or is 24h a wall-clock proxy?

## Missing evidence
- The actual admin/SSH IP source and stability (AE-0300/AE-0305) — not established in packet; determines whether the SSH allow-list is viable at all.
- Cloudflare's range-change frequency and the script's behavior on a *shrinking* fetch (AE-0300) — determines outage risk of the scheduled refresh.
- Langfuse's installed-version rotation/re-encryption capability (AE-0306) — determines whether "documented procedure" is real or masks data loss.
- Whether `unattended-upgrades` reboot coordinates with deploy/rotation locks (AE-0303) — determines mid-state-rotation corruption risk.
- The deploy `.env` write atomicity (AE-0301) — partial-write exposure under an AE-0303 uncoordinated reboot.

## Residual risks if plan proceeds unchanged
- A bad Cloudflare-range refresh can darken the origin for real users before any alert fires (AE-0300).
- Operator/CI SSH lockout if the admin IP is dynamic and was not verified pre-pin (AE-0300/AE-0305).
- Silent loss of historical Langfuse data if key rotation lacks a re-encrypt tool and "accept loss" is not made explicit (AE-0306).
- A corruption event if `unattended-upgrades` auto-reboots mid-deploy or mid-secret-rotation (AE-0303 vs AE-0301/AE-0306).
- A layered outage if cheap-key revocation races the post-redeploy smoke (AE-0301).
