# AE-0295 — Architecture Plan: fix admin blog-posts listing crash

Ticket: AE-0295 · Tier: T1 · Type: Bugfix (frontend) · Date: 2026-07-01

## Root cause (confirmed)

Adapter maps workflow `status` into the content-`category` badge slot
(`blog-post-adapter.ts:12`); the grid passes it as a badge `color`
(`blog-posts-grid.tsx:84`); `BlogPostBadge` destructures `BLOG_POST_BADGE_COLORS[color]` with **no
fallback** (`badge.tsx:5`); status values aren't keys in that content-category map
(`constants.ts:29-44`) ⇒ `undefined` ⇒ `TypeError` crashes every non-featured card.

## Fix

Two independent, defense-in-depth changes:
1. **Correctness (compile-time exhaustive)**: introduce a `BLOG_POST_STATUS_COLORS` typed as
   **`Record<BlogPostStatus, BadgeColor>`** so TypeScript **fails the build** if any
   `BlogPostStatus` lacks a colour — the guard is structural, not "review discipline". Render the
   status badge from it (stop overloading `category`). This is the complete status set —
   `unpublish` reverts to the existing `draft` (verified), so AE-0296 adds **no 6th status**. A
   unit test also asserts `keys(BLOG_POST_STATUS_COLORS) === BlogPostStatus values`. Align semantics
   with the AE-0284 palette where sensible (draft=amber, published=green).
2. **Robustness**: `BlogPostBadge` returns a neutral default badge for any unknown `color` key
   (belt-and-suspenders — never destructure `undefined` again). This is the rule-fires regression:
   a test passes an unmapped key and asserts no throw.
3. **Kill the root cause at the type boundary**: the original bug was **cross-domain type
   confusion** (a workflow status fed into a content-category map). Type the `BlogPostBadge`
   status prop as **`BlogPostStatus`** (not a bare `string`/`BadgeColor`), so handing it a
   `WorkflowStatus` or arbitrary string is a **compile error** at the call site, not a runtime
   fallback. This structurally prevents the recurrence, above and beyond the runtime default in (2).

**Category decision (resolved):** **status replaces category** as this surface's badge — no code
path populates a real content `category` today, so shipping a second badge slot would be untested
layout debt (and the blog-posts grid is outside AE-0299's responsive scope). If category becomes a
real facet later, a follow-up ticket re-introduces a `categoryBadge` with its own responsive test.
Do not ship an ambiguous dual-badge now.

**Status domains are distinct:** this blog_posts status palette is unrelated to the carousel
*workflow* status palette used by AE-0299 (`resolveWorkflowStatusVisual`). They must not be
conflated or merged.

**Cross-layer drift guard:** the FE `Record<BlogPostStatus,…>` closes the FE map, but backend
`BlogPostStatus` is the source of truth — a backend-only status addition (e.g. a future
`scheduled`) could silently re-open the crash class. Add a cross-layer guard: either a test that the
FE `BlogPostStatus` union === the backend enum values, **or** a runtime Zod parse of the API
response that rejects/normalizes an unknown status (surfacing drift loudly, not via `undefined`).

## Testing (rule-fires / regression)

- Unit: `BlogPostBadge` with an unmapped key renders (no throw) — the anti-regression test.
- Unit: each of the five statuses maps to a defined `{bg,text}`.
- Component/render: listing renders draft+published without error.
