# Cold Critic Review

## Verdict
WARN

## Findings

### [BLOCKER] SSH allow-list cannot be safely pinned for GitHub-hosted deploy runners — the origin lockdown may break the auto-deploy pipeline itself
- Assumption: the AE-0300 prerequisite "verify the deploy-runner egress… before pinning the SSH allow-list" is treated as resolvable by enumerating an admin/runner source.
- Risk: GitHub-hosted Actions runners egress from a large, rotating pool of CIDRs that is NOT a small stable set. "Pinning" 22 to that source either means (a) opening 22 to all of GitHub's published ranges — a very wide surface that defeats the hardening and still drifts — or (b) pinning to specific IPs that rotate and silently break deploys. Per CLAUDE.md, every push to `main` auto-deploys over SSH 22; if the allow-list rejects the runner, the deploy silently fails → production drift / an unreconciled Alembic head / out-of-date security patches. The only real fix is infrastructure NOT in any of these 7 tickets: a self-hosted runner, a jump host, or VPN/Tailscale egress.
- Impact: Page-served production deploys break; or, if "safe" means "open to GH ranges," SSH hardening is largely cosmetic.
- Suggested mitigation: Do not let AE-0300 proceed past the prerequisite until a concrete, stable SSH source is named (jump host / self-hosted runner / VPN). If none exists, that finding IS the blocker — scope the infra ticket before AE-0300's close, not after.
- Open question for author: Does a jump host / self-hosted runner / VPN exist today? If not, which ticket builds it?

### [WARN] AE-0302 is NOT blocked-by AE-0301 — the new `REDIS_PASSWORD` will be written 644 (the exact exposure being fixed)
- Assumption: tickets that add secrets to `.env` inherit a safe umask. AE-0306 is correctly `Blocked by: AE-0301`; AE-0302 is not, yet AE-0302 adds a live secret (`REDIS_PASSWORD`) to the deploy `.env`.
- Risk: If AE-0302 lands before AE-0301, the deploy writes a fresh, live Redis credential to a world-readable file — re-creating the class of exposure AE-0301 exists to close, for a brand-new secret. AE-0301's umask fix only protects secrets written after AE-0301 lands.
- Impact: A new secret is exposed at rest for the gap between AE-0302 and AE-0301.
- Suggested mitigation: Add `Blocked by: AE-0301` to AE-0302 (or fold the umask fix into AE-0302's first step). Same for any other ticket that adds/rotates a secret.
- Open question for author: Why is AE-0306 gated on AE-0301 but AE-0302 not?

### [WARN] Authenticated Origin Pulls (mTLS) provides near-zero additional defense given the L4 allow-list — the ticket overstates it
- Assumption: AOP is "defense-in-depth" hardening against non-CF direct hits, complementary to the L4 CF-IP allow-list.
- Risk: The Cloudflare origin-pull client cert is the SAME public Origin CA issued to all Cloudflare zones. The L4 allow-list already restricts 443 to CF's egress IPs. So any other Cloudflare tenant who points their zone at your origin IP would (a) pass the L4 check (they egress from CF's IPs) and (b) pass mTLS (CF sends the same shared cert). mTLS here proves "traffic egressed Cloudflare's network" — identical to what the L4 list already proves. The threat it's claimed to add depth against (non-CF direct hit) is already blocked by L4. The "Authenticated Origin Pulls" tickbox produces audit-readiness warmth, not real isolation; zone-binding requires CF Tunnel or a customer cert, neither of which is in scope.
- Impact: Operator may believe mTLS mitigates a bypass that it does not; CF IP range changes become the only thing standing between origin and any CF tenant.
- Suggested mitigation: Either demote AOP to "explicitly cosmetic / audit checkbox" in the runbook, or adopt Cloudflare Tunnel (scope a separate ticket) for true zone-binding.
- Open question for author: What specific attacker capability does AOP mitigate that the L4 CF-IP allow-list does not already mitigate?

### [WARN] AE-0306 `POSTGRES_PASSWORD` rotation must enumerate ops/backup consumers, not just app services
- Assumption: "every consumer" = carousel checkpoint Postgres URL, app DB URL, Langfuse Postgres (per-instance enumeration).
- Risk: A common consumer class is omitted: backup scripts (`pg_dump`), cron/systemd timers, manual ops runbooks, and any ad-hoc analytics query that authenticates with the password. These will start failing `password authentication failed` after rotation — and a failing `pg_dump` cron is silent data-loss risk (backups stop succeeding with no alert). The ticket enumerates "every Postgres instance and its credential source" but does not enumerate "every consumer of each instance's credential."
- Impact: Quietly broken backups / missing on-call runbook steps post-rotation.
- Suggested mitigation: Add an AC: "For each Postgres instance, enumerate every consumer incl. `pg_dump`/backup/ops scripts; confirm each is migrated and green post-rotation (or documented as retired)."
- Open question for author: Is there a `pg_dump` cron or off-box backup that uses `POSTGRES_PASSWORD`?

### [WARN] AE-0301's "per-provider live-and-in-use" revoke gate is unachievable for providers with no exercisable smoke path
- Assumption: Each of OpenAI/Anthropic/Pinecone/GLM/Gemini/Langfuse has an automated smoke that can assert "this specific provider's new key is live and in use."
- Risk: A single carousel smoke typically exercises only the orchestrator + a couple of providers. GLM may be orchestrator-only; Gemini/Pinecone may have NO automated code path that runs in a smoke. Where no path exists, the "revoke after verify" gate has nothing to verify against → either the old key is revoked on faith (defeating the gate) or never revoked (leaving the exposed key live indefinitely). The gate is only as strong as the weakest provider's smoke coverage.
- Impact: Exposed key for a low-traffic provider stays live, or is revoked unverified and strands that provider.
- Suggested mitigation: For each provider, name the EXACT smoke/assertion that proves its key is in use (endpoint hit, log line, provider billing tick); if none exists, that provider needs a minimal exerciser before AE-0301 closes — or accept the residual risk explicitly.
- Open question for author: Which provider is exercised by which automated smoke today for GLM, Gemini, and Pinecone?

### [WARN] AE-0303 acceptance criterion "all containers healthy after reboot" contradicts the de-scope clause for the known-unhealthy frontend
- Assumption: AC #4 ("All production containers Up/healthy") and the de-scope clause ("the frontend unhealthy state is either resolved or de-scoped") can both be satisfied.
- Risk: They cannot. If the frontend's pre-existing unhealthy state persists post-reboot (the scan showed it unhealthy pre-reboot), the AC "all healthy" fails while the de-scope says "OK." A reviewer cannot tell whether the ticket passed. Worse, an unhealthy frontend may be a signal that the reboot itself produced a regression masquerading as a known state.
- Impact: Ambiguous gate; possible to "pass" with a broken frontend.
- Suggested mitigation: Rewrite the AC as: "All production EXCLUDING the de-scoped frontend are healthy; the de-scoped frontend is tracked as AE-XXXX with a stated reason it was unhealthy before the reboot and a confirmation the reboot did not worsen it."
- Open question for author: Was the frontend healthcheck green at any point pre-Jun 3, i.e. is "unhealthy" the steady state or a regression?

### [WARN] AE-0304's volume-snapshot "reversibility artifact" is snapshotted but not verified restorable
- Assumption: Capturing `docker inspect` + a volume snapshot is "reversibility incl. stored data."
- Risk: A MinIO anonymous-volume copy is not restorable by bare files — MinIO needs its access keys, the bucket metadata, and a compatible binary to rehydrate. "We have the bytes" ≠ "we can restore." If a weekly/monthly dependent surfaces 8 days post-AE-0303 reboot, the operator reaches for an untested restore under pressure.
- Impact: Reversibility is assumed, not proven; a real dependent break becomes a long live-site incident.
- Suggested mitigation: Add an AC: "For any non-empty volume, a restore is demonstrated at least once (mount snapshot → start container → confirm object read) before the original `rm -f`."
- Open question for author: Has the snapshot-restore path ever been run, or only `docker inspect` captured?

### [WARN] AE-0302 leaves `CONFIG` enabled — an authenticated compromised client can silently disable auth
- Assumption: `requirepass` + fail-closed server is the complete control; once authed, Redis is safe.
- Risk: Redis's `CONFIG SET requirepass ""` (and `CONFIG SET rename-command`, flush commands, etc.) remain callable by any client that holds the password. A container compromised via the `minio-test` default-creds → lateral movement (still possible until AE-0304 ships and AE-0302 is unordered relative to AE-0304) could grab the Redis password from `.env` (644 until AE-0301!) and `CONFIG SET requirepass ""` to re-open Redis for all peers — silently regressing AE-0302 from inside.
- Impact: Auth control is one-way-correct on bootstrap but mutable at runtime; regression is invisible.
- Suggested mitigation: Disable/rename dangerous admin commands (`rename-command CONFIG ""`, `FLUSHALL ""`, `FLUSHDB ""`) in the same prod config; add a healthcheck/assertion that `CONFIG GET requirepass` stays non-empty post-deploy.
- Open question for author: Why is `CONFIG` left callable in a prod Redis that only needs data-plane ops (GET/SET/XADD/XREAD)?

## Missing evidence
- Whether a jump host / self-hosted runner / VPN exists for SSH 22 (decides whether AE-0300 is even viable).
- Current `.env` umask at deploy time (decides whether AE-0302 needs AE-0301 first).
- Which automated smokes exercise GLM / Gemini / Pinecone (decides whether revoke-after-verify is real for those).
- Whether a `pg_dump` / backup script uses `POSTGRES_PASSWORD` (decides AE-0306's consumer enumeration completeness).
- Whether Langfuse installed version supports encryption-key re-encryption, and whether the snapshot-restore in AE-0304 has been run once.
- Whether GH-hosted runner egress IP is in `fail2ban` `ignoreip` and how it tolerates runner rotation.

## Residual risks if plan proceeds unchanged
- Deploys may silently start failing once 22 is locked to a non-viable source (AE-0300).
- A new live Redis password may sit world-readable until AE-0301 lands (AE-0302 ordering).
- An exposed low-traffic provider key may stay live because no smoke can prove it's "in use" (AE-0301).
- A future compromised peer container could re-open Redis via `CONFIG` (AE-0302).
- The expensive-secret rotation may leave backups/ops scripts broken (AE-0306).
- "Reversibility" of container removal is asserted by an artifact that has never been restored (AE-0304).
