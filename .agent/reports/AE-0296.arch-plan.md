# AE-0296 — Architecture Plan: blog-post edit/delete/hide controls (dashboard)

Ticket: AE-0296 · Tier: T2 · Type: Feature (frontend-only) · Date: 2026-07-01

## Decision

Mostly frontend wiring over existing endpoints, **plus one small backend integrity guard** on
delete (area = fullstack). Hide is the **default** destructive action; hard delete is guarded.

## API contracts (verified)

- Edit: `PUT /blog-posts/{id}` — `routes/blog_post.py:175`. **The FE edit PUT already sends
  `If-Match: <lock_version>`** (`use-blog-posts.ts:72-79`, seeded from `post.lock_version`). So
  optimistic-lock is already wired; this ticket only adds **412 handling** (refetch + message) and a
  concurrent-delete 404 case.
- Delete: `DELETE /blog-posts/{id}` (204) — `routes/blog_post.py:236`. **Does NOT send `If-Match`
  today** and has **no referential guard**.
- Hide: `POST /blog-posts/{id}/unpublish` — sets status to a **fixed `draft`** and clears
  `published_at`/`submitted_for_review_at` (verified `legacy_publishing_acl.py:124-134`). So the
  public filter (status==published) naturally drops it. Reuses the existing 5-status vocabulary —
  **no new status** (AE-0295's map stays complete).

## Design

- `use-blog-posts.ts`: add a real `unpublish(id)` mutation; wire the dead `delete` path. Each
  mutation invalidates the listing query.
- Hide semantics: **not a symmetric toggle.** Hide = unpublish (→`draft`). "Restore/republish" is
  the normal workflow (draft → submit → approve → publish), not a one-click unhide in v1. UI labels
  and copy must reflect this ("Hide (revert to draft)"), not promise instant re-publish.
- **DELETE integrity guard (backend, small) — allowlist, complete value set:** `BlogPostOrigin` is
  a `StrEnum` with exactly `{standalone, carousel}` (verified `domain/constants/blog_post.py:23-32`;
  set-on-insert per AE-0127 backfill, not rewritten). Guard = **allowlist**: hard `DELETE` permitted
  **only** for `origin == standalone`; `origin == carousel` ⇒ **409** (deleting it makes
  `GET /carousels/{id}/blog` return 404 — the public carousel blog reads that very row,
  `publishing_read_acl.py:250-254`). No unanticipated third origin exists, and a unit test asserts
  the guard covers every `BlogPostOrigin` member (fails the build if a new origin is added without a
  delete policy). Carousel-origin posts get **Hide/Archive only** in the FE. Add `If-Match` to
  DELETE for lost-update safety.
- Listing cards (`blog-posts-grid.tsx`): real Edit link (`BLOG_POST_EDIT`), Hide (published),
  Delete (non-carousel only, typed confirm). Reuse `dashboard.blogPosts.confirmDelete`; add
  hide/edit labels en+pt.
- **Cache/visibility:** hide flips status→draft; per ADR-0013 the public route is `no-store`/short
  cache in v1 so the post disappears immediately. When a CDN lands, unpublish must purge the
  public cache for the id/slug (owned follow-up, declared now).

## Testing

- `.feature`: edit navigates + persists; **412 on stale edit is handled** (existing If-Match →
  simulate lock_version mismatch); delete of a **carousel-origin** post is blocked (409, FE shows
  Hide instead); delete of a non-carousel post confirms then removes; hide calls unpublish and the
  badge updates to draft; cancel-delete sends no request.
- Backend unit/`.feature` for the DELETE 409 guard on carousel-origin rows.
- Mock the API client; assert endpoint + method + `If-Match` header on PUT and DELETE.

## Security / integrity

- `EditorUser`-gated server-side; FE surfaces 403/412/409 rather than assuming success. Non-admins
  scoped to own posts. The 409 guard prevents orphaning the ADR-0011 carousel→blog projection.

## Dependencies / rollout

- Blocked by AE-0295 (non-crashing listing + status badge). Coordinates with AE-0297 (hide removes
  from public feed). Additive; rollback = remove controls + the 409 guard.

## Carousel-origin erasure lifecycle (compliance — the 409 is not a dead end)

The 409 guard blocks **casual** delete from the listing (orphan prevention), but a permanent
erasure path MUST exist (GDPR/DMCA/legal takedown). Defined lifecycle:
- **Erasure of a carousel-origin post = erasing its parent carousel project.** Deleting the
  carousel project is the canonical "remove this content" action; the blog row is then removed as
  part of that flow (or its `project_id` goes NULL via the existing `ON DELETE SET NULL` and the
  now-detached, `standalone`-equivalent row becomes hard-deletable). Either way there is a concrete
  path; the post is never "un-removable".
- The listing UI's Delete stays guarded (409) to prevent *accidental* orphaning; erasure is a
  deliberate admin action through carousel deletion, documented here so compliance never relies on
  "archive forever".
- **Scope note:** wiring a dedicated force-delete endpoint (if product wants erasure without
  deleting the whole carousel) is a **follow-up ticket**, not AE-0296; AE-0296 ships the guard +
  Hide + the documented lifecycle above.

## unpublish concurrency (committed, not "ideally")

`unpublish` is a status flip that clears `published_at`. It **sends `If-Match`** (`lock_version`)
so it cannot silently clobber a concurrent editor's optimistic edit; on **412** the FE refetches and
re-confirms. If the existing endpoint does not yet honor `If-Match`, adding that check is **in
scope** for this ticket (small backend change alongside the DELETE guard) — last-writer-wins is not
an acceptable silent outcome for a visibility change. A `.feature` covers concurrent
unpublish+edit → 412 handled.

## Resolved (was open)

1. **Delete vs Hide:** Hide is the default/primary action; Delete is guarded (409 for
   carousel-origin, allowlist over the complete `{standalone, carousel}` enum) and requires typed
   confirmation for `standalone`. Erasure lifecycle defined above.
2. **If-Match:** already sent on PUT; this ticket adds 412 handling + If-Match on **all three**
   mutations (PUT already, DELETE, unpublish). No hedge — the `.feature` covers each.
