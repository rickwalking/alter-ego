## Findings

### [BLOCKER] AE-0307 introduces a new single point of failure that is itself unsecured and unrotatable in the packet
- Assumption: A jump host / self-hosted runner / Tailscale node is "stable" and is implicitly trustworthy.
- Risk: Whichever option is chosen becomes **the** credential boundary for production deploys AND admin access. The packet specifies no hardening, secret rotation, monitoring, or break-glass FOR the ingress itself. A self-hosted runner holds a GitHub Actions token with `deploy` scope on the main branch; a jump host holds the root SSH key; a Tailscale node keys are long-lived. None of these has its own ticket, its own compromise handling, or a stated rotation cadence.
- Impact: A compromise of the ingress = compromise of all of prod (deploy RCE + root SSH). AE-0300's L4 allow-list is pointed at this same ingress, so once pivoted onto it, you bypass origin lockdown *legitimately*. You've concentrated risk without hardening the new blast radius.
- Suggested mitigation: AE-0307 must enumerate the ingress's own trust material (runner token / bastion SSH key / tailnet key), add a rotation + revocation + "ingress down" recovery procedure, and define how the AE-0303/AE-0301 lock-file/umask gates interact with rebuilds of the ingress. Treat it as a privileged asset, not plumbing.
- Open question for author: What is the failure mode if the self-hosted runner node is compromised — does AE-0300's allow-list protect you, or does it guarantee the attacker reaches origin?

### [BLOCKER] The Blocked-by graph still admits a fatal ordering: AE-0300 depends on AE-0307, but AE-0307 does not gate on the *deploy actually being repointed* before AE-0300 closes public 22
- Assumption: AE-0307's AC ("deploy completes over the new path") is sufficient evidence that the GitHub-hosted-runner path is no longer needed before AE-0300 restricts 22.
- Risk: AE-0307 proving *one* deploy works over the new path is not equivalent to *retiring the old path*. If the old `deploy.yml` SSH-to-22 step still exists (as a fallback, or not fully removed), and AE-0300 then restricts 22, the next deploy that hits the old branch silently fails — or, worse, a hotfix re-enables the GitHub-hosted path under time pressure, reopening the bypass AE-0300 closed. The packet never requires *deleting* the public-22 code path.
- Impact: Silent deploy breakage in prod (the exact AE-0207-style outage scenario the root CLAUDE.md warns about), or a panic-revert that undoes the hardening.
- Suggested mitigation: Add an AC to AE-0307: "the public-22 deploy code path is removed/guarded, not merely superseded" and to AE-0300: "verified that no remaining workflow/job reaches the droplet over public 22." Cross-check the `Blocked by` resolves *only* post-removal.
- Open question for author: Is there any scheduled/cron GitHub Actions job (not just `main`-push deploy) that still egresses to public 22 and would break silently?

### [WARN] AE-0302 rename-command disables `CONFIG`, but the post-deploy probe claimed to assert "auth stays required at runtime" cannot use `CONFIG GET requirepass` once CONFIG is disabled
- Assumption: A healthcheck/post-deploy probe can confirm auth stays required after `CONFIG` is disabled.
- Risk: The ticket's own AC #4 asks for `rename-command CONFIG` (disabled) *and* a probe that confirms `CONFIG GET requirepass` stays non-empty. These are mutually exclusive. The packet notes "(or, if CONFIG is disabled, an equivalent auth-still-required probe)" but never specifies the equivalent probe. A common substitute — attempting an unauthed `PING` and expecting `NOAUTH` — only proves auth is required *at that moment*, not that it cannot be disabled; with CONFIG gone, the "cannot be silently re-opened" half is unverifiable at runtime, reverting to a build-time guarantee.
- Impact: The stated defense ("compromised peer can't `CONFIG SET requirepass ''`") is real, but the *ACs that prove it* are contradictory and one will fail QA; or, worse, an author picks the unauthed-PING probe and ships it thinking it covers the disablement case (it doesn't).
- Suggested mitigation: Split the AC: (a) build-time assertion that `rename-command` lines are present in the prod Redis config; (b) runtime probe = unauthed command returns NOAUTH (covers "auth is on now"). Drop the "stays required at runtime / cannot be re-opened" claim from the runtime probe — that is a config-integrity check, not a runtime one.
- Open question for author: What is the concrete equivalent probe, and does it genuinely cover disablement or only current-required?

### [WARN] AE-0306's POSTGRES_PASSWORD zero-downtime option requires PgBouncer or dual-role credentials that are not in the stack and not provisioned
- Assumption: Option (a) "dual-credential/new-role zero-downtime" is a real option for this deployment.
- Risk: The prod stack is `docker-compose.prod.yml` with Postgres in a container, no PgBouncer in the packet, and consumers initialized from `.env` at container start. A "new role" approach requires every consumer to reconnect with the new role — but the compose services read the credential once at startup from `env_file`; there is no graceful credential hot-swap. So in practice AE-0306's only real option is (b) bounded downtime, but the ticket presents (a) as a live choice without verifying the infra supports it. This may force a mid-rotation pivot to (b) under time pressure.
- Impact: The "rollback as code / do not revoke old password until smoke confirms new" guarantee collapses if (a) is attempted naively — a consumer that buffered the old DSN string at startup may stay on the old role while others move, creating a split-brain that PASS the smoke but fail later, and the revocation then strands the laggard.
- Suggested mitigation: Pre-decision: enumerate every consumer and confirm whether it re-reads credentials on connection (no) — and if not, the only safe option is (b) bounded downtime with a full-stack restart, stated up front.
- Open question for author: Has any consumer's connection-pooling/reconnect behavior been verified, or is (a) aspirational?

### [WARN] AE-0303 + AE-0306 + AE-0300 share a single lock-file concept but no ticket defines the lock's lifecycle, ownership, or watchdog
- Assumption: A lock file shared across deploy / rotation / auto-reboot / CF-range refresh behaves correctly with no defined semantics.
- Risk: Three tickets (AE-0300 refresh, AE-0303 auto-reboot, AE-0306 rotation) reference "the lock" as if it exists, but none owns it: who creates it, where it lives, what happens if the holder crashes mid-operation (stale lock = auto-reboot permanently disabled = the box drifts again), whether it's advisory or enforced, and how a frozen holder is force-cleared. A stale lock after a crashed rotation is a silent regression of the auto-reboot safety net AND a deploy blocker.
- Impact: Either locks leak and disable the auto-reboot safety net (undoing AE-0303), or a lock is force-cleared incorrectly and a reboot fires mid-rotation (the exact corrupting event AE-0306 guards against).
- Suggested mitigation: One ticket (AE-0303, as reboot-owner) must define the lock: path, holder, TTL/watchdog, staleness detection, force-clear procedure, and revert-on-holder-crash. Others reference it; they don't each invent it.
- Open question for author: If the AE-0306 rotation process dies after taking the lock but before releasing it, what is the deterministic recovery — and is it tested?

### [WARN] AE-0304's "restore demonstrated once" AC is	Runtime-heavy and underspecified for a destructive op on a 4GB prod box
- Assumption: Mounting a snapshot + starting a container + reading back an object is cheap and safe on the prod droplet.
- Risk: A MinIO volume restore-demo spins up `minio` + needs RAM — on a 4GB box at 79% disk that may already be stressed, this is itself a risk during the documented cleanup window. The AC also bundles state ("restorable") onto a *cleanup* ticket, expanding its blast radius. If the demo crashes or the MinIO bucket metadata is incompatible with the demo binary, the operator is one `rm -f` away from data loss with no clear "what now" path.
- Impact: The cleanup ticket now carries a restore-verification dependency that can block it indefinitely or, under time pressure, be skipped "because the capture artifact exists" — the exact false-confidence the AC was meant to prevent.
- Suggested mitigation: Either (a) perform the restore demo on a *separate* host (laptop/dev box) with the captured artifact, not on prod; (b) reduce scope to "snapshot bytes are hashable + MinIO `mc ls` of the mounted volume succeeds" and make the destructive `rm -f` a separate, gated follow-up. State which.

### [INFO] AE-0301 cheap-key revocation gates on "per-provider in-use assertion" but GLM/Pinecone have no defined exerciser paths in the ticket's own accounting
- Assumption: A "minimal exerciser" can be added for GLM and Pinecone before revoking.
- Risk: The packet names GLM (`GLM_API_KEY`, model `opencode-go/glm-5.2` per env) and Pinecone but does not confirm any code path exercises them in a carousel smoke. Gemini is explicitly excluded ("not active in prod by design") — but GLM is *used as the system model* here and is also a real provider key. If no exerciser exists for GLM, the ticket's own gate forces either building one (scope creep on a T1 security ticket) or "explicit accepting the residual risk of revoking-on-faith" against the ticket's stated default to rotate.
- Impact: Either the GLM key is revoked unverified (counter to the revoke-after-verify gate), or the T1 ticket expands to build a GLM smoke harness (scope creep), or GLM is left un-rotated (silent accept of the exposure it was meant to fix).
- Suggested mitigation: Pre-list, per provider, the exact in-use assertion (OpenAI/Anthropic: carousel smoke; Pinecone: ?; GLM: ?). If GLM/Pinecone lack one, decided up front whether to build, accept-faith, or exclude-explicit — not discovered during execution.
- Open question for author: What is the in-use assertion for GLM specifically, and is it verified or assumed?
