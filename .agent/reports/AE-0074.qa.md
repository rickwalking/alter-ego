## 🟡 Suggestion Findings

### 1. `logger.exception` includes full traceback — `workflow_event_service.py:144`
**File:** `backend/src/rag_backend/application/services/workflow_event_service.py:144`
**Description:** `_schedule_publish()` uses `logger.exception(...)` when `asyncio.get_running_loop()` raises `RuntimeError`. This logs the full Python traceback including file system paths (e.g., `/home/pmarins/...`) which could leak system structure information. Prefer `logger.warning(...)` with a structured `error=str(exc)` field instead.
**Reference:** OWASP A09:2025 — Security Logging & Alerting; CWE-209: Information Exposure Through an Error Message
**Severity:** 🟡 Suggestion (pre-production, single-user project lowers risk)

### 2. Global SQLAlchemy event listeners on ALL sessions — `workflow_event_service.py:154,165`
**File:** `backend/src/rag_backend/application/services/workflow_event_service.py:154,165`
**Description:** `@event.listens_for(Session, "after_commit")` and `@event.listens_for(Session, "after_rollback")` register listeners on every SQLAlchemy `Session` instance in the process, not only sessions that use `WorkflowEventService`. Every transaction commit/rollback runs through these callbacks and accesses `session.info`. While overhead is minimal (a dict `.pop`), this is added attack surface. Accepted per Decision Log (session-hook mechanism chosen deliberately).
**Reference:** OWASP A02:2025 — Security Misconfiguration; Ticket Decision Log
**Severity:** 🟡 Suggestion (intentional design)

### 3. `stream_entry_id: str | None = None` in API schema always returns null — `workflow_audit.py:24`
**File:** `backend/src/rag_backend/api/schemas/workflow_audit.py:24`
**Description:** The response schema retains `stream_entry_id: str | None = None` even though the column was dropped from the database. The field now always serializes as `null`. This is dead schema code, accepted per the decision log ("response contract byte-stable").
**Reference:** Ticket Decision Log; OWASP A02:2025 — Security Misconfiguration (dead code in API surface)
**Severity:** 🟡 Suggestion

**Reference:** `backend/CLAUDE.md` "Max 3 arguments per function"; `pyproject.toml:343`

### P2. `_on_session_commit` listener is synchronous + creates async tasks
**File:** `workflow_event_service.py:154-162`
**Description:** SQLAlchemy `Session` events (`after_commit`) fire synchronously. The listener calls `_schedule_publish()` which uses `asyncio.get_running_loop()` and `loop.create_task()` to launch async work. This is a well-known pattern but means the publish is fire-and-forget from the request lifecycle — the request completes before the publish finishes (or fails). This behavior is the accepted design (post-commit publish gap addressed by Phase 6 outbox). Not a new issue from this diff.
