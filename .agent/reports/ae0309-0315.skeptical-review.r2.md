# Cold Critic Review

## Verdict
WARN — 5 of 7 tickets are implementable as written. **AE-0314 has a false premise that makes it unimplementable as scoped** (BLOCKER), and AE-0315's reaper introduces a split-brain hazard. AE-0310 misdiagnoses its root-cause mechanism. Cross-ticket, the `edited_localized_slides` phase-allowlist is widened by two tickets toward two different phases with no owner for the full coherent set, and one prerequisite (final_review widening) is orphaned.

## Findings

### [BLOCKER] AE-0314 relies on a `final_review` in-place edit path that does not exist

- **Assumption:** AE-0314 (Problem + Scope) asserts "The backend accepts `structured_feedback.edited_localized_slides` on resume (applied at content/final_review…)" and scopes out backend workflow-node changes ("resume path already exists"). The review-step editor is specified for "content, final_review."
- **Risk:** Verified false at **two** layers. (1) The HTTP gate hard-rejects `edited_localized_slides` unless `checkpoint_phase == PHASE_CONTENT` — `editorial_workflow_routes_validate.py:250-253`, error `ERR_EDITED_SLIDES_CONTENT_PHASE_ONLY`, confirmed by `test_edited_slides_rejected_at_final_review`. (2) Even if it reached the node, `_edited_slide_updates` short-circuits `if phase != PHASE_CONTENT: return {}` — `carousel_workflow_nodes.py:86`. At `final_review` the *only* consumed structured fields are `target_phase` (routes to LLM regeneration, ~15 min) and `edited_text` (feeds the feedback-learning loop only; does **not** mutate `localized_slides`).
- **Impact:** The final_review editor as specified will 422 on every save. To make it work, AE-0314 must widen the validation gate AND `_edited_slide_updates` to `final_review` — a backend workflow-node change it explicitly excludes. No ticket in this packet widens `edited_localized_slides` to `final_review` (AE-0310 widens to *design* only). The prerequisite is orphaned.
- **Suggested mitigation:** Either (a) add "widen `edited_localized_slides` gate + node helper to `final_review`" to AE-0314's scope and Affected Areas (and reconcile with AE-0310 so the allowlist becomes `{content, design, final_review}` coherently in one place), or (b) drop the final_review editor from AE-0314 and route final_review copy fixes through the completed-project slide-update endpoint (which is already in scope). Do not ship the review-step editor claiming "no workflow-node changes."
- **Open question for author:** Is the final_review in-place edit intended to reuse the `edited_localized_slides` resume field, or the post-completion slide-update endpoint? These are different code paths with different consistency contracts.

### [WARN] AE-0310 misdiagnoses the design-revise loop mechanism

- **Assumption:** AE-0310 (Problem) states the loop is caused by "the design node's artifact is `ensure`-d — with `design_applied=True` the revise records feedback and re-interrupts without re-rendering or re-validating." Scope item 3 proposes "no ensure-skip while a blocking validation report is stored," implying a skip to remove.
- **Risk:** Verified: `_ensure_design_artifacts` (`phase_artifact_runner.py:417-439`) calls `apply_design_tokens` → `run_design` **unconditionally** — there is no `design_applied` guard on the ensure. `post_review={"design_applied": True}` fires **only on approval** (`carousel_workflow_nodes.py:213-214, 262-275`), never on a revise. The real gap is that the design ensure **never runs presentation validation at all** (it only applies design tokens); `validated_at` is stale by *omission*, not by *skip*.
- **Impact:** An implementer following the packet literally will look for a `design_applied`-based skip to remove, find none, and may conclude the bug is elsewhere. The actual fix ("add presentation validation to the design ensure/re-render path") is achievable but the problem statement points at the wrong code. The proposed scope items (accept edits, accept send-back, re-validate on plain revise) still address the symptom correctly.
- **Suggested mitigation:** Rewrite AE-0310's Problem to state: the design ensure re-renders design tokens but does not re-run presentation validation, so content-level violations persist untouched across revises. Scope item 3 becomes "the design ensure/re-render path must run `validate_localized_slides` and store a fresh `presentation_validation` report."
- **Open question for author:** On a plain design revise with no edits and no send-back, what should happen if the (newly-run) validation still blocks? Re-interrupt with the fresh report (the packet implies this), or force the reviewer to pick an explicit recovery action?

### [WARN] AE-0315's stale-run reaper risks split-brain on slow-but-healthy runs

- **Assumption:** The reaper resets `phase_status=in_progress` rows whose `run_started_at` exceeds 30 min; "checkpoint also mid-run → reset to `awaiting_human` for a clean re-resume."
- **Risk:** The reaper cannot distinguish a **dead** background task from a **slow-but-alive** one using `run_started_at` + checkpoint `phase_status` alone. Observed healthy runs are 13–15 min, but LLM rate-limit/retry spikes can exceed 30 min. If the reaper resets a row whose owning task is still alive, the user's re-resume starts a **second** run while the first continues mutating checkpoint state → conflicting writes, image double-generation, corrupted `localized_slides`. There is no kill/cancel for the background resume task (`schedule_background_resume`), so reaping cannot guarantee the old run is stopped.
- **Impact:** Exactly the double-run race the prod incident risked (a "premature manual phase_status flip" per AE-0315's own Problem) — but now automated and codified. A 35-min healthy run gets reaped by design.
- **Suggested mitigation:** (a) Require a positive liveness signal, not just age — e.g. the background task heartbeats a `run_heartbeat_at` column; the reaper only reaps when the heartbeat is stale (task provably not making progress), not merely when `run_started_at` is old. (b) Add a run-token/cancel mechanism so reaping sets a flag the background task checks before each mutation. (c) Raise the default threshold well above observed-worst-case (e.g. 60 min) and make it per-phase configurable. (d) If none of these, the reaper must only emit the `run_overdue` alert and **not** transition the row — leaving recovery a human action (safer than automated split-brain).
- **Open question for author:** What is the kill mechanism for an in-flight background resume task? Without one, "reset to awaiting_human" is unsafe by construction.

### [WARN] AE-0315 `run_started_at` lifecycle has a terminal-write race

- **Assumption:** `run_started_at` is "set on run start, cleared on terminal write."
- **Risk:** If the process dies after the run reaches a terminal/awaiting state but **before** clearing `run_started_at`, the column stays set while `phase_status` is already `awaiting_human`. The reaper's query (filter `phase_status == in_progress`) would skip it — but the banner logic and any read of `run_started_at` would show a stale "running" state. Conversely, if the terminal write and the clear are not in the same transaction, a crash between them leaves drift. The packet does not specify that `run_started_at` is cleared atomically with the `phase_status` transition.
- **Impact:** A completed/awaiting project can render a permanent "revision in progress" banner after a crash — the exact "permanently disabled" UI the ticket says it prevents. The "Check again" escape mitigates the user side, but operators reading the column see false run state.
- **Suggested mitigation:** Clear `run_started_at` in the same UPDATE that sets `phase_status` to any non-`in_progress` value (one atomic row write). Add a reaper guard: never reap a row whose `phase_status != in_progress` regardless of `run_started_at`. Add a test for "crash after terminal write, before clear."
- **Open question for author:** Is the `run_started_at` clear colocated with the `phase_status` terminal transition in one statement?

### [WARN] AE-0311 two-commit contract leaves a read-path disagreement window, and the locking domains don't cover repair-vs-republish

- **Assumption:** Write `carousel_slides` projection first (own txn), then checkpoint via `aupdate_state`; re-invocation converges; the repair takes the `bump_resume_lock_version` CAS.
- **Risk:** (1) Read gap: `resolve_presentation_review_from_state` returns the **stored** checkpoint `presentation_validation` on the fast path (`presentation_review.py:258-274`) — it does **not** re-derive when stored validation exists (the packet's AE-0314 claims "re-derives on every read," which is also wrong — see below). So between the two commits, the projection has fixed copy but the checkpoint still holds the old blocking validation → the client shows stale violations over fixed `carousel_slides` rows. The packet says "reconciliation check reports drift" but does not specify what the **read path** returns mid-window or that the UI is consistent. (2) Locking: the repair uses the resume `lock_version` CAS on `carousel_projects`; AE-0313's republish uses a **per-project build advisory lock**. These are different lock domains. A repair and a republish on the same completed project can run concurrently — the repair mutates `carousel_slides`/checkpoint while the republish reads `carousel_slides` to re-render the PDF. No ticket declares a shared lock between repair and republish.
- **Impact:** Transient UI inconsistency (stale violations over fixed copy) for in-flight projects; race-induced corrupt PDF render (republish reads half-repaired slides) for completed projects.
- **Suggested mitigation:** Define the cross-endpoint lock contract: repair and republish must share a serialization point (the same advisory lock, or republish acquires the resume `lock_version` CAS too). For the read gap, either clear `presentation_validation` from checkpoint in the same `aupdate_state` that writes repaired copy, or document that the UI must re-fetch after a 200 from repair.
- **Open question for author:** What locks (or guards) serialize a repair against a concurrent republish on the same completed project?

### [WARN] AE-0314's source-of-truth rule rests on an overstated claim about `resolve_presentation_review_from_state`

- **Assumption:** AE-0314 (pinned rule) states "`resolve_presentation_review_from_state` re-derives validation from checkpoint state on every read," justifying the concern that a projection-only edit leaves stale checkpoint-derived violations.
- **Risk:** Verified: the resolver is a Chain-of-Responsibility (`presentation_review.py:317-332`) whose **first** resolver `_resolve_from_validation` returns the stored `presentation_validation` dict as-is with **no re-validation** (fast path). Re-derivation only occurs on the fallback paths (missing stored validation). So the concern is **real but for a different reason**: for completed projects with a stored validation, a projection-only edit leaves stale violations because the stored report is returned verbatim, not because of per-read re-derivation. The two proposed remedies (a: also update checkpoint copy + validation; b: serve projection-derived validation for completed) are both viable but the packet leaves the choice open ("whichever is chosen must be recorded in the ADR") — meaning the ticket's consistency contract is undecided at plan time.
- **Impact:** An implementer may pick (b) (change state derivation to serve projection-derived validation for `completed`) without realizing this changes read semantics for **all** completed projects, including those never edited — a broad blast radius. Option (a) is narrower but requires checkpoint writes on a completed project (which has no active workflow run — is `aupdate_state` even valid there?).
- **Suggested mitigation:** Decide the option in the plan, not at implementation time. Prefer (a) with an explicit checkpoint-write path for completed-project edits, and assert in a test that the served validation matches the edited projection. If (b), scope a separate read-authority change and assess blast radius across all completed-project reads.
- **Open question for author:** Is `aupdate_state` valid for a project with no active LangGraph run (completed)? If not, option (a) needs a new checkpoint-write path — is that in scope?

### [WARN] AE-0312 absent-severity default is unspecified — a silent unblocking risk

- **Assumption:** Severity model is net-new in AE-0312; "existing rules default to `blocker` (behavior unchanged)"; casing rules are `warning`.
- **Risk:** The packet does not state what happens when a violation's rule code has **no** `severity` metadata in the v2 policy YAML (e.g. a rule added by another ticket, or an existing rule whose metadata wasn't carried forward). If absent → `blocker`, behavior is safe. If absent → `warning`/`None`, then every rule missing the tag silently stops blocking, and `_blocking_from_repaired_localized`/`_blocking_from_drafts` return `False` — `localized_slides` with real violations pass the gate. This is a YAML-data migration risk hiding inside `hero_lower_third_v2.yaml`, not a code risk, and no test in the packet asserts the absent-severity default.
- **Impact:** A single missing severity tag in v2 silently disables blocking for that rule across all new carousels — a regression that the snapshot tests (which pin specific payloads) might not catch unless a "every rule has severity" invariant test exists.
- **Suggested mitigation:** Pin the default explicitly in code (`severity = rule.severity or BLOCKER`), add a policy-loader assertion that **every** rule in v2 carries a severity (fail load otherwise), and add a regression test that seeds a violation for each pre-existing rule code and asserts `blocking=True`.
- **Open question for author:** What is the code-level default when `severity` is absent from a rule definition?

### [WARN] AE-0312 v2-only routing leaves v1 in-flight/completed projects without casing validation or auto-fix

- **Assumption:** New carousels route to `hero_lower_third_v2`; v1 stays supported for in-flight projects; "no retroactive migration"; AE-0311 covers on-demand fixes.
- **Risk:** The packet does not specify whether `repair_casing` (the deterministic transform) is **policy-version-aware**. If `repair_casing` only runs under v2, then a v1 in-flight or completed project hitting the AE-0311 "Fix issues" button gets scaffold/body repair but **no** casing fix — the user sees casing violations persist after "fix all." If `repair_casing` runs regardless of policy version, then v1 projects get casing *repair* without casing *validation* — the repair mutates copy that was never flagged, which is surprising and could break intentionally-lowercase stylistic slides (the packet's own exemption concern).
- **Impact:** Either v1 projects can't be auto-fixed for casing (button lies — "fix all" leaves casing), or they get unvalidated casing mutation (repair without validation = unexpected edits). Both are bad UX; the packet doesn't pick one.
- **Suggested mitigation:** Specify: `repair_casing` runs only when the project's policy version includes the casing rules (v2+). For v1 projects, the AE-0311 button reports casing violations as **unrepairable** (honest) until the project is migrated to v2 (offer a policy-version upgrade path, or accept that v1 projects need manual casing fixes). Add a test for a v1 project hitting the repair button.
- **Open question for author:** Does `repair_casing` execute for a project whose `presentation_policy_version == hero_lower_third_v1`?

### [INFO] Cross-ticket: the `edited_localized_slides` allowlist is widened incoherently across AE-0310 + AE-0314 with no single owner

- **Assumption:** AE-0310 widens the allowlist to `design`; AE-0314 (per the BLOCKER above) needs it widened to `final_review`. Each ticket lists its own slice.
- **Risk:** The validation gate (`editorial_workflow_routes_validate.py:250-253`) and node helper (`_edited_slide_updates`) are a **single shared surface**. Two tickets editing it independently risk a half-applied allowlist (design but not final_review, or merge conflicts) and divergent semantics (e.g. does final_review re-validate after edit the same way content does?). The "phase pinning" constant `ERR_EDITED_SLIDES_CONTENT_ONLY` itself becomes a misnomer.
- **Impact:** Integration bugs at the seam; the constant/error copy drifts from reality.
- **Suggested mitigation:** Have one ticket own the full allowlist change to `{content, design, final_review}` (recommend AE-0310, since it's already in the phase-widening business and declares the "target-phase routing helper shared with AE-0288"), with AE-0314 depending on it. Rename the constant/error to reflect the widened set.

### [INFO] AE-0313's health-check fix alters first-time finalize behavior too

- **Assumption:** "No changes to first-time finalize behavior beyond the shared health-check fix."
- **Risk:** The shared fix (`_verify_artifacts` validates freshly-rendered pre-promotion outputs instead of the versioned root) changes the path `_safe_output_file` evaluates against for **every** finalize, including first-time. First-time finalize has `artifact_version = None` → `serving_root = project_root` (`artifact_path_resolver.py:88-101`), so the fix is a no-op there *in the common case*. But the edge case (a first-time finalize on a project that somehow already has `artifact_version` set from a prior partial run) would now behave differently. The packet's regression coverage is the 66014ba3 *completed-republish* fixture, not a first-time-finalize-with-stale-version fixture.
- **Impact:** Low likelihood, but a first-time finalize regression is high severity (blocks all new carousels).
- **Suggested mitigation:** Add a regression test for first-time finalize with `artifact_version` already set (e.g. a failed-then-retried first finalize) asserting the health check passes.

### [INFO] Security: AE-0314 markdown preservation vs. frontend XSS

- **Assumption:** The editor preserves markdown emphasis (`**`) and round-trips it (AE-0289).
- **Risk:** Headings/bodies are markdown strings that reach the frontend violation panels and slide cards. If any render path uses `dangerouslySetInnerHTML` or an un-sanitized markdown renderer, preserved markdown becomes an XSS vector (a reviewer could inject `<img onerror>` or `[link](javascript:...)`). The packet's QA checklist says "sanitization preserves case without allowing markup injection" but does not name the frontend render sanitization path. Backend sanitization (AE-0289) preserves case + strips *some* markup, but markdown emphasis is intentionally kept.
- **Impact:** Stored XSS via slide copy if the frontend renders markdown raw.
- **Suggested mitigation:** Confirm the slide-copy render path uses a sanitized markdown renderer (e.g. DOMPurify over remark) for both the editor preview and the published output. Name the sanitization boundary in the ticket.
