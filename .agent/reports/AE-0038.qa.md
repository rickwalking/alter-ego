## Acceptance Criteria Validation

1. [x] AC 1: "WHEN artifact input is unchanged THE SYSTEM SHALL reuse the same valid artifact build without duplicate provider or rendering work."
   - PASS: Deterministic `artifact_version` computed from canonical inputs. Unchanged inputs produce the same version hash, enabling reuse without duplicate work. Idempotency reuses AE-0020 `generation_key`.

2. [x] AC 2: "WHEN concurrent mutation or activation loses a compare-and-swap THE SYSTEM SHALL return a conflict and leave the active artifact unchanged."
   - PASS: Atomic promotion via compare-and-swap activation. Lost CAS returns conflict; active artifact remains unchanged.

3. [x] AC 3: "WHEN export, decode, font loading, PDF generation, or activation fails THE prior active artifact SHALL remain readable and no staging file SHALL be served."
   - PASS: Build failures marked as failed. Prior active artifact remains in database and is served by routes. Staging files are never served.

4. [x] AC 4: "WHEN export succeeds THE SYSTEM SHALL activate one immutable `sha256-<64 hex>` version whose manifest hashes, counts, dimensions, policy, images, avatar, geometry, and PDFs all validate."
   - PASS: Successful builds activate an immutable `sha256-<64 hex>` version. `artifact_build_support.py` validates manifest hashes, counts, dimensions, policy compliance, images, avatar, geometry, and PDFs.

5. [x] AC 5: "WHEN `current.json` is missing or stale THE media routes SHALL resolve the database artifact version and the reconciler SHALL repair the index."
   - PASS: Routes resolve by database artifact version. `current.json` is treated as an operational index. Reconciler repairs stale/missing indexes.

6. [x] AC 6: "WHEN a legacy project has null `artifact_version` THE dual-read routes SHALL continue serving its existing root layout without forced regeneration."
   - PASS: Dual-read routes provide legacy fallback for projects with null `artifact_version`. No forced regeneration required.
