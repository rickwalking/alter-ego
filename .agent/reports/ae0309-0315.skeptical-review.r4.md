# Cold Critic Review

## Verdict
**BLOCK**

Three findings rise to design-level gaps where the plan, as written, will not achieve its stated guarantees (cancellation-marker coverage, severity-vs-stored-report consistency, deploy-time split-brain). The rest are WARN-level and fixable. The set is coherent in intent and the dependency graph (AE-0316 as foundation) is the right instinct — but the shared-surface claims that the whole set leans on are, in places, false against the code.

---

## Findings

### [BLOCKER] AE-0315: "3 chokepoints cover all carousel writes" is refuted — the cancellation-safety AC is unachievable as specified

- **Assumption:** A `run_cancelled` marker enforced at ~3 funnels — (a) the engine checkpoint-commit boundary (`aupdate_state` + node-return), (b) `CarouselProjectWriteOwner.commit()`, (c) the per-slide asset writer — intercepts every carousel mutation, so a reaped zombie task cannot land writes.
- **Risk:** Code verification finds **11+ mutation paths that bypass all three chokepoints**, and the highest-volume one is the biggest gap. `PostgresCarouselRepository.update_project` → `update_from_entity` → direct `session.commit()` (`carousel_repository.py:139`) is the funnel for ~15 distinct callers (finalize ×5, distribution ×2, visual ×3, images progress-fallback, refinement, crud) and it re-stamps **all** write-owner columns (`status`, `error_message`, `current_phase`, `phase_status`) directly — exactly the columns the cancellation marker must intercept. Additional unnamed bypasses: `PostgresCarouselArtifactBuildRepository.activate_build` is a **4th direct `UPDATE CarouselProjectModel` SQL** chokepoint (`carousel_artifact_build_repository.py:128`); `LegacyPublishingAcl.release_public` double-writes `current_phase` outside the owner (`legacy_publishing_acl.py:103`); `crud.py` create route commits a plain session; `reopen_carousel_for_resend.py` mutates WO columns + commits directly. The "per-slide asset writer" is **not singular** — there are ≥4 slide-write paths through two different committers (the generic repo's direct commit and `PresentationWriteOwner`'s commit).
- **Impact:** A reaped task that revives and hits any of these bypass paths (notably the finalize/distribution/visual `update_project` funnel, which runs during the very phases a zombie would be executing) writes through unguarded. The "Cancellation safety" AC and the "zombie revives → no write lands through each chokepoint" test will pass for the 3 named chokepoints and **silently miss** the paths that actually carry the bulk of mid-run writes. This is a false-safety guarantee — worse than no marker, because it removes the imperative to check.
- **Suggested mitigation:** Do not enumerate chokepoints by assumption. The ticket already requires a "write-site survey" as a completion criterion — **promote it to a pre-implementation gate**: enumerate every `carousel_projects` + checkpoint mutation site, and prove (grep-able assertion or lint rule) each routes through a marker-checked funnel *before* writing the marker logic. At minimum the generic-repo commit boundary (`update_project`/`create_slide`/`update_slide`/`activate_build`) must become a 4th (and 5th) enforced chokepoint, or the marker must move to a lower layer (the SQLAlchemy session commit / unit-of-work boundary) that all paths share.
- **Open question for author:** The `~28 sites` count is plausible (verification found ~30-34). But why does the plan claim 3 chokepoints cover them when `update_project` alone has ~15 callers and commits the full WO column set directly? Was the survey done, or estimated?

### [BLOCKER] AE-0312: severity model changes the blocking *decision* but not the stored *report* — client shows casing warnings as blocking while approval lets them through

- **Assumption:** Adding `severity` and changing `_blocking_from_repaired_localized` / `_blocking_from_drafts` makes casing warnings non-blocking, so "the reviewer can approve the phase."
- **Risk:** `validate_localized_slides` (`presentation_review.py:91-102`) **hardcodes `blocking=True`** in every report it produces (`build_validation_report(violations, blocking=True)`). That report is what gets **stored** in checkpoint state and what `resolve_presentation_review_from_state`'s fast path returns **verbatim** (verified by identity test `test_presentation_review.py:131-148`) to the SSE stream and the client. The blocking *decision* (`has_blocking_presentation_validation`, `_blocking_from_*`) is a **separate path** that re-derives. The ticket's scope lists changing `_blocking_from_*` but **not** `validate_localized_slides`. So after AE-0312: the approval gate computes `blocking=False` (casing is warning-tier) and lets the reviewer approve, but the stored/served report still says `blocking=True` — the UI violation panel renders casing warnings as **blocking**. The reviewer sees "blocking violations present" yet the approve button works. This directly contradicts the AC "casing violations render with distinct non-blocking styling" and the cross-ticket invariant "warnings pass through to the review panel without blocking approval" — the *panel* will show them as blocking because it reads the stored report.
- **Impact:** User confusion + a broken AE-0310 interaction: AE-0310 shows "blocking violations present → de-emphasize plain revise, show recovery actions" based on the report's `blocking` flag. With the stored report still `blocking=True` for casing-only violations, AE-0310's UI treats casing warnings as requiring recovery actions (edit/send-back) when they should be approvable. The two tickets fight each other on the same flag.
- **Suggested mitigation:** AE-0312 must also change `validate_localized_slides` to compute `blocking` from severity (or thread severity into `build_validation_report` and have `validate_localized_slides` call it without the literal `True`). The stored report and the approval decision must agree. Add an explicit AC: "the stored `presentation_validation` report's `blocking` field equals the severity-derived decision" + a test asserting the stored report for a casing-only violation set has `blocking=False`.
- **Open question for author:** Is the intent that the *stored* report be severity-aware, or only the approval gate? If the latter, how does the client violation panel show non-blocking styling when it reads the stored `blocking=True` report?

### [BLOCKER] AE-0315: deploy-time split-brain — every in_progress row with NULL heartbeat is reaped on the first post-deploy tick

- **Assumption:** The reaper reaps rows where `phase_status=in_progress` AND `run_heartbeat_at` is stale (>5min without heartbeat); a slow-but-alive run keeps its heartbeat fresh and is never reaped.
- **Risk:** The `run_heartbeat_at` column is **new** (this ticket's migration). Every in_progress row that existed *before* the migration has `run_heartbeat_at = NULL`. A NULL heartbeat is "without a heartbeat" — i.e. stale since epoch. The reaper's first tick after deploy will match **every** pre-existing in_progress row, including any run that is **genuinely active across the deploy** (a ~13-15min LLM run that was mid-flight when the deploy added the column). The heartbeat task can't have written to a column that didn't exist when the run started. This is the **exact split-brain the prod incident risked** — reaping a live run starts a second concurrent run. The ticket's own "backend restart killed a revision run" scenario is the intended behavior for *dead* runs, but a deploy that adds the column while a run is active produces the same row shape (in_progress + NULL heartbeat) for a *live* run.
- **Impact:** The first deploy of AE-0315 kills any in-flight carousel generation and risks a double-run race — the incident class the ticket exists to prevent.
- **Suggested mitigation:** Add a post-deploy grace period: the reaper treats `run_heartbeat_at IS NULL` as **not reapable** for a configurable window after the migration (or until the first non-NULL heartbeat is observed for that row), emitting only the `run_overdue` alert. Alternatively, the deploy procedure must drain/wait for zero in_progress rows before rolling. The AC "Reaper guard: rows with phase_status != in_progress are never reaped" should be extended with "rows with NULL heartbeat are alert-only until a heartbeat is observed or a grace period elapses."
- **Open question for author:** What happens to a run that started before the migration and is still alive when the reaper first ticks? How does the reaper distinguish "NULL because the column just shipped" from "NULL because the task died before heartbeating"?

### [WARN] AE-0314: the `Command(goto=...)` legacy-END-thread fallback is refuted — that recovery only fires for END-stuck gates, and no `goto=approved_hold` path exists

- **Assumption:** For legacy END-state threads (finalized before the hold-phase design), the engine's existing `ainvoke(Command(goto=...))` recovery (`carousel_workflow_engine.py:134-140`) re-parks the thread, then the edit is applied.
- **Risk:** Verification refutes this. `needs_gate_reopen` (`carousel_workflow_graph.py:55-70`) requires `snapshot.next` to be **empty** (workflow at END) AND `phase_status=awaiting_human` AND phase not approved. The `Command(goto=phase)` jumps back to `current_phase` — it is for "workflows stuck at END while still awaiting human review," and it targets `current_phase`, **never** `approved_hold`. There is no `Command(goto=PHASE_APPROVED_HOLD)` anywhere in the codebase. An approved END-state thread has `phase_status=approved` (not `awaiting_human`), so `needs_gate_reopen` returns False — the recovery is **unreachable** for the legacy-threads case the ticket describes.
- **Impact:** The "Fallback for legacy END-state threads" is not implementable as written. The residual fallback ("projection-only write still serves the PDF correctly + stale report suppressed via AE-0311's drift alert") is the *only* path — but it depends on AE-0311's reconciler, which is only "Related" not a declared dependency (see next finding).
- **Suggested mitigation:** Either (a) prove a working re-park mechanism for approved END-state threads as the "pre-implementation proof task #1" (the ticket wisely gates on this — hold the editor to it), or (b) drop the re-park fallback entirely and make the projection-only path the explicit design, with AE-0311's reconciler as a hard dependency for stale-report suppression. Do not ship the editor on the assumption the goto-recovery works.
- **Open question for author:** Has anyone confirmed `ainvoke(Command(goto=approved_hold, resume=...))` actually re-parks an END-state approved thread, or is this extrapolated from the END-stuck-gate path?

### [WARN] AE-0314 has an undeclared hard dependency on AE-0311's drift reconciler for partial-failure safety

- **Assumption:** AE-0314's post-completion edit writes projection + checkpoint; if the checkpoint write fails / thread is unrecoverable, "the stale checkpoint report is suppressed via the drift alert (AE-0311's reconciliation)."
- **Risk:** AE-0311 is listed as **Related**, not **Blocked by**. AE-0314's declared blockers are AE-0313 and AE-0310. If AE-0314 ships before AE-0311, a partial failure (projection committed, checkpoint write fails) leaves a stale blocking report served over fixed copy with **no reconciler to converge it** — the exact "stale report over fixed content" condition AE-0314's own source-of-truth rule exists to prevent.
- **Impact:** A user edits a completed carousel, the projection lands but the checkpoint write throws, and the publish page shows the old blocking violations permanently (until manual intervention) — the incident class the ticket is fixing.
- **Suggested mitigation:** Either declare AE-0311 as a hard blocker of AE-0314's post-completion flow, or move the drift reconciler into AE-0316 (the shared foundation) so both AE-0311 and AE-0314 own it jointly rather than one borrowing from the other.
- **Open question for author:** Why is AE-0311 "Related" when AE-0314's stated fallback explicitly invokes AE-0311's reconciler?

### [WARN] The "single serialization domain" (AE-0316 advisory lock) is leaky — the resume runner and the reaper do not acquire it

- **Assumption:** AE-0316's `carousel_project_lock` is "the single serialization domain shared with AE-0313's republish" so "a repair and a republish on the same project can never interleave."
- **Risk:** The advisory lock is acquired by AE-0311 (repair) and AE-0313 (republish). It is **not** acquired by the background resume runner (which writes checkpoint blobs via `ainvoke`, chokepoint a) or by AE-0315's reaper (which flips `phase_status` + reconciles checkpoint). So three actors mutate carousel state with only two holding the lock. A repair (holding the lock) can be mid-`aupdate_state` when the reaper (not holding it) flips the row to `awaiting_human` and reconciles the checkpoint — the repair's CAS checks `lock_version`, not `phase_status`, so it may not see the reaper's flip. The repair then writes checkpoint state while the reaper has declared the run dead. Similarly, a resume run (no lock) and a republish (holding lock) can interleave on checkpoint vs projection.
- **Impact:** The "single serialization domain" claim is true only for repair-vs-republish. Repair-vs-resume and repair/republish-vs-reaper remain unserialized — racing on two different commit domains (DB-row CAS vs checkpoint blobs) with no shared lock.
- **Suggested mitigation:** Either the resume runner and reaper also acquire `carousel_project_lock` (making it truly the single domain), or the plan must explicitly state the serialization boundary is "repair-vs-republish only" and document why resume-vs-repair safety is adequately covered by the lock_version CAS alone (and reaper-vs-repair by the marker). Don't call it a single domain if it isn't.
- **Open question for author:** Does the reaper bump `lock_version` when it flips `phase_status`? If not, a repair's CAS (lock_version-only) will not detect a concurrent reap.

### [WARN] AE-0312: auto-upgrading in-flight v1→v2 projects at deploy time produces casing warnings with no working repair button if AE-0311 hasn't shipped

- **Assumption:** Auto-upgrading in-flight v1→v2 is safe because "casing rules are warning-severity (non-blocking) — the worst outcome is new visible warnings with a working repair button."
- **Risk:** The "working repair button" is AE-0311's endpoint, which has **four dependencies** (AE-0316, AE-0309, AE-0310 for the design button, soft AE-0312). AE-0312 itself has no blockers and no blocks — it can ship independently and **before** AE-0311. If AE-0312's migration runs at deploy time and AE-0311 is not yet live, every in-flight project suddenly shows casing warnings with no endpoint to fix them. The user sees warnings, clicks "Fix issues automatically," and gets nothing (button doesn't exist). `repair_casing` exists in AE-0312's code, but it's only reachable through AE-0311's endpoint or the drafting pipeline — not the UI.
- **Impact:** The highest-review-traffic cohort (in-flight at deploy) gets unfixable warnings — a degraded review experience on every carousel, which is the opposite of the ticket's goal.
- **Suggested mitigation:** Make the in-flight auto-upgrade migration conditional on AE-0311 being live (declare the deploy ordering), OR ship the auto-upgrade as a separate follow-up after AE-0311, OR have AE-0312 itself expose a minimal casing-repair affordance that doesn't depend on AE-0311's full endpoint. At minimum, document that AE-0312 must not deploy its migration before AE-0311.
- **Open question for author:** Is the deploy order constrained so AE-0311 precedes AE-0312's migration? Nothing in the dependency graph enforces it.

### [WARN] AE-0313: the fresh-build path has an *inverted* stale-index risk the ticket doesn't fix

- **Assumption:** The ticket adds `write_current_index` to `_activate_existing` so `current.json` reflects the re-activated version immediately after a 200.
- **Risk:** In the **fresh-build** path (`build_and_activate`, `artifact_build_service.py:128-129`), `write_current_index` runs **before** `activate_build`. If `activate_build` raises `ERR_ARTIFACT_BUILD_CONFLICT` (CAS fails — the exact concurrency the new advisory lock is meant to serialize), `current.json` has already been written pointing at a version that **never activated in the DB** — disk and DB diverge in the *opposite* direction from the `_activate_existing` gap. The ticket's planned fix (add `write_current_index` to `_activate_existing`) does not address this pre-existing inverted risk; adding the advisory lock around the whole critical section mitigates the CAS-failure window but the ordering (write index before commit) remains fragile.
- **Impact:** A concurrent republish that loses the CAS leaves `current.json` pointing at a phantom version. The read-path reconciler self-heals on next read, but a user who downloads immediately after the 409 gets the wrong/stale PDF.
- **Suggested mitigation:** Move `write_current_index` to **after** a successful activation commit in *both* paths (`_activate_existing` and `build_and_activate`), inside the advisory lock critical section. The ticket already pins "lock scope acquired at `build_and_activate` entry... released after the DB commit" — extend the scope fix to reorder the index write.
- **Open question for author:** Why write the on-disk index before the DB activation succeeds in the fresh path? Is there a read that depends on `current.json` existing pre-activation?

### [WARN] AE-0310: changing `validate_revision_cap` to be target-phase-aware has blast radius on the existing final_review→content send-back path

- **Assumption:** The pinned AC ("a design→content send-back increments `revision_count[content]`; the cap check evaluates the counter of the phase that would re-run") fixes the cap accounting.
- **Risk:** Verification confirms a **pre-existing divergence**: today the cap *check* (`validate_revision_cap`) reads `revision_count[current_phase]`, but the *increment* (`persist_phase_feedback`) bumps `revision_count[target_phase]` on send-backs. They already disagree for the **existing** final_review→content send-back. AE-0310's fix (make the check target-aware) touches the **shared** `validate_revision_cap` used by all phases — changing its read from `current_phase` to `target_phase` affects the final_review send-back path too, not just the new design path. If the change is "check the target phase on send-backs, current phase on plain revises," the branch logic must be correct for both, and the existing final_review tests must not regress.
- **Impact:** A subtle cap-accounting regression on the existing final_review→content path (e.g. the cap is now checked against content instead of final_review, changing how many send-backs are allowed before exhaustion).
- **Suggested mitigation:** Add a regression test for the **existing** final_review→content send-back cap behavior (not just the new design→content one) asserting the post-fix counter semantics are intended for both paths. The Gherkin "Send-backs consume the target phase's revision budget" scenario should cover both source phases.
- **Open question for author:** Is the pre-existing check/increment divergence a known bug AE-0310 is fixing, or was it assumed to already be consistent? The Gherkin doesn't mention final_review→content.

### [INFO] AE-0312: "route new carousels to v2" has no mechanism today — selection is stored-value-or-default, with no new-project stamping path

- **Assumption:** Ship `hero_lower_third_v2`, add it to `SUPPORTED_PRESENTATION_POLICY_VERSIONS`, "route new carousels to v2."
- **Risk:** Verification confirms there is **no new-carousels routing logic** today. Policy selection is `policy_version or DEFAULT_PRESENTATION_POLICY_VERSION` (`presentation_review.py:97`) — i.e. the project's stored value, falling back to a constant. New projects are not stamped with a policy version at creation (no stamping path exists). So "route new carousels to v2" requires either changing `DEFAULT_PRESENTATION_POLICY_VERSION` to v2 (which affects *all* projects with NULL stored version, including completed v1-era rows on any re-validation read) or adding a create-time stamping path that doesn't exist. The implementation plan doesn't mention either.
- **Impact:** Either the routing doesn't happen (new carousels silently stay v1), or changing the default has unintended blast radius on legacy NULL-version rows.
- **Suggested mitigation:** Add an explicit create-time stamping step (set `presentation_policy_version=hero_lower_third_v2` on project creation) and keep `DEFAULT_PRESENTATION_POLICY_VERSION` as v1 for legacy fallback. Document this in the implementation plan.

### [INFO] AE-0313: "no client path at all" to re-render a completed project is overstated

- The strategy endpoint `PUT /carousels/{id}/strategy` is **already gated on `status == COMPLETED`** (`strategies.py:75-79`) and calls `re_render_slides`. So a client path exists — it just forces a layout-strategy change. `re_render_slides` is also reachable from the admin `render_slides` loop and `CarouselFormatProducer.produce` (6 call sites total, not 1). The precise gap is "no client path that re-renders *without* forcing a strategy change and *builds a new version*" — which is what the ticket delivers. The overstated framing isn't a blocker, but the regression fixture and scope should acknowledge the existing strategy endpoint to avoid duplicating or conflicting with it.

## Missing evidence

- **AE-0315 write-site survey is a completion criterion but should be a *pre-implementation* gate.** Without it, the 3-chokepoint claim is an unverified premise. The survey must list `PostgresCarouselRepository.update_project` (the ~15-caller funnel), `activate_build` (direct SQL UPDATE), `release_public`, crud routes, and scripts, and prove each is either routed through a marker-checked funnel or explicitly out-of-scope for cancellation.
- **AE-0314's "pre-implementation proof" (aupdate_state on approved-hold thread)** is correctly gated as task #1, but the *legacy END-thread fallback* has no proof and rests on a refuted mechanism. Evidence needed: either a working `Command(goto=approved_hold)` re-park for approved END-state threads, or a decision to drop that fallback.
- **AE-0312 severity model:** no evidence that `validate_localized_slides` (the stored-report producer) will be changed, only the approval-gate decision. Evidence needed: a test asserting the *stored* report's `blocking` field for a casing-only violation set equals the severity-derived decision.
- **AE-0315 reaper vs repair interaction:** no evidence on whether the reaper bumps `lock_version` (determining whether a concurrent repair's CAS detects the reap), nor whether the reaper acquires the advisory lock.
- **AE-0312 deploy ordering:** no evidence that AE-0311 will be live before AE-0312's in-flight auto-upgrade migration runs.
- **AE-0313:** no evidence the fresh-build path's `write_current_index`-before-`activate_build` ordering will be fixed alongside the `_activate_existing` fix.

## Residual risks if plan proceeds unchanged

- **AE-0315** ships a cancellation marker that passes its own tests while ~15 high-traffic mid-run write paths (the `update_project` funnel) bypass it — a reaped zombie reviving during finalize/distribution/visual writes lands them unguarded. The "cancellation safety" guarantee is theatrical.
- **AE-0312** ships warning-tier casing rules whose stored report says `blocking=True` — the UI shows them as blocking, AE-0310's UI treats them as requiring recovery, and the reviewer sees "blocking violations" while the approve button works. The severity model's user-visible value is undermined.
- **AE-0315's first deploy** reaps every in-flight run with a NULL heartbeat, including live ones — risking the double-run race the ticket exists to prevent.
- **AE-0314's post-completion editor** relies on a refuted re-park mechanism and an undeclared AE-0311 dependency for partial-failure safety — a checkpoint-write failure leaves stale blocking violations served permanently.
- **The advisory lock** is called a "single serialization domain" but the resume runner and reaper don't hold it — repair-vs-resume and repair-vs-reaper races on checkpoint state remain unserialized.
