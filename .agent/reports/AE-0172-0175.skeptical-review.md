## Findings

### [BLOCKER] AE-0172 baseline-ratchet is underspecified for “net-new” enforcement
- Assumption: A grandfathered count plus allowlist can reliably block net-new dead exports/files.
- Risk: A count-only ratchet can miss replacement churn: one old unused export removed, one new unused export added, total count unchanged. Broad allowlists can also hide new issues under existing patterns.
- Impact: The gate may create confidence without actually enforcing the claimed invariant.
- Suggested mitigation: Baseline findings by stable identity: rule type, file path, export/file name, and normalized reason. Require tests for add, remove, replace-same-count, rename, and allowlist scope.
- Open question for author: Is the proposed baseline keyed by findings, or only by total count?

### [WARN] Gate proliferation lacks an operating model
- Assumption: Adding knip while ratcheting jscpd will remain low-friction because both mirror existing gates.
- Risk: The frontend already has many gates. Another CI job, baseline file, integrity rule, advisory mode, docs, seeded tests, and threshold ratchet adds maintenance load and failure triage cost.
- Impact: Developers may learn to ignore advisory failures, or spend time debugging tool noise instead of product defects.
- Suggested mitigation: Define a gate budget: runtime target, owner, advisory expiry date, escalation rule, and “what failure means” runbook.
- Open question for author: Who owns false positives and dependency/tool drift after the introductory wave?

### [WARN] Advisory-first CI can normalize drift instead of preventing it
- Assumption: `continue-on-error: true` is safe during introduction.
- Risk: New unused exports can merge while the job is advisory. If the baseline cannot be raised, CI may stay noisy; if it can be refreshed, the ratchet principle is weakened.
- Impact: The project may accumulate fresh debt exactly while claiming the new gate prevents it.
- Suggested mitigation: Keep the first discovery advisory, but make net-new findings blocking after a short fixed window, or block only files changed in the PR from day one.
- Open question for author: What exact event flips AE-0172 from advisory to blocking?

### [WARN] AE-0174 risks metric-chasing and harmful coupling
- Assumption: Remaining jscpd clones are good extraction candidates.
- Risk: Several examples sound structurally similar but semantically separate: route error boundaries, dashboard pages, login/header/sidebar markup, admin create/edit dialogs. Extracting them mainly to lower a percentage can couple unrelated behavior.
- Impact: Future changes become harder, tests become less local, and Next.js server/client boundaries may be accidentally changed.
- Suggested mitigation: Require per-hotspot justification: shared concept, expected future shared evolution, server/client compatibility, and tests. Allow explicit waivers for readable duplication.
- Open question for author: Which listed clones are semantically shared, not merely syntactically similar?

### [WARN] Sequencing between AE-0172 and AE-0174 is not defined
- Assumption: Dead-code and duplication ratchets can be implemented independently.
- Risk: Duplication refactors can create new shared primitives, move exports, alter barrel behavior, and change knip results. Knip baseline timing before or after AE-0174 matters.
- Impact: Baselines may churn immediately, producing noisy reviews and unclear accountability.
- Suggested mitigation: Sequence deliberately: either run AE-0174 first, then snapshot knip, or implement AE-0172 with changed-file-only enforcement until refactor churn settles.
- Open question for author: Which ticket establishes the authoritative frontend quality baseline?

### [WARN] AE-0173 may turn testing policy into a loophole
- Assumption: “Pure refactor” and “CI/config” are easy to classify.
- Risk: Behavior-changing work can be labeled as refactor/config to avoid `.feature` scenarios. CI/config changes can still affect user-visible release safety.
- Impact: QA warnings may disappear, but the ambiguity moves upstream into ticket classification.
- Suggested mitigation: Add explicit examples and required evidence: no public behavior change, seeded violation test, affected gates, and reviewer sign-off for no `.feature`.
- Open question for author: Who decides whether a ticket is behavior-changing when the author and QA disagree?

### [WARN] AE-0175’s proposed tests may over-mock the risky behavior
- Assumption: Mocking `streamSseEvents`, `useConversationMessages`, and QueryClient is enough to cover `useSseChat`.
- Risk: The highest-risk paths are concurrency and lifecycle behavior: abort during stream, double send, unmount, stale history merge, React 19 scheduling, `finalizedRef` idempotency, and query invalidation ordering.
- Impact: Tests may lock in implementation details while missing the regressions users would see.
- Suggested mitigation: Use a real QueryClient where possible and a controllable fake async stream. Add race-oriented cases, not only branch coverage.
- Open question for author: Will tests cover overlapping sends, abort-before-complete, unmount-before-complete, and history arriving during an optimistic stream?

### [INFO] Security and abuse posture is mostly absent from the portfolio
- Assumption: These are quality/debt items, not security work.
- Risk: For a full-stack RAG app, unused code, duplication, and hook tests do not address tenant isolation, prompt injection, source leakage, auth boundaries, or unsafe document exposure.
- Impact: The portfolio may be mistaken for broad hardening when it is mainly maintainability work.
- Suggested mitigation: Label the portfolio honestly as frontend quality/testability debt, and do not count it as reducing RAG security risk.
- Open question for author: Are there separate security/abuse work items, or is this portfolio expected to cover that risk?

## Missing evidence
- Current runtime cost of each frontend gate and projected knip runtime.
- Existing baseline-ratchet implementation details: count-only vs identity-based.
- Actual knip first-run output and false-positive categories.
- CI workflow topology and whether advisory jobs block branch protection.
- Current docs authority model across `CLAUDE.md`, `AGENTS.md`, and QA guides.
- Test harness details for React 19 hooks, timers, QueryClient, and SSE mocks.
- Evidence that each AE-0174 clone is semantically safe to extract.

## Residual risks if plan proceeds unchanged
- New dead exports may still slip through via baseline churn or broad allowlists.
- CI may become slower and noisier without a clear owner or expiry for advisory gates.
- jscpd improvements may reduce a metric while increasing coupling.
- Testing-policy docs may reduce QA noise by weakening classification discipline.
- `useSseChat` tests may miss concurrency bugs despite increasing test count.
- The portfolio may consume quality bandwidth while leaving RAG-specific security and abuse risks untouched.
