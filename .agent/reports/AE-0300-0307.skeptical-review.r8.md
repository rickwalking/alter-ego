# Cold Critic Review

## Verdict
WARN

## Findings

### [WARN] AE-0303: the auto-reboot lock-honoring mechanism is asserted but never connected
- Assumption: "The auto-reboot skips while the lock is held" (scope + AC) implies `unattended-upgrades` consults the app lock.
- Risk: `unattended-upgrades` has **no native concept of an arbitrary app lock file**. The implementation plan (steps 1–5) sets `Automatic-Reboot "true"` and defines the lock lifecycle, but contains **no step that wires the lock check into `unattended-upgrades`** (e.g., a `Dpkg::Pre-Install-Pkgs` hook, an `Unattended-Upgrade::OnlyOnACPower`-style gate, or a wrapper that disables rebooting). The gate (lock) and the actor (unattended-upgrades) are described independently with no bridge between them.
- Impact: AC "auto-reboot … does not fire while a deploy or rotation is in flight (lock-file honored)" is unverifiable as specified — the most likely implementation silently fails to honor the lock, and a kernel reboot fires mid-rotation/`.env`-write in the exact compound-failure the lock exists to prevent.
- Suggested mitigation: Specify the concrete hook (e.g., a `/etc/apt/apt.conf.d/99-lock-guard` that exits non-zero when the lock is held, or disable `Automatic-Reboot` and run a **cron+lock-aware reboot script**). Add it as an implementation step and an AC that proves a held lock prevents the reboot path.
- Open question for author: Which `unattended-upgrades`/reboot mechanism will actually consult the lock, and how is it tested without a real mid-rotation reboot?
- Classification: genuine defect (design gap), not (a)/(b)/(c). R7 addressed the lock's *lifecycle* but not its *consumer wiring*.

### [WARN] AE-0306: POSTGRES_PASSWORD rotation has no in-flight-drain AC, unlike SECRET_KEY
- Assumption: The drain/transition concern applies only to session-signing keys (the drain AC is scoped to `SECRET_KEY`/`ANON_SECRET_KEY`).
- Risk: A carousel mid-generation holds an open DB connection writing checkpoints; the rotation **restarts all consumers** (compose `env_file` read once at start). On `ALTER ROLE … PASSWORD`, existing Postgres sessions survive, but the orchestrated full-stack restart drops them — an in-flight carousel's checkpoint write can fail or strand mid-rotation. The AC enumerates in-flight signed state for session keys but not DB-dependent in-flight work.
- Impact: The Gherkin "failed rotation recovery" scenario passes while a carousel silently corrupts/fails its checkpoint stream during the restart window.
- Suggested mitigation: Mirror the session-key drain AC for POSTGRES: "no in-flight carousel generation during the rotation window" OR prove idempotent checkpoint resume survives a forced mid-DB-restart. Add an AC.
- Open question for author: Does carousel checkpoint resume tolerate losing its DB connection mid-write, and is that tested?
- Classification: genuine defect (AC asymmetry across same risk class). Drain was added to session keys by an earlier round; the analogous gap for DB creds was not closed.

### [WARN] AE-0306: no pre-rotation DB backup gate despite verified absence of any backup cron
- Assumption: Deferring old-password revocation until post-change smoke ("rollback as code") substitutes for a backup.
- Risk: The packet explicitly records **no `pg_dump`/backup cron exists today**. A POSTGRES_PASSWORD rotation that bricks connectivity in a way the smoke misses (e.g., a non-app consumer mis-set, an Alembic head mismatch surfacing only on next deploy) leaves no recent DB restore point. "Reversible" by re-issuing old creds relies on the old password still being live — but if `ALTER ROLE` partially applied or a downstream consumer is wedged, there is no DB snapshot to fall back to.
- Impact: A rotation incident becomes irrecoverable without data loss, against a security ticket whose own justification is a known exposure window.
- Suggested mitigation: Add an AC: "A `pg_dump` (or verified snapshot) is taken immediately before POSTGRES_PASSWORD rotation and stored off-box." Cheap, and it closes the gap that the packet itself flagged as "worth a separate ticket" — but that separate ticket does not block this one.
- Open question for author: Why is rotation allowed to precede the (acknowledged-missing) backup ticket?
- Classification: genuine defect (incomplete). The missing-backup gap is named but not gated.

### [WARN] AE-0307: the recommended Tailscale path adds an unaddressed external control-plane SPOF for deploys
- Assumption: Tailscale (or an off-box self-hosted runner) is "least public surface" and therefore preferred.
- Risk: With Tailscale, **the deploy path now depends on Tailscale's coordination/control plane**. A Tailscale outage (or auth-key expiry, or tailnet ACL change) blocks all deploys — and the documented DR is the DO console break-glass, which is **not** a deploy path (it cannot run `deploy.yml`). The packet enumerates ingress-node hardening, rotation, and compromise handling, but not control-plane-dependency handling.
- Impact: A hardened-ingress design trades a known problem (public-22 pinning) for a quieter one (deploys silently impossible during a Tailscale-side incident), with no fallback deploy path defined.
- Suggested mitigation: Either (i) keep a second, independent deploy path for DR (e.g., the DO console + a documented manual `docker compose pull && up` runbook), or (ii) document the Tailscale-outage deploy-DR procedure explicitly as an AC. State which.
- Open question for author: What is the deploy procedure when the Tailscale control plane is unreachable, and is it drilled?
- Classification: genuine defect (incomplete). Prior rounds hardened the ingress node; none addressed the control-plane dependency the chosen option introduces.

### [WARN] AE-0301: the cheap-key rotation AC is not gated on the per-provider exerciser-name artifact existing
- Assumption: The prose "pre-list, per provider, the exact in-use assertion up front" is enforced by an AC.
- Risk: The only rotation AC is a single bullet: "cheap-to-rotate keys … are rotated … old values are revoked only after a post-deploy smoke confirms the new ones are in use." It does **not** require the per-provider exerciser-name list to exist as a checked artifact before execution. The detailed prose (name GLM's exerciser, Pinecone's exerciser) is process, not an enforceable gate. A developer can satisfy the AC with a single bundled carousel smoke (OpenAI/Anthropic only) and revoke GLM/Pinecone keys on faith — the exact failure the prose warns against.
- Impact: A non-OpenAI/Anthropic provider is stranded after revocation, with the AC nominally green.
- Suggested mitigation: Split the rotation AC per provider, each requiring its named exerciser's "live and in use" assertion as a precondition to that provider's revoke step; state the exclusion rationale inline for any provider with no exerciser (Gemini).
- Open question for author: Why is the per-provider exerciser list prose and not a per-provider AC?
- Classification: genuine defect (gate is soft where it must be hard). The prose was added by an earlier round but never promoted to an AC.

### [WARN] AE-0303: auto-reboot is OFF until AE-0306 completes, with no slip-escalation
- Assumption: "Do not enable auto-reboot until AE-0306 rotation is complete" ties the safety net to a hard gate.
- Risk: AE-0306 has a 14-day deadline (2026-07-15) with an "unless an extension is logged" escape hatch. If AE-0306 slips (it is the most complex ticket in the packet — multi-secret, Langfuse re-encryption capability unknown, Postgres rotation), **auto-reboot stays silently disabled indefinitely** and the droplet resumes kernel/library drift — the precise problem AE-0303 was created to solve. There is no AC tying auto-reboot-enable to either AE-0306 completion *or* a slip-handler that re-drifts the box.
- Impact: The long-term auto-reboot safety net depends on a separate, slip-prone ticket with no compounding guard.
- Suggested mitigation: Add an AC: auto-reboot-enable is gated on AE-0306-close **or** a logged extension with a re-baseline date; if the extension lapses, auto-reboot is enabled with a rotation-safe default (lock honored) regardless. State the default.
- Open question for author: What happens to the drift-prevention goal if AE-0306 slips past 2026-07-15?
- Classification: genuine defect (incomplete). The lock interaction was added (good), but the cross-ticket dependency's slip behavior is undefined.

### [WARN] AE-0305: CSP env-gating mechanism for multi-source (nginx + frontend) is unspecified
- Assumption: "Use an explicit environment signal (`NODE_ENV`/`ENVIRONMENT`) so dev keeps the entry and prod does not."
- Risk: The ticket requires grepping **both** nginx and the frontend for the CSP, but does not specify how a static nginx CSP header is **env-gated**. nginx headers are literals; gating requires templating the nginx config per environment (two configs, or `sub_filter`/`envsubst` on deploy) — or the CSP must live solely in Next.js. If CSP is set in both layers, the gating mechanism (which layer is the truth, how the other is kept in sync across envs) is not designed.
- Impact: A future dev fixes one layer, leaves the other, and `curl -I` (the AC check) passes from one route while a different route serves the dev entry. The regression guard helps but is itself gated on testing the right routes.
- Suggested mitigation: Decide and state the single CSP source of truth (nginx xor Next.js) and how the other is prevented from drifting; if both must remain, specify the templating mechanism.
- Open question for author: Which layer ultimately owns the CSP, and is the other layer verifiably silent?
- Classification: incomplete (the gating mechanism is asserted, not designed). Borderline INFO; flagged WARN because CSP-bypass via a dev artifact is the very bug being fixed.

### [INFO] AE-0302: REDIS_URL vs REDIS_PASSWORD precedence is under-specified
- Assumption: "Define precedence vs any existing `REDIS_URL`/`redis://` string" resolves all cases.
- Risk: The rule for a `REDIS_URL` with embedded credentials that **conflict** with `REDIS_PASSWORD` (or where the URL has a password and `REDIS_PASSWORD` is unset) is not stated. "Which wins" matters because a future `.env` carrying a stale `REDIS_URL` could silently re-open an unauthed path or auth with the wrong credential.
- Impact: Low — both prod and dev currently likely use one form — but the enforcement factory can't reject a contradictory input it hasn't been told how to rank.
- Suggested mitigation: State explicitly: in prod, `REDIS_URL` without embedded non-empty credentials = missing credentials; if both set and conflicting, fail-closed with a clear `ConfigError`.
- Open question for author: What is the documented precedence and conflict behavior?
- Classification: incomplete (implementation edge not pinned); likely acceptable as developer detail but worth one sentence.

## Missing evidence
- The mechanism by which `unattended-upgrades` (or the reboot path AE-0303 configures) consults the app lock — no hook/wrapper is described or testable.
- Carousel checkpoint resume behavior under a forced mid-write DB disconnect (AE-0306 POSTGRES rotation drain question).
- Langfuse 3.185's actual encryption-key-rotation/re-encryption capability — the ticket correctly flags this as a pre-execution unknown, but the AC for "pre-rotation traces remain readable" does not state **how** readability is proven (which traces, which read path, against what expected output).
- Which CSP layer (nginx vs Next.js) is authoritative in production — needed to validate AE-0305's `curl -I` AC against the right route.
- The staleness-alert sink for AE-0300's CF-range refresh (AE-0303 names its smoke sink; AE-0300's staleness alert does not name a sink — "emits a staleness/last-refresh alert" is otherwise unspecified).
- Tailscale control-plane outage DR deploy procedure (AE-0307) — not in the packet or the rollback section.

## Residual risks if plan proceeds unchanged
- A `unattended-upgrades` auto-reboot fires mid-rotation/`.env`-write because the lock is honored by nothing wired to the reboot actor (highest-impact residual).
- A carousel strands mid-generation during POSTGRES_PASSWORD rotation because drain was required for session keys but not for DB-dependent in-flight work.
- An incident during expensive-secret rotation is irrecoverable without data loss because no pre-rotation backup was taken (the missing-backup gap was named but not gated).
- Deploys become impossible during a Tailscale-side incident with no deploy-DR path defined.
- The droplet re-drifts on kernel/library patches indefinitely if AE-0306 slips past its deadline, because auto-reboot stays off with no slip-escalation.
- A non-OpenAI/Anthropic API key is revoked on faith because the per-provider exerciser-name list is prose, not a per-provider AC gate.
- A future CSP drift between nginx and Next.js re-exposes the dev artifact in prod because the gating mechanism and authoritative layer are unspecified.
