# Cold Critic Review

## Verdict
BLOCK

The set is well-reasoned and the tickets have clearly absorbed prior cold-critic rounds (revision 6 is visibly hardened). However, two cross-ticket concurrency claims do not survive scrutiny against the stated system facts, and several shared-surface gaps remain underspecified. The tickets should not merge as-is.

---

## Findings

### [BLOCKER] Two-commit seam in AE-0311 is not serialized against concurrent resume — advisory lock does not cover resume, CAS is point-in-time

- **Assumption:** AE-0311 states "Resume-vs-repair is serialized by the lock_version CAS (both bump it)" and the repair "acquires the per-project advisory lock held across BOTH commits." AE-0316 confirms "the background resume runner deliberately do[es] NOT acquire the advisory lock."
- **Risk:** The advisory lock serializes repair ↔ republish ↔ edit (the artifact-mutator domain) but resume is explicitly outside it. The `lock_version` CAS is a single conditional UPDATE — it is a *gate*, not a *held lock*. Sequence: (1) repair passes `ensure_resume_not_in_progress`, CAS-bumps lock_version 5→6, commits projection (carousel_slides fixed); (2) a client that read state *after* the CAS bump (lock_version=6) submits a resume — the server's CAS 6→7 succeeds because the client's version matches the row; (3) resume starts the workflow run, which reads the **stale checkpoint** (the repair hasn't committed the checkpoint yet). The workflow re-generates from unrepaired content — wasted LLM spend, broken content re-surfaced. The partial-failure autonomous reconciler is tick-based and asynchronous, so it does not close this window.
- **Impact:** Wasted regeneration (~15 min LLM run on stale content), the exact corruption class AE-0309 was written to prevent re-appearing at a different layer, and a user-facing "I repaired it but it's still broken" experience. The claim that CAS "serializes" resume-vs-repair overstates the guarantee — it serializes the *start* of each mutation, not their *duration*.
- **Suggested mitigation:** The repair must acquire a guard that resume also checks for its full duration. Options: (a) the repair sets a lightweight DB-level marker (e.g., repair-specific advisory lock variant that resume's `ensure_resume_not_in_progress` also checks), (b) the repair holds a row-level `SELECT ... FOR UPDATE` on the carousel_projects row across both commits (resume's lock_version read blocks until the row-lock releases), or (c) defer the CAS bump to *after* both commits so the checkpoint is already fresh when any post-CAS resume reads it (though this reopens the partial-failure window in the other direction). At minimum, the ticket must reject the current "CAS is sufficient" claim and specify the held-guard mechanism.
- **Open question for author:** Does the repair set any in-progress marker that `ensure_resume_not_in_progress` checks? If not, what prevents a resume from proceeding in the window between the projection commit and the checkpoint commit? Is `bump_resume_lock_version` a true atomic conditional UPDATE or a read-then-write?

### [BLOCKER] AE-0315's epoch-fence enforcement layer for non-run writes is undefined — the "carry the current epoch by construction" claim is unproven

- **Assumption:** AE-0315 states the SQLAlchemy flush/commit-boundary guard "rejects mutations to CarouselProjectModel/CarouselSlideModel rows when the writer's captured run_epoch mismatches the row's current epoch" and "writes not owned by any run (user API calls, admin actions) carry the current epoch by construction (read-modify-write within their own transaction)."
- **Risk:** "By construction" is asserted but never mechanism-specified. At READ COMMITTED (Postgres default), a user API call reads epoch=5, the reaper bumps to 6 (separate committed transaction), and the user's flush sees row epoch=6 vs captured 5 → **rejected** — a legitimate edit fails. The reaper is one-shot per dead run (it flips in_progress → awaiting_human and never reaps the same row again), which bounds the window to "user edit started before reaper fired." But since the reaper fires *because* phase_status is still in_progress, the user shouldn't have been able to start editing (the 409 blocks them) — so the real question is the *mechanism*: how does the session-event guard know a flush is "run-owned" (has a captured epoch) vs "user-owned" (no captured epoch, should always pass)? If the mechanism is "absent captured epoch = allow," then the guard is correct but the ticket must say so. If it's "stamp every write with a read-epoch," legitimate writes fail under concurrent reap.
- **Impact:** Either (a) silent false rejections of legitimate user/admin writes (if the guard stamps all writes) or (b) a zombie write slipping through (if the guard is too permissive for non-run sessions). The blast radius is the *entire* carousel write surface — the ORM flush guard runs on every CarouselProjectModel/CarouselSlideModel commit. A bug here rejects all writes.
- **Suggested mitigation:** Explicitly define the session-level mechanism: a context variable or session flag that marks "this session is a run with captured epoch N." The guard checks the flag — if set, compare; if unset, pass. Document this in the ticket. Add a test that exercises a non-run write concurrently with a reaper epoch bump and asserts the write is NOT rejected (or is retried). Specify the application's DB isolation level and justify the claim under it.
- **Open question for author:** What database isolation level do carousel reads/writes use? How does the flush guard distinguish a run-owned session from a user-API session? What happens when a non-run write's captured epoch is stale — is there a retry, or is the user stuck?

### [WARN] AE-0314/0311 edit→republish chaining releases the advisory lock between the two HTTP calls — non-atomic PDF propagation with no browser-close recovery

- **Assumption:** AE-0311 states "the frontend chains the AE-0313 republish action (single user gesture)." AE-0314 says "publish page edit flow persists the change and chains republish." Both describe frontend-side two-request chaining.
- **Risk:** The edit endpoint commits the projection + checkpoint (holding the advisory lock), releases the lock, returns 200. The frontend then calls the republish endpoint (which re-acquires the lock). Between these calls, (a) a concurrent repair or another edit can interleave on the same project (the lock is free), and (b) if the user closes the browser after the edit 200 but before the republish call, the DB and checkpoint hold corrected text but the **served PDF is stale** — the entire point of the flow is defeated. There is no server-side guarantee that the republish happens.
- **Impact:** A user fixes casing, sees the edit succeed, closes the tab, and the public PDF still has lowercase "claude" — the exact 66014ba3 incident class, just moved from "no path at all" to "path exists but isn't guaranteed."
- **Suggested mitigation:** Either (a) make the edit endpoint chain the republish server-side in a single request (holding the advisory lock across both — but verify pg_advisory_lock reentrancy if they share a session), or (b) have the edit endpoint enqueue a republish job (idempotent) that the drift reconciler guarantees, or (c) accept the gap explicitly in the ticket and add a publish-page health check that shows "PDF needs rebuild" after an edit without a republish.
- **Open question for author:** Is the chaining guaranteed server-side or is it a client obligation? What happens if the republish HTTP call never arrives (browser close, network drop)?

### [WARN] AE-0315's reaper↔checkpoint reconciliation for "mid-step" state relies on unspecified LangGraph resumption semantics

- **Assumption:** AE-0315 states the reaper "reconcile[s] against the checkpoint (checkpoint parked → flip row to match; checkpoint mid-step → reset to awaiting_human for a clean re-resume)."
- **Risk:** "Mid-step" implies the workflow was between node boundaries when the task died. LangGraph checkpoints at node boundaries; an interrupted workflow resumes from the interrupt point. Resetting the row's `phase_status` to `awaiting_human` and letting the user resume does *not* reset the checkpoint — the workflow engine will resume from whatever the last checkpointed state was. If that state was mid-generation (partial slides written), the resume could produce a half-built artifact. The ticket doesn't specify whether the reaper also resets checkpoint state (it says it only flips the row) or whether LangGraph's resume semantics handle mid-step gracefully.
- **Impact:** A reaped dead run that was mid-generation could resume into a partially-built carousel, producing inconsistent output that downstream validation may or may not catch.
- **Suggested mitigation:** Specify what "clean re-resume" means at the checkpoint level. Does the reaper issue a checkpoint reset? Does it rewind to the last human-approval interrupt? Document the LangGraph behavior for resuming from a mid-node checkpoint after a `phase_status` flip.

### [WARN] AE-0310's cap-accounting fix changes behavior on the *existing* final_review→content send-back path — in-flight projects may be newly rejected

- **Assumption:** AE-0310 states "today the cap CHECK reads `revision_count[current_phase]` while the send-back INCREMENT bumps `revision_count[target_phase]`" and the fix "makes the check target-aware for ALL send-back sources." It acknowledges "a rare project that accumulated ≥cap content send-backs under the old divergent accounting is newly rejected."
- **Risk:** The old divergent check read `revision_count[final_review]` (current phase) against the cap, while send-backs incremented `revision_count[content]` (target). A project with, say, 5 content send-backs (target counter full) but 0 final_review send-backs (current counter empty) would *pass* the old check and *fail* the new check. The ticket calls this "acceptable because the uncapped edit escape hatch always remains" and defers release notes. This is a behavior change on a path that is live in prod today.
- **Impact:** A user mid-workflow could suddenly find their send-back rejected with a cap error they never saw before, on a project that was progressing fine. The uncapped edit escape mitigates the deadlock, but the UX regression is real and the user may not know to use the edit path instead of send-back.
- **Suggested mitigation:** Add a transient migration-time guard: when the new target-aware check would reject a project that accumulated send-backs before the deploy date, surface a *guidance message* ("use the inline edit path instead of send-back") rather than a bare cap error. Or log the affected cohort and proactively notify operators.
- **Open question for author:** How many in-flight projects currently have `revision_count[content]` ≥ cap? Is there a query to enumerate the affected cohort before deploy?

### [WARN] AE-0312's run-once migration re-validates all non-completed projects in a single batch with no size or timeout bound

- **Assumption:** AE-0312 states "a run-once migration bumps presentation_policy_version to v2 for every non-completed project" and "for each upgraded project the migration re-runs validate_localized_slides under v2 and stores the fresh severity-aware report."
- **Risk:** The set of non-completed projects is unbounded by the ticket. If dozens or hundreds of projects are in-flight (parked at various review interrupts), the migration re-validates each one, writing fresh reports. This is a potentially long-running migration with no batch-size, timeout, or retry specification. A migration that runs for minutes blocks the deploy (prod auto-deploys on merge to main). If it times out, the deploy fails or leaves a partial migration.
- **Impact:** Deploy stall or partial migration where some projects are v2-validated and others remain v1.
- **Suggested mitigation:** Specify the migration as batched (e.g., 50 projects per batch with a progress log), make it idempotent and resumable (the ticket says idempotent but not resumable-after-failure), and bound the maximum runtime. Alternatively, run the re-validation lazily (on next state read) rather than eagerly in the migration — bump the version and let a read-path reconciler validate on access.

### [WARN] AE-0313's change to `_verify_artifacts` output-dir resolution is a shared code path affecting first-time finalize — blast radius not fully bounded

- **Assumption:** AE-0313 fixes `_verify_artifacts` / `_pdf_check` to "validate the freshly rendered pre-promotion outputs (project root), never the previously active version root." The AC includes a "first-time-finalize regression" test.
- **Risk:** `_verify_artifacts` is called by both the republish path (new) and the existing images-approval finalize (shared). Changing the output-dir resolution for one path risks the other. If the first-time finalize passes `artifact_version=NULL` (no prior version), the resolution change is neutral. If a partial prior run left a stale `artifact_version` on a first-time-finalize project, the new resolution changes behavior in a way that's covered by the AC test but depends on getting the conditional right. The ticket says "must not regress the images-approval finalize" but this is a shared-function edit with two consumers — a single test doesn't prove every historical state shape is covered.
- **Impact:** A regression in first-time finalize would break the primary carousel completion path — higher traffic than republish.
- **Suggested mitigation:** Enumerate every call site of `_verify_artifacts` and `_safe_output_file` and add a test per call site × state shape (artifact_version set, NULL, stale). Audit the health-check path for any other consumer that relies on version-root resolution.

### [INFO] AE-0309 and AE-0310 consume the `blocking` flag but do not declare AE-0312 as a dependency — forward-compatible by design but the ordering contract is implicit

- **Assumption:** AE-0309 keys its fail-closed chain on `report.blocking`. AE-0310 keys its recovery UI on "blocking violations present." AE-0312 changes what `blocking` means (severity-derived). No ticket in the set declares AE-0312 as a dependency of AE-0309 or AE-0310.
- **Risk:** AE-0309/0310 ship first (all violations are `blocking=True` today — the severity model doesn't exist yet). AE-0312 ships later and refactors the blocking decision. This *is* forward-compatible: AE-0309 reads the flag generically. But AE-0312's cross-ticket invariant test ("a casing-only violation set does not trigger AE-0309's fail-closed chain") lives in AE-0312, not AE-0309. If AE-0312 is descoped or delayed, AE-0309's AC ("violations with `blocking=False` never consume the LLM retry") is untestable until then — it's dead code with no live exerciser.
- **Impact:** Low — the forward compatibility is sound. But the invariant test should be cross-referenced from AE-0309's QA checklist so it isn't lost if AE-0312 slips.
- **Suggested mitigation:** Add a note to AE-0309's QA checklist: "cross-ticket invariant (casing warnings don't trigger fail-closed) is asserted in AE-0312; if AE-0312 is deferred, add a mock-severity test to AE-0309 to exercise the `blocking=False` branch."

## Missing evidence

- **`bump_resume_lock_version` implementation:** Is it a single atomic `UPDATE ... WHERE lock_version = $expected` (true CAS) or a read-then-write (TOCTOU)? This determines whether the two-commit seam race is exploitable.
- **Whether AE-0311's repair sets any in-progress marker** that `ensure_resume_not_in_progress` checks during its execution. The ticket says the repair *returns* 409 when a run is in progress but does not say the repair *marks itself* in progress.
- **Current digest computation in `build_and_activate`:** Does the fingerprint hash rendered PDF/JPG bytes (embed timestamps) or slide/manifest data? AE-0313 calls this a "known gap to fix" but doesn't confirm the current behavior — the fix's scope depends on the answer.
- **Database isolation level** for carousel read/write transactions (READ COMMITTED vs REPEATABLE READ) — critical for the epoch-fence correctness analysis in AE-0315.
- **The mechanism by which AE-0315's flush guard distinguishes run-owned sessions from non-run sessions** (context variable? session flag? per-query annotation?). Without this, the "carry the current epoch by construction" claim cannot be verified.
- **Count of in-flight (non-completed) projects at any given time** — determines whether AE-0312's migration is a 2-second or 10-minute operation.
- **Connection pool configuration** (max lifetime, recycling on error) — determines whether AE-0316's session-scoped advisory lock leaks on process crash mid-request (the ticket says "connection drop releases it" but a pool-recycled connection is not dropped).
- **LangGraph checkpoint resumption semantics for mid-node state** after a `phase_status` flip — does the workflow engine resume from the last checkpoint boundary, and is that safe for a partial-generation state?
- **Whether `aupdate_state` on approved-hold threads preserves the pending interrupt** — AE-0314 defers this to a "pre-implementation proof" (task #1) but the entire post-completion edit feature depends on it. If the proof fails, the fallback (projection-only + drift reconciler) has its own unbounded convergence window.

## Residual risks if plan proceeds unchanged

- **Unbounded stale-checkpoint window after partial repair failure:** The autonomous drift reconciler is tick-based (eventual, not synchronous). A resume that proceeds before the reconciler converges re-runs the workflow against stale content. Re-validation (AE-0309) catches it safely but burns ~15 min of LLM spend and confuses the user.
- **Non-atomic edit→republish flow:** A browser close between the edit 200 and the republish call leaves corrected text in the DB/checkpoint with a stale PDF. The user sees the edit succeeded but the public artifact doesn't reflect it — a reskinned version of the 66014ba3 incident.
- **Epoch-fence false rejections or false admits:** Until the non-run write mechanism is specified and tested, the flush guard could either reject legitimate admin/user edits (degraded ops) or admit zombie writes (split-brain). The blast radius is the entire carousel write surface.
- **Cap-accounting regression on existing final_review→content path:** In-flight projects that accumulated content send-backs under the old divergent check are newly rejected with no user guidance beyond "use the edit path."
- **Migration runtime unbounded:** AE-0312's re-validation migration could stall a prod deploy if the in-flight project set is large.
- **Shared finalize health-check regression:** AE-0313's `_verify_artifacts` change touches a path consumed by the primary completion flow. A regression there is higher-impact than the republish feature it enables.
