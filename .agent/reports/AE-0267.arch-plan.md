# AE-0267 — Architecture Plan: user-creatable global palette catalog

Status: Proposed · Tier: T3 (epic) · Area: backend + frontend + DB · Mode: plan (read-only)
Author: architect-skill · Date: 2026-06-23
Supersedes: AE-0266 Phase 4 (optional "runtime endpoint + FE dynamic catalog")
ADR: `docs/decisions/0019-palettes-as-data.md` (required — see §ADR)

## Problem / goal

Today carousel palettes are a **closed set** defined in code (`PALETTE_REGISTRY`,
AE-0266). The user wants a **catalog** where the frontend can **add, show, and
edit** palettes — new palettes created at runtime, alongside the curated
originals. The originals stay as read-only "root designs"; new ones are
user-created and shared globally.

This inverts the source-of-truth model AE-0266 just shipped (code → data) for the
*custom* slice, while preserving it for the *root* slice. The plan reconciles the
two with a hybrid: **root = code/JSON (read-only), custom = DB (CRUD)**, unified
behind one resolver.

## Decisions (resolved with product owner, 2026-06-23)

| # | Decision | Rationale |
|---|----------|-----------|
| D1 | **Hybrid SoT.** Root palettes stay authored in the typed `PALETTE_REGISTRY` and are *emitted* to `docs/contracts/palettes.json` (served read-only). Custom palettes live in a DB table. | Keeps AE-0266's mypy-typed invariants (light-not-AUTO, light→light-style, brand-lock) for roots; adds runtime data only where editing is required. |
| D2 | **Catalog is global.** One shared `palettes` table, no per-user scoping. | Product owner call. Simplifies auth surface; audit columns still record creator. |
| D3 | **Image style is derived from mode**, never user-picked. light→`flat_editorial`, dark→default dark strategy. | Makes the AE-0264 light-on-dark class **structurally impossible** for custom palettes too. The Phase-2 `IMAGE_STRATEGY_REGISTRY` consolidation is what makes this a one-line lookup. |
| D4 | **Soft-delete.** A custom palette carries an `archived` flag; archiving removes it from the catalog for *new* carousels. Existing carousels are unaffected because they render from their **snapshot** (D9), not a live lookup. | Catalog hygiene without breaking past work. |
| D9 | **Snapshot the resolved palette onto the carousel at generation time.** `carousel_projects` gets a `theme_snapshot` JSONB (`{primary,accent,background,mode,resolved_ref,resolved_at}`); rendering/regeneration reads the snapshot. `project.theme` still stores the user's *choice* (root key \| `"auto"` \| custom UUID) for display. **Palette edits do NOT propagate to already-generated carousels** (reverses the original "propagate" intent). | Resolves the Pass-2 root cause (G1/G3/G5): past carousels are reproducible, a PATCH can't silently recolor them, AUTO is frozen at generation, and there is no mass-re-render fan-out. Product owner call (2026-06-23). |
| D5 | **Keywords required on custom palettes**, used for AUTO detection (not rendered into prompts). | Product owner call: custom palettes participate in topic auto-detection. |
| D6 | **`project.theme` becomes a string reference**: a root key (e.g. `"plasma_magenta"`), the `"auto"` sentinel, or a custom palette's **immutable UUID `id`** (NOT its slug). Replaces the `CarouselTheme` enum on the model + DB column. | Unavoidable spine: user palettes cannot be enum members. The UUID is the stable FK; the slug is display-only (see D8) so a project's resolved palette can never silently change. |
| D8 | **`slug` is human-readable/display only**, immutable on create, globally unique across ALL palettes incl. archived; it is never the value stored in `project.theme`. `PATCH` rejects slug changes. | Resolves cold-critic F1: a mutable/reusable string FK risks silent content mutation. |
| D7 | **Any authenticated user can edit/archive any custom palette** (writes are authed but not owner-restricted). `created_by` is still recorded for audit. | Product owner call (small trusted team). Accepted risk: a shared palette can be changed by anyone — mitigated by audit columns + soft-delete (no hard data loss). Revisit if the user base widens. |

Alternative to D1 (flip root SoT to a hand-edited JSON + Pydantic load-validator)
was considered and **rejected**: it trades away the compile-time invariants for
marginal authoring convenience; the registry already emits the JSON the API serves.

## The spine: theme enum → reference

`CarouselProject.theme: CarouselTheme` (domain model) is persisted as an enum
column (`infrastructure/database/models/carousel.py`) and referenced in ~9 source
files (tool, CRUD route, resolver, editorial module, domain + DB models). Custom
palettes cannot be enum members, so:

- `theme` becomes a **`str`** (a root key like `"plasma_magenta"`, a custom
  palette id, or the `"auto"` sentinel). The `CarouselTheme` enum is retained ONLY
  as the canonical list of **root keys** (for validation + the registry), not as
  the project field type.
- DB column migrates enum → `String` (or text). **Migration-sensitive** — prod has
  historically drifted from Alembic (memory: `prod-db-schema-drift`). Reversible,
  data-preserving migration required; existing enum values are already valid strings.
- Every read site that did `project.theme.value` now reads `project.theme`
  directly; every `CarouselTheme.X` comparison becomes a string/registry lookup.

A second, additive `carousel_projects` change accompanies it: a `theme_snapshot`
JSONB column (D9) holding the resolved `{primary,accent,background,mode,resolved_ref,
resolved_at}` frozen at generation. `theme` = the user's *choice*; `theme_snapshot`
= the *resolved result* the carousel actually renders from. (Lands with the resolver
change in P2, not P1.)

This is the only load-bearing change; everything else is additive.

## Architecture

### 1. Root palettes (unchanged authoring)
`PALETTE_REGISTRY` → `export_palettes.py` → `docs/contracts/palettes.json`. The
catalog API serves roots from the registry (read-only). No change to AE-0266
Phases 1–3 except the resolver refactor below.

### 2. Custom palettes (new persistence)
New table `palettes` (global):

| column | type | notes |
|--------|------|-------|
| id | uuid PK | the stable reference stored in `project.theme` (D6) |
| name | str | user-facing label; **partial-unique among active** (DB index) |
| slug | str | display/URL only, immutable, **globally unique incl. archived** (D8); never the FK |
| primary / accent / background | str(7) | **strict `#rrggbb` hex** (server-validated) |
| mode | enum(light/dark) | drives derived image_style + scrim |
| keywords | JSONB (list[str]) | sanitized; AUTO detection only |
| archived | bool, default false | soft-delete (D4) |
| created_by / created_at / updated_at | audit | |

`image_style` is **not stored** — it is derived from `mode` at resolve time
(D3), so it can never contradict the palette.

**DB-level constraints (resolves cold-critic F3 — app-level checks race):**
- partial unique index `(name) WHERE archived = false` — active-name uniqueness
  under concurrent `POST`;
- standard unique constraint on `slug` (all rows, incl. archived);
- the API maps the resulting `IntegrityError` to a clean `409 Conflict`.

New `PaletteRepository` port (domain) + SQLAlchemy adapter (infrastructure),
async, following the existing repository pattern.

### 3. Unified resolver (purity boundary shift)
`resolve_theme` is **pure** today (constants only). To auto-detect against custom
keywords + resolve custom ids, it becomes a small **application service** with a
`PaletteRepository` port:

- explicit root key → registry lookup (as today)
- explicit custom UUID `id` → DB lookup (active OR archived — existing projects
  keep resolving archived palettes; the slug is never used for resolution, D8)
- `"auto"` → brand-precedence over (root brand keywords ∪ custom keywords), then
  category/hash fallback over the root AUTO pool (custom palettes join the
  keyword-match pool but NOT the blind hash-rotation pool, to keep AUTO
  deterministic and root-curated)
- returns the color triple + mode + **mode-derived image_style**

Resolution moves from a pure function to a service+port — the one clean-arch
boundary this feature crosses. It runs **once per generation**; the result is
snapshotted onto the carousel (D9), so render/regeneration never re-resolves and
never re-hits the DB. Reliability hardening (cache + registry-only fallback +
degraded-mode AC) per Pass-2 G2. Documented in the ADR.

### 4. CRUD API
`/api/carousels/palettes` (or `/api/palettes`):
- `GET` — list catalog (roots + active custom), the "show" + create-form source.
- `POST` — create custom (Pydantic: hex validation, keyword sanitize, mode required).
- `PATCH /{id}` — edit custom (roots are 404/forbidden — not editable).
- `DELETE /{id}` — soft-delete (sets `archived`).

Pydantic request models validate: hex format, mode ∈ {light,dark}, name/slug
uniqueness, keyword count/length + sanitization.

### 5. Frontend
- Catalog view + create/edit form consume the `GET` endpoint **dynamically**
  (replacing the hardcoded `CAROUSEL_THEMES`/`THEME_LABEL_KEYS` lists).
- The create page theme dropdown renders the live union.
- **Visual/UX design of add/show/edit goes through the `impeccable` skill** (a
  parallel track) — this plan owns the data contract + flows, not the pixels.

## Security (first-class, not an afterthought)

- **Prompt-injection surface:** palette colors are interpolated into the LLM image
  prompt (`"Light background (#xxx) with #yyy ink…"`). User-supplied color fields
  MUST be strict server-side `#rrggbb`-validated (reject anything else) so no free
  text reaches the prompt. Keywords are used for matching only and are sanitized;
  if they ever reach a prompt, treat as untrusted.
- AuthN on all write endpoints; **no owner-restriction** (D7) — any authed user
  may add/edit/archive; `created_by` recorded for audit.
- Rate-limit creation to prevent catalog spam.

## Drift-gate evolution

The AE-0266 Phase-3 `check-palette-drift.mjs` gate guards the FE's *hardcoded*
theme lists against `palettes.json`. Once the FE renders the catalog dynamically,
those lists disappear and the gate **narrows** to what stays static: the image
presets and (optionally) the root keys. This feature **supersedes part of** what
shipped today — recorded in the ADR; the gate is retargeted, not deleted.

## Testing

Behavior-changing epic ⇒ **`.feature` files required** (CLAUDE.md AE-0153):
- palette CRUD (create/list/edit/soft-delete; root immutability; hex rejection)
- resolver: root key, custom UUID id, archived-still-resolves-by-id, AUTO over custom
  keywords, brand precedence preserved, mode-derived image_style (light never dark)
- **snapshot (D9):** a carousel renders from `theme_snapshot`; editing/archiving its
  palette afterwards leaves the carousel byte-identical; regeneration reuses the
  snapshot (no DB re-resolve); `theme="auto"` resolves once and freezes
- **resolver degraded mode (G2):** repo unavailable → registry-only fallback resolves
  root keys, logs + emits a metric, does not 500 the generation
- **concurrency:** two simultaneous same-name creates → exactly one succeeds, other
  gets `409` (partial-unique index, not app check); slug uniqueness incl. archived
- **slug immutability:** `PATCH` attempting to change slug is rejected
- migration: enum→string round-trips existing projects; post-deploy monitoring query
  passes (every theme value resolvable); staging downgrade trial documented
- security: non-hex color rejected; injection attempt in color/keyword neutralized
- FE: catalog renders union; create/edit happy + validation paths

## Rollout / rollback (phased, each increment shippable)

- **P1 — theme-as-reference (no behavior change).** Migrate `theme` enum→string;
  resolver still reads only the registry. Pure refactor + migration.
  **Migration is expand-only, not "reversible" (resolves cold-critic F2):** prod
  has drifted from Alembic, so we do NOT rely on `alembic downgrade` as the
  recovery path. Instead — (1) the enum's existing values are already valid
  strings, so the column converts in place (`USING theme::text`) with zero data
  rewrite; (2) **trial both upgrade and a downgrade on a staging DB restored from
  a recent prod backup** before merging; (3) primary recovery is **roll-forward**
  (a follow-up migration), not rollback; (4) ship a post-deploy **monitoring query**
  asserting every `theme` value is a known root key / valid UUID / `"auto"`, and
  row counts are unchanged. A tested contingency for a failed downgrade is an
  explicit P1 acceptance criterion.
- **P2 — custom palette persistence + resolver union + snapshot.** Table, repo,
  resolver reads DB (with cache + registry-only fallback, G2); add the
  `theme_snapshot` column and freeze the resolved palette at generation (D9);
  render/regeneration read the snapshot. No FE yet; API read-only.
- **P3 — CRUD API + security.** POST/PATCH/DELETE with validation.
- **P4 — FE dynamic catalog + create/edit** (impeccable-designed). Retarget the
  drift gate.

Each phase is independently deployable; P1 is the riskiest (prod migration) and
ships alone first.

## Skeptical review (cold critic) — resolved

External blind review by `gemini-2.5-pro` — verdict **PROCEED_WITH_CAUTION**, 3 WARN
findings, all accepted. Full text: `.agent/reports/AE-0267.skeptical-review.md`.

| Finding | Resolution |
|---------|------------|
| F1 — slug as fragile FK (silent content mutation) | **D6/D8:** `project.theme` stores the immutable UUID `id`; slug is display-only, immutable, globally unique, never the FK. |
| F2 — understated migration/rollback risk on drifty prod | **P1 hardened:** expand-only in-place convert (`USING theme::text`), staging trial from prod backup, roll-forward as primary recovery, post-deploy integrity monitoring query; tested downgrade-failure contingency is a P1 AC. |
| F3 — app-level uniqueness races | **DB constraints:** partial unique index `(name) WHERE archived=false`, unique `slug`; `IntegrityError`→`409`. |

Residual (carried into tickets): exact Alembic DDL, slug-generation rules, and the
keyword-sanitization spec are P1/P3 deliverables, not yet written.

## Skeptical review Pass 2 (GLM-5.2 via OpenCode) — triage

Second blind pass, different model family (`glm-5.2`/Zhipu). Verdict **WARN**, 6 WARN
+ 2 INFO. Full text: `.agent/reports/AE-0267.skeptical-review-glm.md`. It went
materially deeper than Pass 1. Triage:

**Resolved directly (engineering):**
- **G2 — resolver reliability regression.** `resolve_theme` going pure→DB adds a
  failure mode on the generation hot path. → P2 AC: in-process **LRU+TTL cache**
  (invalidate on PATCH/DELETE), documented **fallback to registry-only on repo
  error** (logged + metric'd), resolver-latency metric. "Degraded resolution when
  the palette repo is down" is an explicit P2 AC, not a footnote.
- **G4 — P1 "zero rewrite" is unverified on a drifty prod.** → add a **pre-P1 gate**:
  run a prod census (`SELECT theme, count(*) FROM carousel_projects GROUP BY 1`) +
  `pg_typeof(theme)` + actual column DDL/CHECK; assert every value is a current root
  key or `"auto"` BEFORE writing the migration. Confirm the staging backup is a
  current, unsanitized prod snapshot (else the downgrade trial is meaningless).
- **G6 — P3→P4 blind window.** A shipped write API with a static FE + still-green
  drift gate = false confidence. → **co-deploy P3+P4**, OR gate `POST` behind a
  feature flag until P4 lands. No interval where customs are writable but invisible.
- **G7 — D3 depends on Phase 2.** CONFIRMED RESOLVED: AE-0266 Phase 2
  (`IMAGE_STRATEGY_REGISTRY`) shipped to prod in PR #61 (merged 2026-06-23), so the
  mode→style derivation rests on live infra. Stated cost (accepted): custom palettes
  are visually homogeneous by style (all light→`flat_editorial`, all dark→default).
- **G8 — slug/name lifecycle under-specified.** → P3 Pydantic spec must define:
  un-archive policy (recommend **forbidden/terminal** to avoid the recreate→unarchive
  collision), slug charset (URL-safe; reject `/`, `..`, reserved routes), and **name
  XSS** handling (escape-on-render + length cap). Slug monotonic-consumption accepted.

**RESOLVED — G1 ∧ G3 ∧ G5 (shared root cause) via D9 snapshot-at-generation:**
product owner chose **snapshot** (2026-06-23). Each carousel freezes its resolved
palette at generation, so — (G1) `theme="auto"` is frozen → past carousels are
reproducible regardless of later catalog growth; (G3) a PATCH cannot recolor
already-generated carousels (they render from the snapshot) → "no hard data loss"
is now TRUE for generated work, and a `palette_revisions` audit table is **optional**
(catalog-forensics nice-to-have, not an integrity requirement); (G5) no propagation
⇒ no mass-re-render fan-out. **Residual (still required):** AUTO-capture of *future*
projects by broad keywords is inherent to custom keywords in the AUTO pool, so keep
keyword guards — reject overlap with root brand keywords, cap count + per-keyword
length, dedupe across the active catalog — plus creation rate-limiting (P3 spec).
