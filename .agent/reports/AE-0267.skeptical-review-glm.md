# Cold Critic Review (GLM 5.2 via OpenCode)

## Verdict
WARN

The plan is well-structured and the prior review caught the slug-FK, migration-rollback, and create-race classes of risk. But three of its load-bearing claims are weaker than presented, and a first reviewer missed a cluster of issues around the **AUTO keyword path**, the **purity→I/O reliability regression**, and the **id-FK integrity gap** (the actual chosen mechanism, not the rejected slug-FK). These are unresolved and material.

## Findings

### [WARN] AUTO is NOT deterministic once custom keywords enter the match pool — silent content mutation via the `theme="auto"` path
- Assumption: §3 states custom palettes join the keyword-match pool but are excluded from hash-rotation "to keep AUTO deterministic and root-curated."
- Risk: Hash-rotation is deterministic, but **keyword-match is not**. The keyword pool is `(root brand keywords ∪ custom keywords)` and custom palettes are added/edited/archived at runtime by any authed user (D5, D7). A project with `theme="auto"` that resolved to palette X today can resolve to a brand-new custom palette Y tomorrow purely because someone created Y with overlapping keywords — no project edit, no migration, no signal to the owner. This is the *same class of silent content-mutation risk* F1 addressed for explicit references, but the AUTO path is now exposed to it and F1 did not cover AUTO.
- Impact: Existing auto-themed carousels silently shift visual identity over time as the catalog grows; reproducibility of a previously-generated carousel breaks; the "deterministic" claim in §3 is half-true and misleads reviewers/operators.
- Suggested mitigation: Either (a) snapshot the resolved palette id onto the project at generation time so future regenerations are stable (explicit reference thereafter), or (b) scope custom-palette AUTO participation to projects created *after* the custom palette existed, or (c) drop the "deterministic" claim and explicitly accept AUTO drift in the ADR with an operator-visible diff. Option (a) is the cleanest and also fixes finding #3 below.
- Open question for author: Is a regenerated carousel expected to produce the same palette as its first generation? If yes, AUTO-over-mutable-pool contradicts that. If no, say so in the ADR.

### [WARN] `resolve_theme` purity loss is an unmitigated reliability regression — no cache, no fallback, no SLO
- Assumption: ADR-0019 acknowledges the resolver "crosses the previously-pure resolver's I/O boundary — an accepted, explicit consequence," and the plan treats this as a documentation matter.
- Risk: A pure function over constants **cannot fail**; an application service over a DB port **can**. `resolve_theme` is on the hot path of every carousel generation. The packet specifies no caching (palette rows are stable-ish and tiny — ideal for an in-process LRU + TTL), no fallback behavior when the DB is slow/unreachable (return a safe default? fail the request?), and no latency/SLO target. Combined with the DigitalOcean single-droplet deploy and prod's drift history, a DB hiccup now breaks carousel generation that previously could not break.
- Impact: New production failure mode for a previously-infallible path; potential DB read amplification (one lookup per generate); no defined degraded-mode behavior.
- Suggested mitigation: Specify a cache layer (keyed by id/root-key, invalidated on PATCH/DELETE via event or short TTL), a documented fallback (e.g., registry-only resolution on DB error, logged + metric'd), and a resolver-latency metric. Make "degraded resolution when palette repo is unavailable" an explicit P2 acceptance criterion, not just a consequence.
- Open question for author: What is the intended behavior of carousel generation when the `PaletteRepository` is unavailable?

### [WARN] No referential integrity is possible for the UUID id-FK, and "no hard data loss" overstates the audit story
- Assumption: D6 stores the immutable UUID `id` in `project.theme`; D4 says soft-delete preserves resolution and "edits to a live palette still propagate"; the security section asserts "no hard data loss."
- Risk: `project.theme` is a string over a **mixed reference domain** (root key | `"auto"` | custom UUID), so it **cannot** carry a SQL FK to `palettes.id`. Referential integrity for the custom slice is therefore **convention-only**: the API soft-deletes, but nothing structurally prevents a hard-delete (manual ops intervention, a future migration, a buggy script) from orphaning every project referencing that id. Separately, "no hard data loss" is true for *row existence* but **false for value history**: edits to a live palette overwrite `primary/accent/background` with no edit log, so a malicious or careless PATCH (any authed user, D7) to a popular palette propagates new colors to all referencing projects and the previous colors are **gone, unrecoverable**. Audit columns (`updated_at`, `created_by`) record *that* a change happened, not *what* the colors were.
- Impact: A single bad-faith or fat-fingered edit to a shared palette silently recolors an unbounded set of live carousels with no revert path; a hard-delete orphans projects with a resolver that has no defined fallback (see previous finding).
- Suggested mitigation: (a) Operationally forbid hard-delete and state "no DB FK is possible by design — integrity is API-enforced" explicitly in the ADR; add a lifecycle guard that refuses `DELETE` on a row whose id appears in any `project.theme` even via ops tooling. (b) Add an append-only `palette_revisions`/audit table capturing the pre-image color triple on every PATCH so edits are revertible — this is what "no hard data loss" actually requires. (c) Re-snapshot semantics (finding #1) limit blast radius.
- Open question for author: Is edit history of custom palette color values required for revert/forensics? If not, retract the "no hard data loss" claim.

### [WARN] P1 "zero data rewrite / all enum values are already valid strings" is an assertion without a prod value census
- Assumption: P1 frames the enum→string migration as `USING theme::text`, "zero data rewrite," "existing enum values are already valid strings," guarded by a post-deploy monitoring query.
- Risk: The prod DB has a **known drift history** (create_all bootstrap, hand-added columns — per project context). The plan assumes the `theme` column is a clean PG enum whose labels are exactly the current `CarouselTheme` members. If prod ever held a removed/renamed/lowercase/typo theme value, or if the column is already a varchar with a CHECK (not a true enum), then (i) `USING theme::text` may behave differently than assumed, (ii) a CHECK constraint may need expanding, and (iii) a dangling value survives the in-place convert and the monitoring query **fails** — at which point P1 is no longer "no behavior change / zero rewrite," it requires an unplanned data-fix sub-step that isn't designed. The monitoring query catches the problem, but the plan's framing pretends the problem cannot exist.
- Impact: P1 deploy stalls mid-window on a drifted prod; the "tested downgrade-failure contingency" AC is met on staging but staging (from a backup) may still not reproduce a drift scenario that the backup itself encodes — need to confirm the backup is *current prod*, not a sanitized one.
- Suggested mitigation: Before P1 design completes, run and attach a **prod value census** (`SELECT theme, count(*) FROM carousel_projects GROUP BY 1`) plus the column's actual DDL/type. Make "every present value is a current root key or `'auto'`" a **pre-P1 gate**, not a post-deploy surprise. Specify the column's actual type (enum vs varchar+CHECK) in the migration DDL that is currently "not yet written."
- Open question for author: What does `pg_typeof(theme)` return on prod today, and what distinct values are present?

### [WARN] Any authed user can hijack AUTO and amplify cost via required keywords + edit-propagation (D7 × D5 × D4)
- Assumption: D7 (no owner-restriction) + D5 (keywords required, feed AUTO) + D4 (edits propagate) are each individually accepted; the plan does not analyze their **composition**.
- Risk: Composition: (1) Any authed user creates a custom palette with broad/generic keywords (e.g., `["image","photo","carousel","the"]`) — those keywords enter the AUTO match pool (§3) and can **capture non-brand AUTO routing** away from curated roots for *all* future auto projects (finding #1). There is no described guard against keyword overlap with root brand keywords, nor against duplicates across custom palettes. (2) The same user PATCHes a popular custom palette's colors — D4 propagates to all referencing projects. If propagation triggers asset re-render, a single edit spawns an unbounded number of LLM image generations (cost + OpenAI/Gemini rate limits); if propagation does NOT re-render, D4's "edits still propagate" is false. The rate-limit in §Security is on **creation only**, not edit/archive, so PATCH spam is unthrottled.
- Impact: AUTO-capture abuse; cost amplification (any authed user → mass regen bill) OR silent inconsistency between propagated config and rendered assets; no throttling on the mutating verbs that matter most post-creation.
- Suggested mitigation: Keyword validation rules (residual, unwritten) must include: reject keywords overlapping root brand keywords; cap keyword count and per-keyword length; dedupe across the active catalog or warn. Define edit-propagation semantics explicitly (does PATCH invalidate rendered assets? yes/no — pick one and cost it). Extend rate-limiting to PATCH/DELETE, per-palette and per-user.
- Open question for author: Does editing a live palette re-render already-generated carousel assets? If yes, what caps the fan-out cost? If no, retract "edits still propagate."

### [WARN] Phase ordering: P3 (write API) before P4 (dynamic FE) opens a window where customs exist but the catalog/gate are blind to them
- Assumption: P2 = read-only API incl. custom (none exist yet); P3 = CRUD; P4 = FE dynamic catalog + drift-gate retarget. The AE-0266 `check-palette-drift.mjs` gate asserts FE hardcoded lists match `palettes.json` (roots only).
- Risk: Once P3 ships, users can POST custom palettes that **exist in the DB and resolve via the API**, but the FE still renders the hardcoded root list and the drift gate still asserts FE == `palettes.json` (roots). The gate stays **green while the FE is functionally stale** — false confidence precisely during the P3→P4 gap. If P4 slips, prod runs with a write API whose outputs are invisible to users and unguarded by the gate.
- Impact: Invisible-to-user custom palettes that still affect AUTO resolution (finding #1) — users can't see or manage the things changing their carousels; the drift gate, the project's integrity backstop, is green-but-wrong.
- Suggested mitigation: Either collapse P3+P4 into one deploy (write API and dynamic FE ship together), or in P3 retarget the gate to assert "FE consumes the API dynamically, not a static list" (which requires the P4 work), or block POST in P3 behind a feature flag until P4 lands. Do not leave a shipped write API with no user-facing surface and a still-static gate.
- Open question for author: Is there an acceptable interval where custom palettes are writable but not user-visible? If not, P3 and P4 must be co-deployed.

### [INFO] D3's "one-line lookup" depends on ADR-0018 Phase 2 (`IMAGE_STRATEGY_REGISTRY`), whose shipped status is not confirmed in the packet
- Assumption: §3 / ADR-0019: "the Phase-2 `IMAGE_STRATEGY_REGISTRY` consolidation is what makes this a one-line lookup" and "pays off — deriving style from mode is a single lookup."
- Risk: ADR-0018 lists Phase 2 as a **follow-up ("phased, non-breaking")**, not confirmed delivered. If Phase 2 is not yet in prod, D3's derivation mechanism rests on unbuilt infrastructure and the "structurally impossible" light-on-dark guarantee for customs is only as strong as work the packet doesn't prove exists.
- Impact: If Phase 2 is unshipped, P2 needs to either land Phase 2 first or define the derivation against the *current* two-map structure (re-introducing the drift surface ADR-0018 was built to kill).
- Suggested mitigation: Confirm Phase 2 status in the plan; if unshipped, make it an explicit P2 predecessor. Also note a quality trade-off the plan doesn't state: all light custom palettes are forced to `flat_editorial` and all dark customs to the single default dark strategy — visual homogeneity among customs is a deliberate cost of the safety guarantee; say so.

### [INFO] Soft-delete lifecycle and slug namespace are under-specified
- Assumption: D4 soft-delete sets `archived`; D8 slug immutable + globally unique incl. archived; DELETE = soft-delete.
- Risk: (a) Is un-archive supported? If yes: concurrent archive-A → recreate-A (allowed by the partial-unique `WHERE archived=false` index) → un-archive-A collides with the new active A and the un-archive fails with `IntegrityError`; the UX for that is undefined. If un-archive is terminal, the plan should say so. (b) Because archived slugs are never reclaimed and slugs are immutable, the slug namespace is **monotonically consumed** — over years, desirable slugs are permanently blocked by archived rows, forcing ugly generated slugs. (c) Slug generation rules are explicitly unwritten (residual); since slug is "URL only" (D8), an unconstrained charset could collide with FE routes (`..`, `/`, reserved paths). (d) The `name` field is user-facing display but neither XSS-sanitization nor a charset cap is specified (only color hex and keyword sanitize are mentioned).
- Impact: Long-tail slug exhaustion; possible route/path-safety issues; display XSS surface on name; undefined un-archive semantics.
- Suggested mitigation: Specify un-archive policy (allowed/forbidden), slug charset (URL-safe, reject `/`, `..`, reserved words) and generation rules, and name sanitization (escape on render, length cap) in the P3 Pydantic spec that is currently deferred.

## Missing evidence
- **Prod `theme` column census and DDL** — actual `pg_typeof`, distinct values + counts, and any CHECK constraint. Required before P1 DDL can be written. (Finding #4)
- **ADR-0018 Phase 2 shipped status** — is `IMAGE_STRATEGY_REGISTRY` live in prod? Required to validate D3's "one-line lookup." (Finding #7)
- **Edit-propagation semantics** — does PATCH on a custom palette re-render referencing projects' assets? Required to resolve the cost-vs-consistency tension and to validate D4's "propagate" claim. (Finding #5)
- **Prod backup provenance for the P1 staging trial** — is it a current, unredacted prod snapshot (to reproduce drift), or sanitized? Determines whether the "tested downgrade-failure contingency" AC is meaningful. (Finding #4)
- **Keyword-overlap/dedup rules** — none specified; needed to assess AUTO-capture abuse. (Finding #5)
- **i18n decision (O3)** — open; blocks the P4 FE label contract. (Plan §Open questions)

## Residual risks if plan proceeds unchanged
- AUTO-themed projects silently change palette as the custom keyword pool grows (Finding #1) — contradicts the plan's "deterministic" wording; no snapshotting.
- Carousel generation gains a failure mode (DB dependency on the resolve hot path) with no cache, fallback, or SLO defined (Finding #2).
- A single PATCH by any authed user can irreversibly recolor all projects referencing a popular custom palette, with no color-value history to revert from (Finding #3); "no hard data loss" is inaccurate as written.
- A hard-delete of a custom palette (not structurally prevented — no FK possible) orphans referencing projects with no defined resolver fallback (Findings #2, #3).
- P1 may stall on a drifted prod value/type not covered by the "zero rewrite" framing, because no pre-migration census is required (Finding #4).
- Between P3 and P4, writable custom palettes are invisible to users and to the (still-static, green) drift gate, while still affecting AUTO (Findings #5, #6).
- All custom palettes are visually homogeneous in image style by design (D3) — acceptable, but unstated as a cost; and only safe if Phase 2 actually shipped (Finding #7).
