# Cold Critic Review

## Verdict
WARN

## Findings

### [BLOCKER] AE-0300: mTLS origin-pull + L4 Cloudflare-range firewall compound a single misconfiguration lockout with a weak rollback claim
- Assumption: The rollback script "drops the firewall and disables origin pulls in one command" can always be executed.
- Risk: If a Cloudflare range refresh removes the IP the GitHub Actions runner egresses-through (or the admin's own IP changes), AND origin-pull mTLS is on, the operator may be unable to reach the box except via SSH 22 — which is the very path AE-0305 keeps on port 22 and which the firewall restricts to "admin IP(s)". A stale admin-IP allow rule + rotated Cloudflare ranges = no app path AND no fix path if SSH is also mis-scoped. The "canary health path" AC partially mitigates the Cloudflare-range refresh case but does not address SSH/firewall drift for the admin IP.
- Impact: Total production lockout requiring DO console access (out-of-band), with no documented path in the packet to the DO web console as the break-glass.
- Suggested mitigation: Explicitly define the break-glass path (DO console / recovery console / IPMI-equivalent) in the runbook, and make the SSH allow-list a separate object from the Cloudflare-range allow-list so range refresh cannot touch SSH. Add an AC that the rollback script is exercised against a simulated Cloudflare-range rotation, not just a "dry run" against live state.
- Open question for author: What is the documented out-of-band access path if both 443 and 22 are mis-scoped, and is the SSH admin-IP allow-list refreshed on the same cadence as the Cloudflare ranges?

### [BLOCKER] AE-0301: Ticket defers the actual key-risk decision (rotation) by making it an AC checkbox with no criteria
- Assumption: "A rotation decision is recorded in the Decision Log (rotate now vs accept the risk, with rationale)" is sufficient closure.
- Risk: The scan confirmed 644 on files holding `OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, `POSTGRES_PASSWORD`, `SECRET_KEY`, etc., for an indeterminate time window. "Recorded decision to accept" is indistinguishable from silently deferring — the AC allows accepting the risk with no threshold, no exposure-window estimate, and no rotation timeline. This is the highest-impact finding in the packet (live, billable, identity-bearing keys were world-readable) and the ticket treats remediation of the *files* as the resolution while the *keys themselves* remain unrotated.
- Impact: A key compromised during the 644 window remains live indefinitely; the "hardening" is cosmetic against an attacker who already exfiltrated.
- Suggested mitigation: Default rotation IN unless an explicit sign-off with a stated exposure-window analysis says otherwise; rotation should be a tracked follow-up ticket with a deadline, not a free-form decision-log line. At minimum, name which keys are rotation-cheap (API keys) vs expensive (POSTGRES_PASSWORD, SECRET_KEY with restart implications) and split them.
- Open question for author: What is the exposure-window estimate for the 644 state (uptime of the files vs first known 644 scan), and why is rotation not the default for the cheap-to-rotate API keys?

### [WARN] AE-0300: "Authenticated Origin Pulls" answer diverges from the stated goal and does not replace L7 CF-IP firewalling
- Assumption: Requiring the Cloudflare origin-pull client cert guarantees only Cloudflare can reach the app.
- Risk: The CF origin-pull CA is a publicly distributed cert used by all Cloudflare customers' origin-pull configurations; any party that can terminate that mTLS handshake (e.g., an attacker on the L4-allowed CF IP range who can present a copy of the same public origin-pull cert) passes mTLS. The real isolation is the L4 CF-IP allow-list; the mTLS adds defense-in-depth but is not a privacy boundary. The packet presents them as co-equal gates. Also, if the L4 firewall is the DO cloud firewall, note CF publishes ranges that change — a stale fetch silently shrinks the allow-list.
- Impact: Over-confidence in mTLS as the gate; operators may relax monitoring of range refresh believing mTLS covers it.
- Suggested mitigation: Document explicitly that L4 CF-IP filtering is the primary control and mTLS is defense-in-depth (not a second independent trust boundary given the shared public CA). Add monitoring for CF range staleness (last-refresh timestamp alert).
- Open question for author: Is there a scheduled refresh + alert when CF ranges drift, or is refresh only on-deploy?

### [WARN] AE-0302: Empty-password-URL-fragment check is easy to bypass and the "single shared construction path" is asserted not proven
- Assumption: "All Redis consumers use one shared construction path" can be verified by audit before merge.
- Risk: The packet says "First audit and enumerate all construction sites and route them through the one path" — meaning the enforcement does not yet exist; a future consumer can bypass it, and the empty-fragment heuristic (`redis://:@host`) misses `redis://host?password=` variants, env-injected `REDISCLI_AUTH` without `requirepass`, or a raw `redis.Redis(host=...)` call that never touches the URL builder. The fail-fast is production-gated on ENVIRONMENT, but `ENVIRONMENT` is itself a deploy secret that could be mis-set.
- Impact: A new Redis client added next quarter silently connects unauthenticated in prod, and the "shared path" invariant is unenforced by the type system.
- Suggested mitigation: Make the shared path a hard architectural constraint (a single factory function with no public constructor for Redis clients) plus a lint/test that greps for `redis.Redis(` / `from redis` direct usage outside the factory. Treat `ENVIRONMENT` mis-set as a separate risk (it gates the DB too — is it load-bearing here?).
- Open question for author: Is `ENVIRONMENT` already the load-bearing prod flag for other secrets, and what prevents a drift where it's unset in prod (which would make Redis unauth-OK, the wrong direction)?

### [WARN] AE-0303: "justify Automatic-Reboot-Time by traffic data" assumes traffic data exists and is consulted
- Assumption: Low-traffic data is available and the chosen window is derived from it.
- Risk: The packet does not state where traffic data comes from (Cloudflare analytics? nginx logs? Langfuse?). If none is set up, the AC becomes unfalsifiable and the time defaults to a guess — the exact failure the revision claimed to fix. Separately, auto-reboot on a box where AE-0300 has locked down origin access and AE-0301 must re-apply 600 after each `.env` rewrite: does a reboot+unattended-upgrade sequence re-trigger the deploy `.env` rewrite path, or could an auto-reboot strand the box in a state where the deploy re-write hasn't run and... irrelevant here, but does auto-reboot run before containers auto-restart via compose `restart:` policies? That ordering is asserted healthy post-reboot but not for the auto-reboot case.
- Impact: Auto-reboot may fire during real traffic; reboot-failure alerting is "added" but not specified (where does the post-reboot smoke alert go — email? webhook? Cloudflare?).
- Suggested mitigation: Name the traffic-data source as an explicit AC input; specify the alert destination for post-reboot smoke failure; confirm `restart: unless-stopped` (or equivalent) is present on ALL prod services so the auto-reboot case mirrors the manual-reboot verification.
- Open question for author: What is the alert sink for an auto-reboot smoke failure, and does every prod compose service have a restart policy that the auto-reboot path depends on?

### [WARN] AE-0304: "Soak day" verification is not enforceable and `docker rm -f` of minio-test risks a hidden mount losing data
- Assumption: Capture of `docker inspect` before removal is sufficient reversibility.
- Risk: `docker inspect` reveals mounts and env but not the *data inside the container* (volumes are external, but a `minio-test` may use an anonymous volume not surfaced in the named-volumes list). If an operator later discovers a hidden dependency, "recreate from inspect" reproduces the config but not the stored objects. The "soak day" AC for AE-0303 sequencing is stated but not gated — nothing stops both tickets landing the same day.
- Impact: Irreversible data loss if minio-test held something; compounding with AE-0303 reboot if both merge without the soak gap.
- Suggested mitigation: Before `rm -f`, also `docker inspect` anonymous/named volumes and snapshot anything non-empty; make the soak-day gap a merge-time gate (the orchestrator/board must block AE-0303 from Dev-Complete until AE-0304 has 24h green), not a prose sequencing note.
- Open question for author: Were minio-test volumes inspected for content, and is the 24-hour gap enforced anywhere besides the ticket text?

### [WARN] AE-0305: fail2ban on a single admin-IP-allowed, key-only SSH is near-zero-value and may ban the operator
- Assumption: fail2ban default sshd jail is pure upside.
- Risk: With SSH already key-only and (post-AE-0300) restricted to admin IP(s), the brute-force threat fail2ban addresses is largely already closed; the residual risk is fail2ban banning the admin's own IP after a flaky client retries, locking out the one access path — compounding AE-0300's SSH allow-list fragility. The FN (false-negative) cost is already near zero given key-only; the FP (false-positive) cost is a lockout.
- Impact: Non-trivial chance of self-DoS for marginal security gain; no "whitelist admin IP / use recidive-only after N fails from non-allowlisted" tailoring mentioned.
- Suggested mitigation: Either `ignoreip` the admin IP range, or scope fail2ban to a stricter threshold and longer findtime; document the unban procedure (`fail2ban-client unban`) in the same runbook as the SSH break-glass.
- Open question for author: Is the admin IP static enough for `ignoreip` to be safe, and is the unban procedure documented alongside the lockout break-glass from AE-0300?

### [INFO] Cross-ticket: Sequencing dependency AE-0302 → AE-0300 is stated in AE-0300 but AE-0302 has no blocker AC reflecting it
- Assumption: Cross-ticket sequencing notes are tracked by the orchestrator.
- Risk: Sequencing constraints scattered across tickets ("AE-0302 before 0300", "AE-0304 before 0303") are not machine-enforced unless the board tooling reads them. If AE-0300 lands first, a Redis auth misconfig strands the backend behind a locked origin with no `curl --resolve` sanity path — the exact scenario AE-0300's own notes warn of.
- Impact: The most dangerous dep (own ticket text acknowledges it) has no gate on the depended-on side.
- Suggested mitigation: Add a `Blocked-by: AE-0302` field on AE-0300 (not a prose "Related" note), or add an AC on AE-0302 that it must be Deployed before AE-0300 starts.
- Open question for author: Does the orchestrator/`move_ticket.py` enforce cross-ticket `Blocked-by` for these sequencing constraints, or are they advisory?

## Missing evidence
- Exposure-window duration for the 644 secret state (AE-0301) — needed to judge rotate-vs-accept.
- Traffic-data source for the `Automatic-Reboot-Time` choice (AE-0303).
- The DO console / out-of-band break-glass procedure, referenced nowhere despite three tickets (0300, 0305) playing with the only admin path.
- Whether `ENVIRONMENT` (AE-0302 fail-fast gate) is reliably set in prod and absent in dev/CI today — load-bearing for safety.
- Restart policies per prod service — captured "pre-reboot" in AE-0303 but not stated as a precondition; needed to trust auto-reboot.
- Whether the GC anti-gaming / `gates.sh` integrity check (per CLAUDE.md AE-0259) was run on the infra-script tickets — `.feature` files exist but `.feature` for infra (firewall, sshd) is largely untestable in-repo; how is "Dev Complete" gate captured for these?

## Residual risks if plan proceeds unchanged
- Production lockout with no documented break-glass if CF-range refresh or admin-IP drift mis-scopes both 443 and 22 (AE-0300).
- Compromised-during-644-window keys remain live indefinitely; "decision recorded" can equal "silently deferred" (AE-0301).
- Over-trust in mTLS origin-pull as an independent trust boundary when it shares Cloudflare's public origin-pull CA (AE-0300).
- A future Redis client bypasses the "single shared path" since it's an audit invariant, not a type/code constraint (AE-0302).
- Auto-reboot may fire without reliable restart policies/alerting on all prod services (AE-0303).
- fail2ban self-DoS of the only admin path compounding AE-0300's SSH allow-list fragility (AE-0305).
- Unenforced cross-ticket sequencing can strand the backend behind a locked origin (AE-0300 ↔ AE-0302).
