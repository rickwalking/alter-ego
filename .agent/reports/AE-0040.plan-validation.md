## 1. Blocking Gaps

These must be resolved before tickets move to **Ready**:

### 1.1 Missing Rollback Plan for Blanket Ignore Removal (Épico 1)

**Problem:** Removing blanket ignores (Tema 1.1) will break CI immediately. If ~47+ files have violations, the team could be blocked for days.

**Required addition:**
- Define a **feature flag / incremental approach**: e.g., remove blanket ignores per sub-package, not globally.
- OR: Define a **commit-by-commit sequence** where each commit removes one rule from the blanket ignore and fixes all violations in `src/rag_backend/` for that rule.
- MUST include: `git revert` rollback procedure if CI remains red > 4 hours.

### 1.2 In-Flight Project Migration (Épico 2 + 3)

**Problem:** Refactoring `presentation_review.py`, `editorial_distribution_persist.py`, `manifest_from_payload`, and `validate_localized_slides` changes the internal API of modules that are actively used by the editorial workflow. The PR #11 itself is still open — projects in production could be mid-workflow.

**Required addition:**
- Define backward compatibility window: old function signatures kept as deprecated wrappers with `warnings.warn(DeprecationWarning)` for 1 sprint.
- OR: Refactor in-place but verify no behavior change via **contract tests** (input → output equivalence before/after refactor).

### 1.3 Mutation Test Targets Not Specified

**Problem:** ADR-005 requires mutation testing for all business logic. The plan mentions mutation score targets but doesn't specify which new code paths need mutation-killing tests.

**Required addition:**
- For each Strategy/Builder/Validator extracted: must have mutation-killing tests achieving 80%+ mutation score before the old code is deleted.
- New `_VALIDATORS` dispatch table in Tema 2.3: must kill mutants for unknown content_kind, missing required fields, forbidden fields present.

## 3. Warnings (Non-Blocking)

### 3.1 Épico 4 (Frontend) Scope Creep Risk

The frontend refactoring (Temas 4.1–4.4) is well-scoped but depends on the backend Épicos 2–3 being stable. If backend refactoring changes API response shapes (e.g., `build_workflow_state_response` output), the frontend types/interfaces may need to change too.

**Mitigation:** Do frontend refactoring AFTER backend Épico 2 is merged and API contracts are stable.

### 3.2 Strategy Pattern Overengineering Risk

Tema 2.2 proposes 5 separate Builder classes for `_build_locale_presentation`. This is a repair script (`repair_workflow_malformed_drafts.py`), not production code. Overengineering a script that may be deleted after the malformed drafts are fixed is wasted effort.

**Recommendation:** Consider a lighter approach — use a dictionary dispatch table instead of full Strategy pattern classes. Only promote to Strategy if the script becomes permanent:

```python
_BUILDERS: dict[str, Callable[[Mapping[str, object], str | None, int], dict[str, object]]] = {
    "intro": _build_intro,
    "summary": _build_summary,
    # ...
}
```

### 3.3 Missing: `assert_carousel_public_or_editor` Legacy Code

ADR-007 §5 mandates removing `assert_carousel_public_or_editor` from public routes. The PRD does not address this. While not blocking for AE-0040, it should be tracked.

**Recommendation:** Add a non-goal: "Legacy pipeline removal (per ADR-007) is out of scope for AE-0040 but tracked separately."

### 3.4 Missing: High_risk_areas Tag

The CLAUDE.md defines a `high_risk_areas` concept. The plan should tag:
- Tema 1.1 (blanket ignore removal) → HIGH RISK (CI breakage)
- Tema 2.1 (Builder for response builder) → MEDIUM RISK (API contract change)
- Tema 2.3 (Validation refactor) → MEDIUM RISK (behavior change)

### 3.5 Missing: Observability Impact

The plan does not mention Langfuse traces for refactored code. If `validate_localized_slides` or `build_presentation_review_updates` are refactored, their Langfuse trace metadata must be preserved.

## 4. Sequencing Concerns

### 4.1 Dependency Graph is Correct but Risky

```
Épico 1 (CI/ignores) ──depends on── Épico 2 + 3 ──depends on── analysis (done)
Épico 4 (frontend) ──independent── of 1, 2, 3
```

**Risk:** Épico 1 should NOT be last. Blanket ignores actively harm the project by hiding violations. Recommended reorder:

| Priority | Epic | Rationale |
|----------|------|-----------|
| P0 | Épico 3 (Code Smells) | Low risk, high impact, immediate improvement |
| P1 | Épico 2 (Architecture) | Medium risk, unblocks Épico 1 |
| P2 | Épico 1 (Ignores) | Cannot do until 2+3 are resolved |
| P3 | Épico 4 (Frontend) | Independent, can be parallel |

**Recommended order:** Semana 1: Épico 3 + Épico 4 (parallel) → Semana 2: Épico 2 → Semana 3: Épico 1

### 4.2 Within Épico 2, Builder Before Strategy

Tema 2.1 (Builder) should precede Tema 2.2 (Strategy) because:
- Builder pattern affects API response — needs more testing
- Strategy pattern is for a repair script — lower priority
- Tema 2.3 (Validation refactor) is independent of both
