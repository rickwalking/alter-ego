# Architecture Plan: Image Phase Resilience & Design-phase Review

**Ticket:** AE-0010 (proposed)
**Status:** Plan
**ADR required:** No — extends existing patterns, no architectural change

---

## Problem

Two related issues in the design/images phase workflow:

### Issue 1: Missing `PHASE_STATUS_FAILED` guard in `design_phase_async` and `images_phase_async`

In `carousel_workflow_nodes.py`, `content_phase_async` properly guards against artifact failure:
```python
async def content_phase_async(state, config):
    merged = dict(state)
    merged.update(await ensure_artifacts(state, config, PHASE_CONTENT))
    if merged.get("phase_status") == PHASE_STATUS_FAILED:   # ← guard exists
        return merged
    sync_result = content_phase(...)
    return {**merged, **sync_result}
```

But both `design_phase_async` and `images_phase_async` are **missing this guard**. When `_ensure_design_artifacts` throws (e.g., `run_design()` failure), the exception propagates through LangGraph into a blanket `except Exception:` in the background resume runner, marking the project as failed with no user feedback. Reference: `docs/guides/workflow-resume-failure-analysis.md`.

Additionally, `_ensure_design_artifacts` in `phase_artifact_runner.py` calls `apply_design_tokens()` without a try/except — any exception escapes to the caller.

### Issue 2: No meaningful content to review before approving the design phase

When the workflow reaches `current_phase: "design"`, `phase_status: "awaiting_human"`, the user sees an "images" tab with:
- A "Design applied — Ready" badge
- Outline/draft counts
- **No meaningful content to review before approving**

The approve payload contains only `{"design_applied": true}` — a bare boolean.

Meanwhile, the backend already generates **image prompts** per slide during slide creation (`SlideData.image_prompt`, persisted in `CarouselSlide.image_prompt` DB column). These prompts are:
- Used downstream by the images phase to call DALL-E/Stable Diffusion
- **Never exposed to the frontend**
- The user cannot review or correct them before image generation runs

---

## Proposed Solution

### Overview

Two tracks:
1. **Resilience**: Add `PHASE_STATUS_FAILED` guards to `design_phase_async` and `images_phase_async`; wrap `apply_design_tokens` in try/except in `_ensure_design_artifacts`
2. **UX**: Expose each slide's `image_prompt` in the workflow state response during the design phase. Display prompts in the images tab as reviewable text.

### Backend Changes

#### 1. Add `PHASE_STATUS_FAILED` guard to `design_phase_async`

**File:** `backend/src/rag_backend/agents/carousel_workflow_nodes.py`

```python
async def design_phase_async(state, config):
    merged = dict(state)
    merged.update(await ensure_artifacts(state, config, PHASE_DESIGN))
    if merged.get("phase_status") == PHASE_STATUS_FAILED:
        return merged
    sync_result = design_phase(cast(CarouselWorkflowState, merged), config)
    return {**merged, **sync_result}
```

#### 2. Add `PHASE_STATUS_FAILED` guard to `images_phase_async`

Same pattern:
```python
async def images_phase_async(state, config):
    merged = dict(state)
    merged.update(await ensure_artifacts(state, config, PHASE_IMAGES))
    if merged.get("phase_status") == PHASE_STATUS_FAILED:
        return merged
    sync_result = images_phase(cast(CarouselWorkflowState, merged), config)
    return {**merged, **sync_result}
```

#### 3. Wrap `apply_design_tokens` in try/except in `_ensure_design_artifacts`

**File:** `backend/src/rag_backend/application/services/carousel/phase_artifact_runner.py`

```python
async def _ensure_design_artifacts(self, state, pending):
    if self._db is None:
        return {}
    outline = pending.get("outline") or state.get("outline") or []
    if not isinstance(outline, list):
        return {}
    slides = await ensure_slides_from_outline(...)
    if not slides:
        return {}
    try:
        await apply_design_tokens(self._db, str(state.get("project_id", "")), slides)
    except Exception:
        return {
            "phase_status": PHASE_STATUS_FAILED,
            WORKFLOW_ERROR_KEY: "Design token application failed",
        }
    return {"design_applied": True}
```

#### 4. Extend `EditorialWorkflowStateResponse` schema with image prompts

**File:** `backend/src/rag_backend/api/schemas/carousel_workflow.py`

Add:
```python
class SlideImagePrompt(BaseModel):
    slide_index: int
    title: str
    image_prompt: str

class EditorialWorkflowStateResponse(BaseModel):
    ...
    slide_image_prompts: list[SlideImagePrompt] | None = None
```

#### 5. Build `slide_image_prompts` in state response builder

**File:** `backend/src/rag_backend/api/routes/carousels/editorial_workflow_routes_support.py`

In `build_workflow_state_response()`:
- Accept an optional `slides: list[SlideData] | None` parameter (fetched by the route handler)
- When `current_phase in ("design", "images")` and slides are available, extract `(slide_index, title, image_prompt)` for each slide with a non-empty `image_prompt`
- Populate `slide_image_prompts` in the response
- Graceful degradation: if slides param is None, set `slide_image_prompts = None`

**In the route handler** (`GET /carousels/{id}/workflow/state`): fetch `CarouselSlide` records via `PostgresCarouselRepository` and pass to the builder.

#### 6. No workflow state change needed

Image prompts are already persisted in the DB (`CarouselSlide.image_prompt`). The workflow state TypedDict (`CarouselWorkflowState`) does NOT need a new field — prompts are fetched on-the-fly.

## Data Flow

### Design phase resilience
```
ensure_artifacts → _ensure_design_artifacts
  → ensure_slides_from_outline + apply_design_tokens
  → try/except catches failure → phase_status = failed, WORKFLOW_ERROR_KEY set
  → design_phase_async guard detects → returns merged with failed status
  → human gate skipped → workflow error state persisted
```

### Prompt review
```
DB: CarouselSlide.image_prompt
  ↓ (route handler fetches slides, passes to response builder)
build_workflow_state_response(slides=slides) → slide_image_prompts[]
  ↓ (API JSON → frontend)
EditorialWorkflowState.slide_image_prompts
  ↓ (React render)
ImagePromptReview component → editable textareas (v2: submit edits)
```

### V1 Scope (this ticket)
- Add guards to prevent unhandled exceptions in design/images phases
- Error handling in design tokens application
- Show prompts for visual inspection in the images tab
- Approval continues with existing prompts (edits not submitted back in v1)

### V2 Future
- Submit edited prompts back to backend
- Persist to `CarouselSlide.image_prompt`
- Use edited prompts during image generation

## Acceptance Criteria (EARS format)

### Resilience
- AC-1: `design_phase_async` returns early with `phase_status == "failed"` when `ensure_artifacts` sets failed status (no sync phase call)
- AC-2: `images_phase_async` returns early with `phase_status == "failed"` when `ensure_artifacts` sets failed status (no sync phase call)
- AC-3: `_ensure_design_artifacts` catches exceptions from `apply_design_tokens` and returns `phase_status: "failed"` with `WORKFLOW_ERROR_KEY`
- AC-4: On design token failure, `workflow_error` is set to `"Design token application failed"`
- AC-5: Existing `content_phase_async` guard is unchanged (regression guard)

### Prompt review
- AC-6: `EditorialWorkflowStateResponse` includes `slide_image_prompts: list[SlideImagePrompt] | None`
- AC-7: `slide_image_prompts` is populated when `current_phase in ("design", "images")`
- AC-8: `slide_image_prompts` is null for all other phases
- AC-9: Each `SlideImagePrompt` matches `CarouselSlide` DB record (slide_index, title, image_prompt)
- AC-10: Slides with empty/null `image_prompt` are skipped
- AC-11: Response builder degrades gracefully when slides param is None (sets field to None)

### Frontend
- AC-12: `EditorialWorkflowState` type includes `slide_image_prompts?: SlideImagePrompt[]`
- AC-13: `ImagePromptReview` component renders N cards for N prompts with index, title, prompt text
- AC-14: `ImagePromptReview` textareas are disabled when `readOnly` is true
- AC-15: Artifacts summary shows "N image prompts ready" count when prompts exist
- AC-16: Images tab renders `ImagePromptReview` between artifact summary and live controls
- AC-17: All i18n keys exist in both `en.json` and `pt.json`

## Risks and Mitigations

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| Prompt data is stale (slides edited after prompts regen) | Low | `ensure_slides_from_outline` is idempotent — same outline produces same prompts |
| Large number of slides → many prompts | Low | Max ~10 slides, card list is fine |
| User expects to edit prompts in v1 | Medium | V1 read-only; document clearly for v2 follow-up |
| Response builder DB query fails | Low | Graceful degradation: `slide_image_prompts = None`, response succeeds |
| Regression: content phase guard breaks | Low | AC-5 ensures explicit regression test |
