# Cold Critic Review

## Verdict
WARN

## Findings

### [WARN] AE-0300 monotonic-growth guard can permanently accumulate stale Cloudflare ranges
- **Assumption:** Cloudflare ranges only grow; removals are rare and diff-approved.
- **Risk:** The guard "refuses to apply a fetch smaller than last-known-good" treats set-shrinkage as always-suspicious. Cloudflare *does* occasionally retire ranges (and publishes v1/v2 endpoints that differ). A legitimate shrink + an operator who diff-approves removal still has *no documented procedure* for how the monotonic state advances past an approved removal — i.e. how "last-known-good" gets re-based after an approved shrink. If you simply re-run the additive apply, it will *re-add* the approved-removed range (additive = never removes), so the allow-list silently drifts back to the LKG superset regardless of approval.
- **Impact:** The firewall never truly reflects CF's published set; only grows. Over months this compounding superset widens the L4 surface (more CF egress IPs than necessary) and, worse, presents a *false sense* that removals are controlled when the control is unenforceable against an additive apply.
- **Impact category:** defect-in-plan (the procedure for an approved removal is unspecified, which makes the "removals require diff-approval" AC unimplementable as written).
- **Suggested mitigation:** Define the LKG re-basing step explicitly: a diff-approved removal is committed as a new LKG snapshot that the additive apply *compares against* (not the stale superset). The "monotonic" guard must guard against *unvalidated* shrinkage, not *all* shrinkage. State the artifact that records a new LKG.
- **Open question for author:** After an approved range removal, what is the exact sequence that advances LKG so the next additive apply doesn't re-add it? Is LKG a versioned file or an in-memory set? Where does the additive step read its "current applied" set from?

### [WARN] AE-0300 / AE-0307 / main-push auto-deploy create a live-lock window during rotation/reboot
- **Assumption:** The AE-0303 lock blocks deploy, rotation, and CF-refresh simultaneously, so they can't collide.
- **Risk:** `main`-push auto-deploy is *not* mentioned as a lock-holder anywhere. AE-0306 lists "a coincident `main` auto-deploy" as something the lock should prevent, and AE-0303 says the lock defers "deploy" — but the auto-deploy is triggered by a `git push to main`, which is *outside any lock-holding entity's control*. There's no described mechanism by which a `main` push (a) acquires the lock, (b) *waits* for the lock if held, or (c) is *rejected/fails closed*. A push mid-rotation either ignores the lock entirely (so the stated invariant is false) or blocks waiting (undefined — does the GitHub Action hang? timeout? fail the deploy?).
- **Impact:** Either the lock invariant is unenforced against the most common mutator (auto-deploy on `main`), or the deploy behavior when the lock is held is unspecified and could manifest as a mysteriously-hung CI run that an operator force-cancels, racing the rotation. This is the exact class of "compound failure" AE-0303 is supposed to prevent.
- **Impact category:** defect-in-plan (the lock is claimed to cover deploys, but the only fully-automated deploy path has no defined lock interaction).
- **Suggested mitigation:** Specify the `main`-push deploy's lock behavior as an AC: does it acquire-wait, acquire-or-fail, or is `main` push itself frozen (branch protection / deploy freeze) during rotation/reboot windows? The "coincident auto-deploy" risk is only closed if the *triggering* path is controlled, not just the consumer.
- **Open question for author:** While AE-0303/AE-0306 hold the lock, what happens to a `git push origin main`? Does the deploy wait, fail, or is the push blocked upstream? Who enforces this — the lock, branch protection, or a deploy-freeze bot?

### [WARN] AE-0302 `rename-command CONFIG` blocks the very runtime probe that's supposed to verify auth
- **Assumption:** Build-time config-presence check + runtime NOAUTH probe are "two non-contradictory" independent checks, and disabling `CONFIG` is fine because the build-time check covers integrity.
- **Assumption risk:** The runtime probe "an unauthed command returns NOAUTH" uses `PING`/`SET`, fine. But the *broader* auth-integrity story ("`CONFIG GET requirepass` returns a non-empty value", listed as a *separate* AC) is now *unverifiable at runtime* because `CONFIG` is disabled. So two ACs are in tension: AC requires `CONFIG GET requirepass` return non-empty; the `rename-command CONFIG` AC makes that command fail. The ticket even notes this contradiction in passing ("since disabling `CONFIG` makes `CONFIG GET requirepass` itself unavailable") but leaves *both* ACs in place.
- **Impact:** A reviewer running the AC checklist hits a genuine failure on `CONFIG GET requirepass` (it returns an error, not a non-empty value) — and either the AC must be relaxed (un-documenting the assurance) or the `rename-command` must carve out `CONFIG GET` (re-opening the `CONFIG SET requirepass ""` path the lockdown is meant to close). This is not "implementation detail"; it's a contradictory acceptance gate.
- **Impact category:** defect-in-plan (two ACs are mutually exclusive as written; the "non-contradictory" framing in scope doesn't resolve the AC-level contradiction).
- **Suggested mitigation:** Pick one: (i) drop the `CONFIG GET requirepass` AC and rely solely on the build-time config-presence + runtime NOAUTH pair (the honest integrity story); or (ii) use `CONFIG GET requirepass` with a *narrow* rename that permits `GET requirepass` but disables `SET` (Redis supports per-subcommand renaming via `rename-command CONFIG ""` + a custom — but this is fragile and undocumented for subcommands; better to drop the AC). Document which AC is the load-bearing one.
- **Open question for author:** Which AC is authoritative when `CONFIG` is disabled — the `CONFIG GET requirepass` AC or the `rename-command` AC? They cannot both pass.

### [WARN] AE-0306 session-key rotation impact is asserted schedulable but no user-facing comms path is defined
- **Assumption:** `SECRET_KEY`/`ANON_SECRET_KEY` rotation "invalidates all existing sessions/tokens" and the impact is "scheduled/communicated."
- **Assumption risk:** The packet states only that the impact was "scheduled/communicated and verified (login + anon flows work post-rotation)." But there's no AC for *who* is communicated to, *how* (the site seems to be a single-operator/public blog per AE-0297 context — but the packet doesn't state the user population), or what the *re-auth UX* is. If anon sessions silently drop, a mid-carousel WebSocket/SSE user sees a broken stream with no explanation. The Gherkin "users can re-authenticate" is unbounded — how many users, is there a login page, is there an anon-session migration path (e.g. re-homing in-flight carousels之上)?
- **Impact:** The "low blast radius" framing may be wrong if there's any active anon-session usage at rotation time; in-flight carousel checkpoints keyed to the old `SECRET_KEY` could become unverifiable, stranding a workflow mid-generation (cross-link to AE-0302's checkpoint consumer audit — which lists *app services* but not *in-flight carousel state signed by `SECRET_KEY`*).
- **Impact category:** incomplete-because (the consumer/impact enumeration for session-key rotation omits in-flight signed state).
- **Suggested mitigation:** Add an AC: enumerate in-flight state signed/validated by `SECRET_KEY`/`ANON_SECRET_KEY` (carousel checkpoints, WebSocket auth tokens, anon session cookies) and define the migration — either drain first (no in-flight carousels during rotation) or a dual-key validation window. "Login + anon flows work post-rotation" doesn't cover the *transition* of in-flight work.
- **Open question for author:** Are carousel checkpoints or WebSocket sessions HMAC-bound to `SECRET_KEY`? If yes, what happens to a carousel mid-generation at rotation? Is there a drain step, or is in-flight work accept-loss?

### [INFO] AE-0307 self-hosted-runner-on-droplet option conflates the ingress with the thing it protects
- **Assumption:** A self-hosted runner "on the droplet's private network" is a listed option alongside a separate static-IP runner.
- **Risk (low):** A runner *on the production droplet itself* makes the deploy trust boundary == the production trust boundary. A compromised runner (the thing AE-0307 explicitly hardens against elsewhere) is then *on* the target — the "ingress down → rebuild" recovery and "compromise handling" sections assume an external node. The plan lists this as an option without flagging that it inverts the isolation model the whole ticket is building.
- **Impact:** If the operator picks the on-droplet runner for expediency, AE-0307's own "ingress is a privileged asset that must be as hardened as the droplet" AC becomes vacuous (it *is* the droplet). The pinning in AE-0300 also degrades: 22 is pinned to the droplet's own IP.
- **Impact category:** incomplete-because (the option matrix doesn't distinguish "self-hosted runner *off*-box static IP" from "self-hosted runner *on*-box").
- **Suggested mitigation:** Either drop the on-droplet-runner option, or mark it explicitly as "degrades AE-0300 isolation to nil; only acceptable if the droplet has a private-network/IP allow-list path that doesn't traverse public 22 — state which." The recommendation should be off-box.
- **Open question for author:** Is the on-droplet runner option intended to use `localhost`/private-network SSH (so 22 doesn't need to be open at all) or public 22 from the droplet to itself? The two have opposite security postures and the ticket treats them as one.

## Missing evidence
- **Cloudflare range history:** Has CF ever *removed* a published range (not just added)? The monotonic-guard design assumes additions dominate; one confirmed removal would force the re-basing procedure (see Finding 1). No CF change-log/changelog is cited.
- **`main`-push deploy frequency + branch-protection posture:** Is there a deploy-freeze mechanism (branch protection "rule set", a PAUSE bot, GitHub Environment required-reviewer) that can hold `main` during a rotation? The lock story depends on this; none is referenced.
- **Langfuse version + key-rotation feature matrix:** AE-0306 correctly gates on "does the installed version support re-encryption" but provides no version string. Whether re-encryption is even *possible* is the single highest-impact unknown in the packet; the plan should carry the version now (it's on the box).
- **Population of active sessions/anon-users at any time:** Drives Finding 4. The packet never states whether the site has 0, 10, or 10k concurrent anon sessions that `SECRET_KEY` rotation would drop.
- **Self-hosted runner vs. Tailscale cost/ops delta on a 4GB droplet already at 79% disk:** AE-0304 is reclaiming disk; a new long-running runner process (if on-box) trades disk/CPU for network isolation. No resource envelope is given for the chosen ingress.
- **The 644 exposure-window "worst-case lower bound" is an input to AE-0306's accept-loss decisions but is *produced* by AE-0301 — a hard cross-ticket data dependency with no schema.** If AE-0301 records "at least 3 days, possibly since droplet creation," AE-0306's Langfuse accept-loss call changes meaning dramatically. No format/units are specified for the handoff.

## Residual risks if plan proceeds unchanged
- The CF allow-list silently only grows; a future CF range retirement is either silently ignored (additive re-adds) or requires an undocumented re-basing dance. Long-term the "restricted to CF ranges" claim weakens to "restricted to a historical superset of CF ranges."
- A `main`-push during a rotation/reboot window either races the lock (deploy proceeds, invariant false) or hangs the CI run with no defined operator behavior — the very compound failure the lock exists to prevent.
- AE-0302's QA cannot pass both `CONFIG GET requirepass` (non-empty) and `rename-command CONFIG` (disabled) simultaneously; one AC must be silently dropped at verification time, which is exactly the kind of gate-relaxation CLAUDE.md warns against.
- Session-key rotation may strand in-flight carousel/WebSocket state bound to the old `SECRET_KEY` if a carousel is generating at rotation time; the "drain first" step is absent.
- An on-droplet self-hosted runner, if chosen, makes AE-0300's 22-pin and AE-0307's "ingress hardened separately" both vacuous — the plan permits the option but doesn't bound its consequences.
