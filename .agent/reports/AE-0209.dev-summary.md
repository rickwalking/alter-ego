## Developer Completion Report
Ticket: AE-0209 — Per-slide partial-commit and idempotent re-entry for the images phase
Status: Dev Complete

### Acceptance Criteria Implemented
- [x] Partial image successes are persisted to workflow state on a mid-batch failure.
  `run_images` now gathers with `return_exceptions=True`, so one slide's exception
  no longer cancels its siblings. Each `_run_one_image` commits its own slide
  (image_path + metadata + generation record) on success, so successful images
  persist even when another slide fails. After the batch, `_raise_on_batch_failures`
  re-raises an aggregate `RuntimeError` (chaining the first underlying error) so the
  phase failure is still surfaced for the resume runner to mark the workflow failed.
- [x] Re-running the images phase regenerates only missing slides and completes.
  Re-entry was already idempotent via `prompt_hash`/generation_key: `reuse_recorded_generation`
  (record lookup) and `_reuse_existing_image` (on-disk by generation_key) short-circuit
  generation for already-generated slides. With per-slide commit, a retry regenerates
  only the still-missing slides. Verified by a seeded re-run test.
- [x] Verified the background-resume failure AND cancellation paths release the lock.
  `_mark_background_resume_failed` (unchanged, sets `phase_status='failed'`) already
  releases on failure. ADDED: a dedicated `except asyncio.CancelledError` handler in
  `_execute_background_resume` — `CancelledError` is a `BaseException` so the generic
  `except Exception` never caught it; a cancelled/killed background task could leave
  the workflow holding the `in_progress` lock. The handler rolls back, marks failed
  (recoverable, releasing the lock), then re-raises to honor cooperative cancellation.
- [x] Seeded single-slide-failure test passes.
