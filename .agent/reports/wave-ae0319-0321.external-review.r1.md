## Blind review — AE-0319 / AE-0320 / AE-0321

### Per-ticket verdict on AC coverage

- **AE-0319**: ACs ~75% covered. `Retry-After` scope item dropped; the 503 path has no log assertion.
- **AE-0320**: All four lock-safety ACs are nominally implemented, but the reaper's `rollback()` semantics break the "no wedged worker tick" guarantee at the *transaction* level (prior successful flips in the same tick are rolled back). Root-cause scope (commit-before-awaits, short-lived sessions) NOT implemented — only symptom-bounded with a 2s timeout.
- **AE-0321**: Clean. Markers are reasonable; tests cover hit/deep-marker/normal. Minor false-positive risk on generic phrases.
- All three feature files exist; gherkin scenarios ↔ tests line up.

### BLOCKER / MAJOR / MINOR / NIT

**BLOCKER**

1. `carousel_run_reaper.py:~181` — On `OperationalError` the code calls `await db.rollback()`, which aborts the *outer* worker-tick transaction, not just the failed flip. Any earlier successful `_reap` CAS flips performed in the same `reconcile()` invocation on the same shared `db` session are silently rolled back. The AC ("blocked reaper flip fails fast … retries next tick") holds for *that* row, but the side-effect is data loss for *other* rows in the same tick. Fix: wrap the flip in a nested transaction (`async with db.begin_nested()`) or use a short-lived session per row.

**MAJOR**

2. `editorial_workflow.py:~243` — The 429 path raises `HTTPException(429, detail=…)` without a `Retry-After` header even though `openai.RateLimitError.response.headers` commonly carries one. AE-0319 scope explicitly requires "*429 with `Retry-After` when available*". Scope item missed.
3. `test_carousel_workflow_safety_net.py::test_start_provider_outage_maps_to_503` — Does not `capture_logs` and does not assert `workflow_start_provider_error`. AC #3 ("both paths log …") is unverified for the 503 path; this test cannot fail on a regression that drops the log.
4. `provider_errors.py:10-11` — Hard top-level `import anthropic` / `import openai` in a module that is imported unconditionally by the workflow-start route. If either SDK is not a runtime dependency today (the project uses LangChain; anthropic SDK presence is unverified here), or a future SDK release renames the symbols, the entire editorial workflow route fails to import — total availability regression. Guard with `try/except ImportError` or `TYPE_CHECKING` + lazy classification.

**MINOR**

5. `editorial_workflow_resume_runner.py:159` — The diff only swaps `_beat_and_stage` to `write_run_heartbeat_once`. The AE-0320 scope items ("commit before long-running engine work", "short-lived sessions per write batch") are **not** implemented; the self-deadlock root cause (runner awaits a separate-session write inline while its own uncommitted tx holds the row lock) is only *bounded* to 2 s, not removed. A stage beat now blocks the runner for up to `RUN_WRITE_LOCK_TIMEOUT_MS` before soft-failing — every gate approval pays a 2 s ptol.
6. `carousel_drift_reconciler.py:~111` — The convergence UPDATE has `WHERE phase_status == FAILED` but no `run_epoch` predicate, unlike the reaper's seen-epoch CAS. CAS prevents concurrent reconciler wins on status, but the row is bumped unconditionally; there is no "this run owns it" check, so a transient FAILED state (e.g. mid-transition) cannot be discriminated from a genuinely dead one.
7. `carousel_drift_reconciler.py` — The new raw-UPDATE site does **not** call `apply_run_write_lock_timeout`. The reconciler targets FAILED rows (dead), so normally uncontended, but if a long-running transaction ever locks a FAILED row (reaper mid-flip on a stale epoch, manual admin UPDATE, vacuum, etc.), the reconciler queue-blocks forever — the exact behavior AE-0320 explicitly forbids for the other run-column writers.
8. `test_carousel_run_reaper.py::test_blocked_flip_fails_fast_and_logs` — Sets `bind.dialect.name = "sqlite"`, so `apply_run_write_lock_timeout` is a no-op; the OperationalError is synthetic and the test cannot exercise the *real* Postgres `SET LOCAL lock_timeout` → driver-level `OperationalError` → catch/rollback path. The companion test only asserts the SET LOCAL string was emitted, not consumed. Coverage gap on the actual fail-fast flow.
9. `research_enrichment.py::_is_block_page` — Markers include generic phrases ("access denied", "you have been blocked") with only a 600-char head scan. A legitimate article whose preamble contains "Access denied: …" (legal/security domain content) will be silently discarded. Consider tighter markers (Cloudflare ray-id + title combination, or marker repetition).

**NIT**

10. `carousel_run.py` — `LOG_EVENT_PHASE_DRIFT_CONVERGED = "carousel_phase_drift_converged"` is inconsistent with the pre-existing `LOG_EVENT_DRIFT_CONVERGED = "carousel_drift_converged"` nose; two related events now read as siblings under different stems.
11. `pg_lock_timeout.py` — `_SET_LOCK_TIMEOUT_SQL` is built at import-time by string concatenation of `str(int(RUN_WRITE_LOCK_TIMEOUT_MS))`. Trivially safe (the constant is an int), but a `bindparam`-parameterized `text("SET LOCAL lock_timeout = :ms")` with `ms=f"{...}ms"` would future-proof against accidental mutation of the constant to non-int.
12. `carousel_run_progress.py::write_run_heartbeat_once` — The catch is `except Exception`, wider than it needs to be (the AC root is `OperationalError` from the SET/UPDATE path). Catching `Exception` here also swallows `asyncio.CancelledError` subclasses under Python <3.8 behavior — on 3.8+ `CancelledError` is `BaseException`, so fine, but a narrower `(OperationalError, DBAPIError)` would document intent.

---

### Verdict: **FAIL**

The AE-0320 reaper rollback semantics (#1) are a genuine correctness regression: a single lock-timeout in the middle of a worker tick silently undoes the reaps that succeeded earlier in the same tick — strictly worse than the prior "blocks until manual `pg_terminate_backend`" behavior in terms of *outcome consistency* (now you get *partial* ticks with no warning). Combined with the dropped `Retry-After` (#2), the unverified 503 log assertion (#3), the hard `anthropic`/`openai` import (#4), and the un-bounded reconciler (#7), the package is not mergeable as-is.
