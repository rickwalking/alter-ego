# Cold Critic Review

## Verdict
WARN

## Findings

### [WARN] AE-0300 + AE-0302 ordering creates an availability catch-22
- Assumption: Each ticket is independently deployable; the deployment workflow `deploy.yml` rewrites the server `.env` from GitHub Secrets and reaches the droplet over SSH 22 (not 443).
- Risk: AE-0300 restricts 80/443 to Cloudflare ranges *and* enables mTLS origin pulls via nginx. AE-0302 adds `REDIS_PASSWORD` to the GitHub Secrets → `.env` pipeline. If AE-0300 lands first, a misconfigured Cloudflare-range refresh or cert rotation can lock out legitimate traffic to the *only* admin path that is validated by the app serving 200 — and there is no documented check that the deploy runner, healthchecks, or rollback path survive a partial state where the firewall is on but origin-pull is off (or vice versa). Combined with AE-0302, a deploy that fails to write `REDIS_PASSWORD` brings the backend down *behind* the new firewall, making the failure mode harder to diagnose and slower to roll back (you can't just `curl --resolve` to sanity-check anymore).
- Impact: Extended outage of the public site / blog API / WebSocket streaming with a degraded diagnostic path; the very WAF bypass the plan cites (AE-0297 rate limit) becomes untestable in-place.
- Suggested mitigation: Define the rollback as an *executable, idempotent* script (not a runbook paragraph) that drops the firewall + disables origin-pulls in one command, and stage AE-0300 changes with a canary window where direct-origin is still reachable via a temporary second health endpoint. Sequence AE-0302 *before or together with* AE-0300 so a Redis auth failure cannot strand you behind a locked-down origin.
- Open question for author: What is the exact ordered rollout, and is the rollback script actually tested (not just documented) against a live droplet? Can you verify the deploy runner's SSH path is unaffected if Cloudflare ranges change mid-deploy?

### [WARN] AE-0301 chmod-in-deploy does not close the world-readable window
- Assumption: Adding `chmod 600` after the write step in `deploy.yml` guarantees 600 on every deploy.
- Risk: `deploy.yml` writes `.env` via a heredoc/`cat`/`tee`-style step. Between the file creation (open with default umask 022 → 644) and the subsequent `chmod`, the file is momentarily world-readable on the filesystem. On a multi-tenant concern this is negligible, but the *stated threat model* is "any future non-root user / container breakout / app file-read bug." A container breakout that reads the file in that window still wins, and the plan claims the hardening "cannot silently regress" — which is only true if no other process (cron, ad-hoc script, manual edit) ever touches the file. The plan does not address the at-rest `backend/.env` *source* in GitHub Secrets or the `.env.backup.*` creation path — it removes the backups but does not identify what keeps creating them (the "memory" note says deploy rewrites `.env`, but backups imply a separate manual or cron step).
- Impact: Residual plaintext-secret exposure window per deploy; backups may recur if the creation source is unidentified; "cannot regress" claim is overstated.
- Suggested mitigation: Set `umask 077` at the top of the deploy step (and in the server-side deploy script) so files are created 600 from the start, not chmod'd after; add a post-deploy `stat` assertion that *fails the deploy* if any `.env` is not 600; identify and stop whatever creates `.env.backup.*` (or document it does not recur); consider rotating the secrets given the documented 644 window included live API keys + DB password + Langfuse secrets — the plan explicitly punts, but "no evidence of exposure" is weak given world-readable + 64-day uptime + unauth Redis.
- Open question for author: What creates `.env.backup.*`, and is there a scheduled or manual job that will recreate it? Why is rotation deferred when the scan confirms a 644 window on live secrets, not just hypothetical exposure?

### [WARN] AE-0302 missing-credential path is under-specified and asymmetrical
- Assumption: "Fail fast if missing in production" — the backend raises a clear config error rather than connecting unauthenticated.
- Risk: The plan adds `REDIS_PASSWORD` to deploy `.env` but does not specify the precedence vs the existing `REDIS_URL`/`redis://` connection strings. If any client builds URLs by string concatenation or uses a cached settings object, a missing password can silently fall back to an unauthenticated connection (depending on how the URL is parsed — an empty `: @host` fragment is often treated as no-auth), contradicting the acceptance criterion. Worse, fail-fast in *production* is specified, but the behavior in dev/test (where the secret may intentionally be absent) is unspecified — the Gherkin "missing credential fails fast" scenario has no environment guard, so it may break local dev or CI which have no Redis password.
- Impact: Either false green (silent unauth fallback) or broken dev/CI; healthcheck reconnect-after-auth-failure behavior untested.
- Suggested mitigation: Make the fail-fast explicit: settings validator raises `ConfigError` only when `APP_ENV=production` and `REDIS_PASSWORD` is unset *or* when the resolved Redis URL lacks credentials in prod; document dev/test behavior (unauth OK) in the ticket; add a unit test that asserts a URL with an empty-password fragment is treated as missing-cred in prod; ensure all Redis clients (publisher, consumer, cache, checkpoint) share one construction path so auth is not wired per-client.
- Open question for author: How many distinct Redis client construction sites exist, and does the plan audit all of them (Streams, cache, checkpoint, healthcheck)? What is the dev/test behavior when the secret is absent?

### [WARN] AE-0303 reboots a box with a known unhealthy frontend container and no verified restore path
- Assumption: Reboot will bring all containers back healthy (or the unhealthy `frontend-1` is pre-existing and unrelated).
- Risk: The plan depends on `docker-compose.prod.yml` `restart:` policy auto-restarting containers, but does not verify the restart policies are set, nor whether the unhealthy frontend container will come back at all (it may be unhealthy *because* of something the reboot changes — e.g., a stale mount, a race with nginx, a certbot renewal window). The plan's own note flags the frontend `unhealthy` state but defers it. Rebooting into a state where the frontend is worse (or the maintenance window is half-open) leaves the site down with no rollback for a *reboot* (you can't un-reboot). Also, `unattended-upgrades` auto-reboot is enabled *after* the manual reboot, with no canary — if the config is wrong, the next auto-reboot could land during traffic hours.
- Impact: Extended or unmaintained-window outage; frontend healthcheck root cause deferred indefinitely; auto-reboot misconfigured timing → off-hours prod blip.
- Suggested mitigation: Before reboot, capture `docker inspect` restart policies + the frontend healthcheck definition and *resolve* the unhealthy state (or explicitly document it as out-of-scope with a follow-up ticket referenced, not "track separately if it persists" — that is not a commitment). Validate `Automatic-Reboot-Time` against actual low-traffic window from analytics, and add a post-auto-reboot smoke that pages on failure.
- Open question for author: What are the restart policies in `docker-compose.prod.yml`, and has the frontend `unhealthy` cause been root-caused before the reboot? What is the chosen `Automatic-Reboot-Time` and what traffic data supports it?

### [WARN] AE-0303/AE-0304 sequence: no rollback if cleanup removes an undeclared dependency
- Assumption: `minio-test` and `langfuse-worker-test` are "genuinely orphaned" because they are not in `docker-compose.prod.yml`.
- Risk: Absence from the compose file is necessary but not sufficient. A prior ad-hoc runbook step, a manual operator procedure, or an external cron may depend on `minio-test` (e.g., as a scratch S3 target for a Langfuse export job, or a test data sink). The plan's verification step ("not referenced by any compose/staging file") only checks compose files, not shell history, cron, systemd timers, or operator runbooks. Removing them and then rebooting (AE-0303) is a compound change with no rollback (containers `rm -f` is not reversible without re-running them, which requires the original env/image).
- Impact: Silent breakage of an undocumented operational dependency; difficult to restore because the container env/image may not be recorded.
- Suggested mitigation: Before removal, grep shell history / cron / systemd timers / runbooks for the container names and the `minio-test` endpoint; capture `docker inspect` of both containers (image, env, mount, network, command) into a runbook artifact so they can be recreated if needed; sequence AE-0304 *before* AE-0303 and verify one full day of production operation after cleanup before rebooting.
- Open question for author: What created these test containers on 2026-06-12 (operator action? a script?), and is there a runbook or cron that references them by name?

### [WARN] AE-0305 CSP change lacks a regression net and gates on undefined environment detection
- Assumption: The `http://localhost:8000` entry can be "gated to non-production only, or removed."
- Risk: CSP changes are high-blast-radius and silent in tests — a too-strict `img-src` breaks images *in the browser only* (no server-side signal), and the plan's verification is "images still load (no CSP regression in the browser console)" with no automated assertion. The ticket is Tier: T1 / Priority: Low, but a CSP regression blocks the blog/dashboard visually. Additionally, "locate where the CSP is set — frontend `next.config`/middleware or nginx" is an unknown at plan time: if the CSP is set in *both* places (nginx adds headers, Next.js also sets them), removing one may be a no-op and the localhost entry persists; the acceptance criterion (curl `-I` shows no localhost) would then fail with no clear single fix. Environment-gating "non-production" requires a defined env signal (`NODE_ENV`, `APP_ENV`, hostname check) — none is specified, risking a dev-only entry leaking to prod again or a prod build breaking local image loading.
- Impact: Visual breakage shipped to production with no automated guard; ambiguous fix location leads to a flaky acceptance check.
- Suggested mitigation: Add an automated CSP assertion (frontend headers test or a Playwright browse + console-error check) that asserts the prod CSP across key routes *before* this ticket closes; first `grep` both nginx and Next.js/middleware for `localhost:8000` and `img-src` to locate *both* sources in the plan, not as an implementation discovery; specify the exact env signal used for gating. Escalate priority to at least Medium given CSP blast radius.
- Open question for author: Is the CSP set in nginx, Next.js config, middleware, or multiple places? What automated test will catch a CSP regression on the routes that actually load images?

### [INFO] AE-0300 rollback of mTLS origin-pull is not in the GitHub-controlled surface
- Assumption: The nginx config is in the repo (`nginx/nginx.conf.ssl`), so the rollback is repo-controllable.
- Risk: Enabling Authenticated Origin Pulls also requires a Cloudflare *dashboard/API* toggle for the zone, which is outside the repo. The plan acknowledges "note the manual DO/Cloudflare dashboard steps in the runbook," but the acceptance criterion "A documented rollback exists" is satisfied by prose, not by a runnable artifact. If the zone toggle and the nginx `ssl_verify_client` drift out of sync, the origin serves 5xx to Cloudflare (if nginx requires a cert CF isn't sending) or accepts non-CF traffic (if CF sends but nginx doesn't require). There is no described invariant check coupling the two.
- Impact: Misdial rollback is manual and slow during an incident; config drift between zone and nginx is undetected.
- Suggested mitigation: Add a runnable `scripts/deploy/verify_origin_pull.sh` that asserts both the Cloudflare zone setting (via API) and the nginx `ssl_verify_client` state match, run as a deploy post-step. Treat the dashboard toggle as a codified API call in the script, not a manual step.
- Open question for author: Is the Cloudflare zone API token available to scripts, and can the zone-state assertion be automated?

## Missing evidence
- The `deploy.yml` step that writes `.env` (heredoc vs `tee` vs `ssh ... cat`) — needed to judge whether the umask/chmod fix is adequate (AE-0301).
- The number and construction sites of Redis clients in the backend, and the existing `REDIS_URL`/settings precedence — needed to judge whether "fail fast" is enforceable without silent fallback (AE-0302).
- `docker-compose.prod.yml` restart policies and the frontend container healthcheck definition — needed to judge reboot safety (AE-0303).
- Shell history / cron / systemd / runbooks referencing `minio-test` / `langfuse-worker-test` — needed to judge true orphan status (AE-0304).
- Where the CSP is set (nginx vs Next.js vs middleware) and whether it is set in multiple layers — needed to judge fix locality and the acceptance criterion's reliability (AE-0305).
- Cloudflare zone API token availability — needed to make the AE-0300 rollback/origin-pull assertion codified rather than manual.
- The actual low-traffic window data backing the chosen `Automatic-Reboot-Time` (AE-0303).
- Whether secret rotation was considered given a confirmed 644 window on live keys + 64-day uptime + unauth Redis (cross-ticket concern, not covered by any single ticket).

## Residual risks if plan proceeds unchanged
- A Cloudflare-range refresh or origin-pull toggle drift can strand the site with a prose-only rollback and no executable verification of the mTLS/firewall coupling (AE-0300).
- Secrets remain momentarily world-readable per deploy and the `.env.backup.*` creator is unidentified, so backups may recur; "cannot regress" is overstated (AE-0301).
- Redis auth may silently fall back to unauthenticated in prod, or break dev/CI, depending on URL parsing and per-client construction divergence (AE-0302).
- Reboot could worsen the pre-existing frontend unhealthy state with no rollback from a reboot, and auto-reboot timing is unvalidated against real traffic (AE-0303).
- Removing `*-test` containers may break an undocumented operator dependency with no reversible restore path (AE-0304).
- A CSP change ships with no automated regression net and an undefined env-gating signal on a low-priority ticket (AE-0305).
