# Cold Critic Review

## Findings

### [WARN] Slug as a Fragile Foreign Key
- Assumption: using a string `slug` as the reference in `project.theme` will be stable/reliable for custom palettes.
- Risk: ambiguity on slug lifecycle/uniqueness. If a slug can be reused after a palette is archived, a project referencing it could silently resolve to a different palette. Plan is inconsistent: "custom id" (D6) vs "custom id/slug" (resolver), while the table has both a UUID `id` and a string `slug`. A mutable/non-unique/reusable string FK is a data-integrity risk.
- Impact: High — projects could change appearance with no user action or error (silent corruption of user content).
- Suggested mitigation: store the immutable UUID `palettes.id` in `project.theme`. If slug is also stored, make it immutable on create with a DB unique constraint across ALL palettes (incl. archived); `PATCH` must reject slug changes.
- Open question: is `slug` the FK stored in `project.theme`, or only a human-readable API/display key with UUID `id` as the real reference?

### [WARN] Understated Migration and Rollback Risk
- Assumption: the P1 enum→string migration is "reversible," implying a safe rollback exists.
- Risk: prod DB has historically DRIFTED from Alembic; that makes both `upgrade` and `downgrade` high-risk. Confidence in a simple rollback understates the danger — a failed downgrade on a live DB could extend the ~12-min outage and need manual recovery.
- Impact: High — extended downtime; a failed rollback could leave app+DB inconsistent.
- Suggested mitigation: mandatory pre-flight + contingency plan — (1) trial both upgrade AND downgrade on a staging DB restored from a recent prod backup; (2) prioritize roll-forward over rollback as primary recovery; (3) define post-deploy monitoring queries to confirm column data integrity.
- Open question: what is the specific, tested contingency if `alembic downgrade` fails against the prod schema?

### [WARN] Unspecified Concurrency Controls
- Assumption: app-level logic is sufficient to enforce constraints like name "unique among active."
- Risk: without DB-level enforcement, two concurrent POSTs with the same name can both pass app validation before either commits → duplicates.
- Impact: Medium — integrity violation, UI confusion, downstream bugs assuming uniqueness.
- Suggested mitigation: DB-level constraints for all uniqueness — partial unique index `... (name) WHERE archived = false` for active-name uniqueness; standard unique constraint on `slug`.
- Open question: beyond name/slug, are other implicit integrity rules that should be DB-enforced?

## Missing evidence
- Exact `ALTER TABLE` DDL for the theme-column migration (the Alembic script) is not provided.
- `slug` generation/validation/immutability logic is undefined.
- Keyword "sanitization" process is undefined, hard to assess the abuse mitigation.

## Residual risks if plan proceeds unchanged
- Silent content mutation if palette slugs are reused.
- Extended production outage if a migration fails on the known-drifty prod DB without a pre-vetted rollback/roll-forward plan.
- Race conditions producing duplicate palette names, violating a system invariant.
