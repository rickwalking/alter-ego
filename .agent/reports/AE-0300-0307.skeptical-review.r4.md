# Cold Critic Review

## Verdict
WARN *(bordering BLOCK — two findings would, if shipped as written, void the stated invariant of the very ticket that introduces them)*

## Findings

### [BLOCKER] AE-0302: fail-closed is enforced only on the backend client, not on the Redis container — a missing `REDIS_PASSWORD` recreates the exact hole the ticket closes
- Assumption: The trust boundary is the backend's startup `ConfigError` ("fail closed … raises rather than connecting unauthenticated").
- Risk: `requirepass` is set on the Redis service via `redis-server --requirepass ${REDIS_PASSWORD}`. If the secret is absent/empty in the compose env (a very plausible failure — new secret, typo, drift), **`requirepass ""` means no-auth in Redis**. The backend then fails closed and cannot connect — *but Redis is up and open on the internal `alter-ego` network*, exactly the unauthenticated-read/write-to-workflow-state condition AE-0302 exists to close. The AC "unauthenticated access is rejected" is only verified when the secret is *present*; the failure direction is unverified.
- Impact: The single most likely deploy failure (missing env) yields the one state the ticket is meant to prevent, silently, with the app down so nobody notices Redis is also wide open. This is worse than today, because today the app at least runs.
- Suggested mitigation: Add a symmetric server-side fail-closed — Redis container refuses to start (or refuses `requirepass ""`) when `ENVIRONMENT` is prod/unset. AC: empty `REDIS_PASSWORD` in prod ⇒ Redis container unhealthy/exit, not "open, app down." Test the empty-secret branch in prod-gate, not only the present-secret branch.
- Open question for author: What is the contract on the Redis container when the secret is missing? Currently undefined; the AC only pins the backend half.

### [BLOCKER] AE-0306: "atomic" `POSTGRES_PASSWORD` rotation has no rollback/failure procedure — a mid-rotation stall strands the stack on *neither* password
- Assumption: Changing the DB password and every consumer "atomically" is achievable, and the QA checklist's "partial rotation rollback" suffices.
- Risk: A DB password change is atomic on the DB side, but applying it across "DB + `CAROUSEL_CHECKPOINT_POSTGRES_URL` + app DB URL + Langfuse's Postgres" requires a redeploy via GitHub Secrets over SSH — a multi-step, non-atomic path. If the deploy fails between "DB password now new" and "app container reading new" (deploy error, container restart loop, a coincident `main` auto-deploy per CLAUDE.md mid-rotation), the app cannot authenticate to the DB with either old (revoked) or new (not yet live) credentials. "Atomic" is asserted, not constructed; the Gherkin only covers the happy path. There is **no rollback** defined: you cannot un-rotate a DB password without a second rotation.
- Impact: An undefined outage window during rotation with no documented escape. The expensive-rotation ticket should be *most* rigorous about the failure path, and it is the least.
- Suggested mitigation: Specify the actual sequence (e.g., dual-password support if Postgres/PgBouncer support it, or accept a hard downtime window with the site deliberately offline), and define rollback *as code*: e.g., gate on a confirmed `pg_hba.conf`/role state before revoking the old password; revoke old only after smoke; sequence against the AE-0303 reboot lock so a reboot cannot fire mid-rotation. Add a Gherkin *failure* scenario.
- Open question for author: Does the installed Postgres support overlapping password validity for zero-downtime rotation, or is a deliberate downtime window required? This determines the whole rotation design and is never answered.

### [WARN] AE-0305 (fail2ban `ignoreip`) and AE-0300 (SSH admin-IP allow-list) share one critical input but have no hard cross-ticket gate
- Assumption: Both tickets reference "the same administrative IP source," and prose cross-references ("its `ignoreip` must include this ticket's SSH admin-IP," "AE-0300 … its fail2ban `ignoreip` must include this") are sufficient coordination.
- Risk: AE-0305 is `Blocked by: none`. AE-0300's admin-IP source is explicitly a *prerequisite yet unverified* ("Verify the admin-IP source BEFORE pinning"). If AE-0305 ships first (or in parallel), fail2ban is configured against an unverified, possibly-dynamic admin IP — the operator's own IP gets banned, and the only break-glass is the DO console (whose credential reachability is itself untested, see below). The two tickets create a joint lockout risk that neither `Blocked by` captures.
- Impact: Self-lockout with no tested recovery.
- Suggested mitigation: Add `Blocked by: AE-0300` on AE-0305 (at least the fail2ban sub-item), or split AE-0305 so the SSH/fail2ban work is gated on AE-0300's verified admin-IP and the CSP sub-item is not.
- Open question for author: Are AE-0305 and AE-0300 intended to be parallelizable? If not, encode it.

### [WARN] AE-0301: the exposure-window estimate methodology is mathematically invalid and it is load-bearing for AE-0306's accept-loss decision
- Assumption: "earliest file mtime / deploy history" bounds the 644 exposure window.
- Risk: Each deploy *rewrites* `.env` (per the deploy model), so mtime reflects the *last* write, not the first creation at 644. The window could be far longer than mtime suggests (the files could have been 644 since first bootstrap). This estimate is the recorded justification for AE-0306, including the "accept-loss-of-historical-Langfuse-data" branch — a weak bound under-justifies an irreversible data-loss decision.
- Impact: An accept-loss call made against an under-estimated (or unbounded) exposure period.
- Suggested mitigation: Bound the window by proxy — droplet creation date, first deploy date, CI deploy history, CT log / Shodan earliest sighting of the origin cert — and explicitly record the *lower* confidence limit, i.e., "exposed for at least X, possibly since droplet creation." Document that Langfuse accept-loss is justified against the worst case.
- Open question for author: Can the window actually be bounded, or should AE-0306 default to rotate-and-accept-loss regardless of the estimate?

### [WARN] AE-0301: "post-deploy smoke confirms the new value is live and in use" is underspecified across five distinct providers
- Assumption: A smoke confirms each rotated key is exercised.
- Risk: OpenAI, Anthropic, Pinecone, GLM, and Gemini keys are consumed by different code paths. A single carousel smoke typically exercises only a subset (often OpenAI + Anthropic). If smoke passes via the OpenAI path while the GLM/Gemini key is mis-set, "revoke-after-verify" revokes the old GLM/Gemini key and strands those providers until the next deploy.
- Impact: Silent provider breakage post-revocation.
- Suggested mitigation: AC should enumerate a per-provider "live and in use" assertion (one smoke step per rotated provider key), not a single blanket smoke.
- Open question for author: Which smoke exercises Pinecone / GLM / Gemini distinctly today?

### [WARN] AE-0300 canary ("temporary health path until edge-only traffic confirmed 200") contradicts the L4 allow-list being the trust boundary
- Assumption: A direct-origin health path can coexist during canary with "80/443 only from CF ranges."
- Risk: A direct-origin health path is, by definition, not from a CF range; providing it requires temporarily opening the firewall to a non-CF source — which is the very bypass this ticket removes. The canary is unspecified about *how* the hole is opened/closed and re-closed automatically.
- Impact: Either the canary is meaningless (runs through CF anyway) or it reintroduces the bypass during rollout with no defined closure gate.
- Suggested mitigation: Define the canary as either CF-routed only, or a host-local check (ssh + curl from the droplet itself) that never touches the public 443 path. State which.
- Open question for author: What concrete network path does the canary use?

### [WARN] AE-0306: "all consumers" of `POSTGRES_PASSWORD` may not be a single instance — "atomic" may be two independent rotations
- Assumption: One Postgres, one password, atomic change across consumers.
- Risk: AE-0306 lists "the app, carousel checkpointing, and Langfuse" as consumers. Langfuse commonly ships its *own* Postgres in compose. If "Langfuse's Postgres" is a separate instance with its own credentials (possibly not in GitHub Secrets but in Langfuse's own DB init/env), then "rotate POSTGRES_PASSWORD atomically" is actually two rotations on two databases with two credential stores — collapsed into one AC.
- Impact: A second Postgres whose password rotation path is never specified, possibly unrotated (still exposed).
- Suggested mitigation: Decompose: enumerate every Postgres instance and its credential source (GitHub Secrets vs Langfuse-managed) before the AC; split the AC per instance.
- Open question for author: Is there one Postgres or two? Where does Langfuse's Postgres password originate?

### [WARN] AE-0303: Cloudflare zone analytics as the "canonical" reboot-time source has unverified availability
- Assumption: Cloudflare zone analytics is queryable and of sufficient granularity/retention.
- Risk: Cloudflare analytics API/retention is plan-dependent (free-tier dashboards are coarse and short-retention; GraphQL analytics API requires an API token with `Zone.Analytics` scope that may not be provisioned). The ticket names the source but doesn't confirm the operator can actually query it — the fallback (nginx logs) may end up the *de facto* source without the intended justification.
- Impact: Reboot time chosen on weaker data than claimed, or the AC silently satisfied by the fallback with no recorded query.
- Suggested mitigation: Pre-verify the analytics endpoint/token exists and record the actual query+screenshot *before* adopting the time; if only nginx logs are usable, say so up front.
- Open question for author: Is the Cloudflare zone on a plan with the analytics API enabled, and is a token provisioned?

### [WARN] AE-0300 / AE-0303: the scheduled CF-range firewall refresh is not coordinated with AE-0303's auto-reboot lock
- Assumption: AE-0303's lock file coordinates "deploy or rotation."
- Risk: AE-0300 runs a *scheduled* CF-range firewall apply on its own cadence with a monotonic guard. AE-0303's auto-reboot defers only on deploy/rotation locks. A CF-range refresh firing mid-reboot (or mid-AE-0306 rotation) is uncoordinated: the monotonic guard refuses a smaller fetch and alerts, but the reboot and the refresh are still racing the firewall state, and a refused refresh during a reboot leaves the origin on a stale allow-list with no recorded reconciliation.
- Impact: Firewall state ambiguity during reboot/rotation; possible origin darkening or stale ranges lingering past a reboot.
- Suggested mitigation: Add the CF-range refresh to the set of things AE-0303's lock honors (refresh skips while reboot/rotation lock held), and a post-reboot reconciliation step that re-confirms the applied range == last-known-good.
- Open question for author: Does the CF-range refresh take or honor any lock today? (None specified.)

### [WARN] AE-0302: a single Redis client factory may collapse distinct connection semantics (DB index, pool, Streams vs cache)
- Assumption: One factory cleanly routes all consumers with no behavioral change.
- Risk: Existing consumers (Streams publisher/consumer, cache, checkpoint, healthcheck) may use different Redis logical DBs / connection-pool settings / socket timeouts. A single factory with a single `(url, password)` contract could silently flatten these (e.g., force DB 0, share a connection pool across a blocking XREAD and a cache). "Route every consumer through it" is stated but the factory's parameter surface is not defined.
- Impact: Subtle behavioral regression ( Streams blocked by cache pool exhaustion, wrong DB writes) masked as "auth works."
- Suggested mitigation: Specify the factory's parameters (db, socket timeouts, pool size, decode_responses) and audit each consumer's pre-factory params; assert no param loss in a unit test, not just "all use the factory."
- Open question for author: Do current Redis clients differ on `db`, `decode_responses`, or pool settings?

### [INFO] AE-0304: the 24h-and-timers-fired soak only yields evidence if the timer set is fully enumerated first
- Assumption: "Every daily cron/systemd timer has fired" is a meaningful verification.
- Risk: If no daily job touches `minio-test` (likely, since it's orphaned), the soak reverts to a pure wall-clock with no actual dependency verification; weekly/monthly hidden dependents are still uncovered. The criterion proves something only relative to an *enumerated* timer set, which the orphan-grep is meant to produce but isn't explicitly linked.
- Suggested mitigation: Record the enumerated cron/systemd list as a precondition artifact; the soak check is "every timer in *that list* fired," with the list referenced by the AC, not free-floating.
- Open question for author: Where is the enumerated timer list captured?

### [INFO] AE-0300 break-glass via DO console — procedure documented, credential reachability untested
- Assumption: The DO web/recovery console is a usable recovery path.
- Risk: "Documented" ≠ "tested." If SSH (and 443) are mis-scoped, recovery depends on DO console login — which presumes an operator knows/has the DO credentials *and* 2FA second factor from a device reachable when SSH is down (no shared password manager entry? lost YubiKey?). The runbook documents the path; it does not verify it.
- Suggested mitigation: Add an AC that a mock break-glass access succeeds end-to-end (login, console, firewall flush) as a one-time drill before the ticket closes; record who holds DO access and 2FA.
- Open question for author: Has any operator actually exercised the DO console as a break-glass, ever?

## Missing evidence
- The operational truth of the admin-IP plumbing: static? VPN/jump host? GitHub Actions runner egress IP? This single fact gates AE-0300's, AE-0305's, and fail2ban's `ignoreip` design and is repeatedly declared a *prerequisite* but never *resolved in the packet*.
- Whether the installed Langfuse version supports encryption-key/salt rotation with re-encryption (AE-0306's own AC flags this as TBD but the schedule/deadline doesn't budget time to find out).
- Whether there is one Postgres or two (app vs Langfuse) and where Langfuse's DB password originates.
- Whether the Cloudflare analytics endpoint is actually queryable by the operator.
- Per-provider key-usage smoke coverage today (which smoke touches GLM/Gemini/Pinecone).

## Residual risks if plan proceeds unchanged
- A missing `REDIS_PASSWORD` at deploy produces an open Redis + down app (the exact failure AE-0302 is meant to prevent) — masked because the AC only verifies the happy path.
- A failed `POSTGRES_PASSWORD` rotation has no rollback and no defined downtime window → open-ended outage.
- A wrong/dynamic admin IP can compound AE-0305's fail2ban with AE-0300's SSH allow-list into a self-lockout with an untested DO-console break-glass.
- The exposure-window estimate may understate the real window, weakening AE-0306's accept-loss justification for historical Langfuse data (an irreversible decision resting on an invalid lower bound).
- Provider-specific keys (GLM/Gemini/Pinecone) may be stranded post-revoke because the "verify" smoke doesn't exercise them.
