# AE-0298 — Architecture Plan: expose custom_visual_details input in create-carousel Brief step

Ticket: AE-0298 · Tier: T2 · Type: Feature (frontend + optional response echo) · Date: 2026-07-01

## Decision

Frontend exposure of an already-landed backend field. Backend `custom_visual_details` is fully
wired (accept → persist → fold into every slide image prompt via `_compose_scene`, AE-0263 code).
Add the input; optionally echo the field in the create response for round-trip.

## Contract

- Add `custom_visual_details: z.string().max(500).nullable().optional()` to
  `carouselCreateRequestSchema` (`schemas/carousel.ts`, before `.refine`).
- Payload builder (`create/helpers.ts`): `custom_visual_details: form.customVisualDetails?.trim() || null`.
- Form state (`create/types.ts`) + input (`create-topic-section.tsx`, multiline `LabeledField`,
  maxLength 500) after niche. i18n label/placeholder/hint in en+pt under `create.form`.
- Optional: add `custom_visual_details` to backend `CarouselProjectResponse` + frontend
  `carouselProjectResponseSchema` (strictly additive optional) for read-back.

## Testing (deterministic — no live generation gate)

- Frontend `.feature`/unit: filled field is in the POST payload; blank → null/omitted; >500 chars
  blocked in UI.
- Backend unit: `_compose_scene(raw, custom_visual_details)` **includes** the guidance in the
  composed scene (already-landed function). This proves prompt-construction deterministically.
- **Do NOT** gate on a full carousel generation run (LLM + image-gen is non-deterministic,
  expensive, CI/Langfuse-hostile, and gameable). Leave full end-to-end to integration/Langfuse
  inspection, not a `.feature`.

## Guardrails

- Brand lock (style/palette/no-text) unchanged — this only exposes the existing *scene* guidance.
- Not the strategy-directive split / revision-feedback threading (AE-0263/AE-0261).
- **Length cap parity (verified):** backend `custom_visual_details = Field(None, max_length=500)`
  already matches the 500-char UI cap, so a direct-API caller can't exceed the per-slide image-gen
  cost the UI bounds. Keep them equal.

## Prompt-safety (user text folded into image prompts)

`_compose_scene` folds `custom_visual_details` into the per-slide scene, and the brand lock is
applied **after** via `strategy.wrap()` (style directives + palette + no-text are prepended;
user text lands in the `Scene:`/"Visual direction:" trailer — verified in the AE-0263 code path).
- **Structural test (positioning only — no overclaim):** assert `_compose_scene` places the user
  text in the delimited `Visual direction:` clause **after** the locked directives. This proves
  *positioning*, NOT injection-immunity — a trailer directive ("with the word BUY visible") can
  still attempt to override the no-text lock. Do not claim the test prevents override; it doesn't.
- **Residual risk (explicit, not hidden):** free text → image prompt is an injection surface —
  cost-amplification ("8k photorealistic…") **and** brand-bypass (burned-in text / named persons)
  are both possible under 500 chars; positioning-after-lock does not neutralize them. v1 accepts
  this at T2 with the cap + post-lock positioning + the brand lock's own no-text/style constraints.
  **Low-cost optional mitigation:** a deny-list regex on `custom_visual_details` rejecting
  `text|caption|overlay|word|8k|hdr|photoreal` patterns before compose. A content filter is a
  follow-up if abuse appears. The point: the control is honestly bounded, the test's claim matches
  what it verifies.

## Resolved (was open)

1. **Response echo:** include it now (add `custom_visual_details` to `CarouselProjectResponse` +
   frontend `carouselProjectResponseSchema`, strictly additive optional) — a field that can't be
   read back vanishes on reload (latent UX bug).
2. **i18n:** add keys for the new field only; do not expand scope to relocalize the whole topic
   section.
