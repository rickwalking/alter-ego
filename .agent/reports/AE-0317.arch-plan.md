## 3. Validation verdict

**The idea is SOUND, with two architectural corrections.**

✅ **Valid**: web research (navigate provided URLs + DuckDuckGo topic search) belongs
in the initial workflow's research phase, uniformly across ALL entry points. Today the
research phase silently degrades (GLM answered "I am unable to access external links"
and the workflow would have proceeded with that as a research finding). This gap is
exactly what AE-0249/ADR-0015 set out to close and never finished.

⚠️ **Correction 1 — deterministic runner, not an autonomous subagent, inside the
workflow.** ADR-0007 and AE-0249 itself ("deterministic phases must **stay** LangGraph
nodes — over-agentifying them would regress ADR-007's determinism/HITL guarantees")
forbid making the workflow's research phase an open-ended DeepAgents loop. The initial
workflow should get research via a **deterministic pipeline** (scrape given URLs →
DDG-search the topic → synthesize), reusing the exact same `ResearchTool` the
researcher subagent wraps. Deterministic = testable, bounded latency/cost, CI-mockable.

⚠️ **Correction 2 — the orphaned subagent's designed home is the chat pipeline.**
`build_researcher_subagent` was specified (AE-0249) for mid-conversation URL research
in the DeepAgents chat pipeline. The orphan fix is to **register it in
`RAGAgent._build_subagents()`** — not to force the spec object into the LangGraph
engine. Result: one shared `ResearchTool`, consumed two ways — deterministically by
the workflow research phase, agentically by the chat researcher subagent. Both
consume the same Protocol; no duplication.

🔒 **Security finding (must-fix, pre-existing)**: F7 — the RAG edge scrapes
user-provided URLs with no SSRF guard (internal IPs / cloud metadata endpoints
reachable from the backend container). Any new wiring MUST route through
`is_safe_research_url`, and the existing unguarded edge must be fixed in the same
change.

**ADR check**: no new ADR required. This *implements* accepted ADR-0015/ADR-0016
intent and *preserves* ADR-0007 (research stays a deterministic pre-graph step;
tool logic stays in `application/` behind the Protocol; agent packages stay façades).
Decision recorded here + in ticket Decision Logs. (Borderline call — if review
disagrees, a 1-page ADR "research enrichment is a deterministic pre-graph step"
can be extracted from §4.)

## 4. Target design

### 4.1 New module: `application/services/carousel/research_enrichment.py`

A deterministic, bounded enrichment step owned by the application layer, executed by
`EditorialWorkflowService.start_workflow` **before** `synthesize_research` — i.e. it
runs for every entry point (HTTP route, RAG chat tool, DeepAgents subagent) because it
lives in the service, not at route edges.

```python
@dataclass(frozen=True)
class ResearchEnrichmentConfig:
    research_tool: ResearchTool | None      # None → no-op (CI, tests, kill-switch)
    settings: ResearchEnrichmentSettings    # caps + toggle, see 4.4

async def enrich_sources(
    sources: list[dict[str, str]],
    topic: str,
    config: ResearchEnrichmentConfig,
) -> list[dict[str, str]]:
```

Behavior (all constants in `domain/constants/research_enrichment.py`):

1. **URL navigation** — for each source where `source_type == "url"` OR content
   matches the anchored `_URL_PATTERN` (same superset rule as today's RAG edge):
   - `is_safe_research_url(url)` guard → on failure, keep original content, log
     `research_url_blocked` (warning).
   - `research_tool.scrape_url(url)` with per-URL timeout (15s, the tool's own
     `page.goto` timeout) → `sanitize_web_content(scraped)`.
   - Graceful degradation on failure: keep the URL string as content, log
     `research_url_scrape_failed` — a dead link must never 400 the workflow.
   - Concurrency: `asyncio.Semaphore(MAX_CONCURRENT_SCRAPES = 2)` +
     `asyncio.gather` (Chromium memory pressure in the backend container).
   - Cap: first `MAX_URL_SOURCES = 5` URL sources scraped; the rest pass through
     (log `research_url_cap_hit`).
2. **DuckDuckGo topic search** — `research_tool.search_web(topic, …)`, take top
   `MAX_SEARCH_RESULTS = 3` hits, append them as synthetic sources
   `{title, content: snippet, source_type: "web_search", url}`. Snippets only —
   do NOT scrape search hits (bounded latency; search hits are unvetted).
   Failure → skip enrichment, log `research_search_failed` (warning).
3. Return enriched list; `synthesize_research` then runs per source exactly as today.

Latency budget (worst case): 5 scrapes / 2 concurrent × 15s ≈ 45s + search ~5s +
existing synthesis. Fits nginx 300s (F10) with margin; still, see rollout toggle.

### 4.2 De-duplicate the route-edge scraping

`_scrape_url_sources` + its two call sites in `api/dependencies/agents.py` are
**deleted**; both RAG paths now get scraping via the service (4.1). This closes the
unguarded-SSRF edge (F7) as a side effect of consolidation — one implementation,
one guard. `EditorialWorkflowConfig` (already a config object, `editorial_workflow_service.py:80`)
gains `research_tool: ResearchTool | None = None`; the container wires
`container.research_tool()` at both build sites (`build_editorial_workflow_service`
route seam + `build_rag_agent`).

### 4.3 Fix the orphan: register the researcher subagent where it was designed

`agents/rag_agent.py:_build_subagents()` additionally returns
`build_researcher_subagent(ResearcherSubagentConfig(research=…, search_documents=…, model=…))`
when carousel tool access is present (same condition as the editorial subagent).
This completes AE-0249's unfinished registration: chat users get agentic
`search_web`/`scrape_url` (guarded by the @tool adapter's existing SSRF check),
and the module stops being dead code. (Explicit product alternative: delete the
subagent instead — NOT recommended; the capability is chat's only way to research
interactively, and the code is tested and shipped.)

### 4.4 Settings / kill switch

`Settings.research_enrichment_enabled: bool = True` (+ optional cap overrides).
Disabled → `enrich_sources` no-ops (yesterday's behavior). This is the rollback
lever: no redeploy of code needed to restore prior behavior, only an env flip.

### 4.5 Synthesis hardening (companion ticket, T1 — the actual 400 root cause)

1. **Cache poisoning**: `SourceSynthesisAgent.extract_key_points` moves
   `self._cache.set(...)` to AFTER a successful `_parse_response` (`source_synthesis_agent.py:50-53`).
2. **Repair retry**: parse via the existing `extract_json_with_repair` (F9) instead
   of bare `extract_json` — one bounded LLM repair round-trip on truncated/malformed
   JSON, already battle-tested in the content drafter. (Requires threading an
   `LLMService`-compatible callable or refactoring the helper to accept
   `BaseChatModel`; developer's choice, keep ≤3-arg rule via config object.)
3. **Observable failure**: `editorial_workflow.py:178-182` logs the swallowed
   exception (`structlog`, `workflow_start_failed`, project_id + error) and returns
   a specific detail constant (e.g. `ERR_RESEARCH_SYNTHESIS_FAILED`) instead of the
   opaque `ERR_INVALID_REQUEST`. NOTE: new constant in
   `domain/constants/carousel_workflow.py`; response remains a 400 → **no OpenAPI
   shape change** (detail is a string), but re-run the 4 pinned regenerations if the
   route metadata changes at all (memory: backend-pinned-artifacts-and-ratchets).

## 6. Testing plan

Gherkin first: `tests/features/research_enrichment.feature` — scenarios:
- Happy: url source scraped → findings reference page text (mock ResearchTool).
- Happy: topic search appends ≤3 `web_search` sources.
- Edge: unsafe URL (private IP / non-http) blocked, original content kept, no raise.
- Edge: scrape timeout/error → graceful pass-through, workflow still starts.
- Edge: enrichment disabled via settings → byte-identical legacy behavior.
- Failure: synthesis invalid JSON → repair retry runs; if repair also fails →
  400 with `ERR_RESEARCH_SYNTHESIS_FAILED` (not generic), error logged.
- Failure-regression: bad raw response is NOT cached (poisoning test: first call
  malformed → second call re-invokes LLM, does not replay the bad raw).

Unit: `enrich_sources` (caps, concurrency, guard, no-op modes) with a stub
`ResearchTool` (Protocol — trivially fake; **CI has no network/keys**, so no live
ddgs/playwright in tests; keep lazy imports intact). `RAGAgent._build_subagents`
returns researcher + editorial (kills the orphan-detection finding). Route test:
400 detail constant + log emitted.

Integration: existing `test_researcher_url_navigation.py` keeps covering the
subagent; add one consolidation test asserting BOTH RAG entry points and the HTTP
route produce scraped content via the single service path (regression against
re-introducing edge scraping).

Mutation: business logic lands in `application/services/carousel/` — inside
mutmut's scope; keep tests free of repo-root file reads (memory: mutmut-sandbox).

## 8. Risks

| Risk | Mitigation |
|------|------------|
| Latency pushes sync `workflow/start` toward timeout | Caps + concurrency (4.1); 300s nginx headroom; kill switch |
| Chromium OOM in prod container under concurrent starts | Semaphore(2); observe post-deploy; PDF export already exercises this Chromium |
| DDG rate-limiting/403 from droplet IP | `ddgs` handles backend fallbacks; failure path is non-fatal by design |
| Scraped page content injects prompt garbage | `sanitize_web_content` (existing, F2 path already relies on it) |
| GLM stream truncation recurs (transport flake) | 4.5 repair retry + no-cache-on-failure make it self-healing instead of self-poisoning |
| Search results pollute editorial quality | Snippets-only, capped at 3, labeled `web_search` so synthesis/outline can weigh them; product can tune via settings |
