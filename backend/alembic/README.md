# Alembic Migrations

## Baseline (AE-0086)

The active chain starts from a **single squashed baseline** —
`versions/63eaefa67b8c_initial_baseline_schema.py` (`down_revision = None`) — which
creates the entire current schema exactly as defined by `Base.metadata`
(`infrastructure/database/models`).

The previous incremental migrations `0001`–`0011` were never runnable from an empty
database: the base tables (`users`, `projects`, …) were created at startup by
`Base.metadata.create_all`, so `0001` (which does `op.add_column('projects', …)`) failed
with `relation "projects" does not exist`. Per ADR-0009's pre-production operating context
(single user, replaceable data, no external consumers), they were squashed into one
autogenerated baseline rather than reconstructed (AE-0086, Option B). The originals are kept
in `versions_archived_pre_squash/` for reference (and in git history); that directory is **not**
on the scanned migration path.

> Startup still uses `Base.metadata.create_all` (see `infrastructure/database/config.py`).
> Switching startup to `alembic upgrade head` is a separate, later phase — out of AE-0086 scope.
> Alembic is now self-contained and CI-validated; it becomes authoritative-at-runtime later.

## Fresh database (CI / new environments)

```bash
DATABASE_URL=postgresql+asyncpg://user:pass@host:5432/db uv run alembic upgrade head
```

Applies the baseline cleanly from empty. `alembic downgrade base` reverses it (round-trip
validated by the `backend / Migrations (fresh DB)` CI gate, AE-0084).

## Existing dev database (already created via `create_all`)

The schema already matches `Base.metadata`, so **do not** run `upgrade head` (it would try to
re-create existing tables). Instead, mark the database as already at the baseline — no DDL, no
data loss:

```bash
DATABASE_URL=postgresql+asyncpg://user:pass@host:5432/db uv run alembic stamp head
```

This inserts the baseline revision into `alembic_version` without touching any tables.

## Adding new migrations

```bash
DATABASE_URL=... uv run alembic revision --autogenerate -m "describe change"
```

Review the generated file (FK-dependent table order, server defaults, indexes), then
`alembic upgrade head`. A fresh-DB `upgrade head` + `downgrade base` round-trip must stay green.
